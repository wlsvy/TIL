## 24.10.22

**How to Rename a MySQL Database**

- Database 는 이름을 변경하는 직접적인 방법은 없고, dump 를 복구하는 우회법을 사용

- 1. Create a New Database
  - `CREATE DATABASE new_database_name;`
- 2. Export Data from the Old Database
  - `mysqldump -uroot -p old_database_name > backup.sql`
- 3. Import Data into the New Database
  - `mysql -uroot -p new_database_name < backup.sql`
- 4. Drop the Old Database (Optional)
  - `DROP DATABASE old_database_name;`

    - 그럼 안 하던가, 혹은 점검과 별개로 사이드로 진행합니다. 

3. 모집단이 많으면(약 천만 Row수 이상?) Update문 보다는 Insert문을 활용 하는게 더 효율적입니다.

**UPDATE 쿼리가 INSERT 쿼리보다 느린 이유**

1. **수정되는 행이 많은 경우**
   - `UPDATE`는 수정할 행을 찾는 과정에서 시간이 걸리며, 수정 작업 후 여러 추가 작업이 필요합니다.
   - 수정되는 행이 많을수록 처리 속도가 느려집니다.

2. **인덱스의 영향**
   - `UPDATE`는 관련된 인덱스를 다시 작성해야 할 수 있어 시간이 오래 걸릴 수 있습니다.
   - `INSERT`는 새로운 데이터를 추가하는 것이라 인덱스 처리 부담이 상대적으로 적습니다.

3. **트리거 및 제약 조건**
   - `UPDATE`는 트리거 및 제약 조건을 확인하고 실행해야 하기 때문에 성능 저하가 발생할 수 있습니다.
   - `INSERT`는 새로운 데이터에만 해당 조건을 적용하므로 더 빠르게 처리될 수 있습니다.

4. **데이터베이스의 복제 및 트랜잭션 로그**
   - `UPDATE`는 기존 데이터를 수정하므로 많은 트랜잭션 로그 기록이 필요합니다.
   - `INSERT`는 새로운 데이터만 추가하면 되기 때문에 로그 기록이 단순합니다.

5. **행 잠금(Locking)**
   - `UPDATE`는 수정할 행을 잠가야 하며, 여러 트랜잭션 간 락 경합이 발생할 수 있습니다.
   - `INSERT`는 새로운 행을 추가하는 것이므로 일반적으로 락 경합이 적습니다.

6. **스토리지 오버헤드**
   - `UPDATE`는 MVCC(Multi-Version Concurrency Control) 방식에서 원본 데이터를 유지하고 새로운 버전을 생성해야 하므로 스토리지 오버헤드가 발생합니다.
   - `INSERT`는 새로운 데이터만 추가하므로 스토리지 오버헤드가 적습니다.

**MVCC (Multi-Version Concurrency Control)**

1. MVCC란?: 데이터베이스에서 **동시성 제어**를 구현하는 방법
  - 쓰기 트랜잭션은 임시 데이터를 수정하고, 그 동안 읽기 트랜잭션은 원본 데이터를 참조하므로, 서로 충돌하지 않고 동시성을 유지할 수 있다. 이 방식은 데이터의 일관성을 유지하면서 동시 작업 가능

2. 동작 방식
  - **타임스탬프 또는 트랜잭션 ID**: 각 데이터 행에 트랜잭션 ID나 타임스탬프를 부여하여 여러 버전의 데이터를 유지.
  - **Snapshot Isolation**: 트랜잭션 시작 시 해당 시점의 데이터 스냅샷을 생성해 일관성 유지.

3. MVCC의 장점
  - **읽기-쓰기 충돌 방지**: 읽기와 쓰기가 동시에 발생 가능, 성능 향상.
  - **데드락 방지**: 트랜잭션 간 대기 없이 동시 작업 가능.
  - **일관된 읽기 제공**: 긴 트랜잭션에서도 일관된 데이터 제공.

4. MVCC의 단점
  - **스토리지 오버헤드**: 여러 버전을 유지하기 때문에 스토리지 공간을 더 많이 사용.
  - **Garbage Collection 필요**: 오래된 버전을 제거하는 가비지 컬렉션 필요.
  - **복잡한 구현**: 트랜잭션 간 동시성을 제어하는 시스템이 복잡.

5. MVCC를 사용하는 데이터베이스
  - **MySQL (InnoDB)**: InnoDB 스토리지 엔진에서 MVCC 사용.
  - **PostgreSQL**: 일관된 읽기를 제공하기 위해 MVCC 사용.
  - **Oracle**: Oracle Database에서 동시성 제어를 위해 MVCC 사용.

6. 예시
  - **A 트랜잭션**이 주문 정보를 읽는 동안, **B 트랜잭션**이 같은 데이터를 수정해도 A는 변경 전 버전의 데이터를 계속 참조.
  - B 트랜잭션은 새로운 데이터를 추가하거나 수정해도 다른 트랜잭션에 영향을 주지 않고 커밋 가능.

[MySQL  MySQL 8.4 Reference Manual  17.8.2 Configuring InnoDB for Read-Only Operation](https://dev.mysql.com/doc/refman/8.4/en/innodb-read-only-instance.html)

- mysql 에서 Read Only 속성으로 권한 제한

```sql
CREATE USER 'readonly_user'@'%' IDENTIFIED BY 'password';

-- 특정 데이터베이스에 대한 SELECT 권한 부여
GRANT SELECT ON database_name.* TO 'readonly_user'@'%';

-- 모든 데이터베이스에 대해 읽기 전용 권한 부여 (필요한 경우)
GRANT SELECT ON *.* TO 'readonly_user'@'%';

-- 권한 확인
SHOW GRANTS FOR 'readonly_user'@'%';

-- 권한 적용
FLUSH PRIVILEGES;
```

- View 를 생성하는 경우

```sql
CREATE VIEW readonly_view AS SELECT * FROM database_name.table_name;

-- 권한 부여
GRANT SELECT ON database_name.readonly_view TO 'readonly_user'@'%';
```

## 25.02.24

**mySql 의 테이블 컬럼 추가 방식**

- [MySQL  MySQL 8.0 Reference Manual  15.1.9 ALTER TABLE Statement](https://dev.mysql.com/doc/refman/8.0/en/alter-table.html)

**ALGORITHM=COPY**

- 테이블을 새로 생성한 후, 기존 데이터를 새로운 데이블로 복사한 뒤 교체하는 방식.
- 테이블 크기가 클수록 시간이 오래 걸리며, 복사 중에는 테이블이 잠길 수 있다. 
- `ALTER TABLE my_table ADD COLUMN new_col INT;`

**ALGORITHM=INPLACE**

- 데이터를 복사하지 않고, 메타데이터만 변경하는 방식.
- 실행 속도가 빠르고, 데이블 잠금(lock)이 짧게 발생함.
- `ALTER TABLE my_table ADD COLUMN new_col INT NOT NULL DEFAULT 0, ALGORITHM=INPLACE;`

**ALGORITHM=INSTANT ( 8.0.12 이상)**

- 테이블 크기와 관계없이 즉시(Instant)컬럼을 추가할 수 있는 방법.
- 테이터를 복사하지 않고, 테이블의 메타데이터만 변경하는 방식.
- 매우 빠르고, 테이블이 크더라도 몇 초 이내에 작업 완료됨. 
- ADD COLUMN 만 지원, 새 컬럼은 맨 마지막 컬럼으로만 추가 가능.
- AUTO_INCREMENT, PRIMARY KEY, UNIQUE KEY, FOREIGN KEY는 사용할 수 없습니다.
- `ALTER TABLE my_table ADD COLUMN new_col INT DEFAULT 0, ALGORITHM=INSTANT;`

- 일부 작업 도중에는 임시테이블에 인덱스를 추가하고 `Force Index` 쿼리 적용하는 사례도 있음.



### FORCE INDEX 절 (MySQL) - Gemini 대답

    `FORCE INDEX` 절은 MySQL에서 쿼리 옵티마이저가 특정 테이블에 대해 명시적으로 지정된 인덱스를 강제로 사용하도록 지시하는 데 사용됩니다. 일반적으로 MySQL 옵티마이저는 쿼리 실행 계획을 최적화하기 위해 자동으로 가장 적절한 인덱스를 선택하지만, 때로는 개발자가 특정 인덱스를 사용하는 것이 더 효율적이라고 판단될 때 `FORCE INDEX`를 활용할 수 있습니다.
    `FORCE INDEX`는 특정 상황에서 쿼리 성능을 향상시키는 유용한 도구가 될 수 있지만, 옵티마이저의 자동 최적화 기능을 개발자가 수동으로 제어하는 것이므로 신중하게 사용해야 합니다. 충분한 이해와 테스트를 바탕으로 적절한 상황에만 적용하는 것이 중요합니다.

```sql
SELECT column1, column2, ...
FROM table_name
FORCE INDEX (index_name)
WHERE condition;
```
* **신중한 사용:** `FORCE INDEX`는 옵티마이저의 자동 최적화 기능을 무시하는 것이므로, 잘못 사용하면 쿼리 성능을 오히려 저하시킬 수 있습니다.
* **인덱스 존재 확인:** `FORCE INDEX`에 지정하는 인덱스가 실제로 테이블에 존재하는지 확인해야 합니다. 존재하지 않는 인덱스를 지정하면 오류가 발생합니다.
* **성능 테스트:** `FORCE INDEX`를 적용한 후에는 반드시 쿼리 성능을 테스트하여 실제로 성능 향상이 있는지 확인해야 합니다.
* **옵티마이저 통계 업데이트:** 옵티마이저가 잘못된 판단을 내리는 주된 이유는 통계 정보가 오래되었기 때문일 수 있습니다. 이 경우 `ANALYZE TABLE` 명령어를 사용하여 테이블 통계를 업데이트하는 것이 더 근본적인 해결책일 수 있습니다.
* **쿼리 구조 개선:** `FORCE INDEX`를 사용하기 전에 쿼리 구조 자체를 개선하여 옵티마이저가 더 나은 선택을 할 수 있도록 유도하는 것을 먼저 고려해야 합니다.


### WITH 절 (Common Table Expressions - CTEs) - Gemini

`WITH` 절은 SQL 쿼리 내에서 **임시적인 명명된 결과 집합**을 정의하는 데 사용됩니다. 이러한 임시 결과 집합을 공통 테이블 표현식 (Common Table Expressions - CTEs)이라고 부릅니다. CTE는 복잡한 쿼리를 더 읽기 쉽고 관리하기 쉽게 만들거나, 동일한 하위 쿼리 결과를 여러 번 참조해야 할 때 유용합니다. 쿼리가 실행되는 동안에만 존재하며, 쿼리 종료 후에는 사라집니다.

```sql
WITH 임시_테이블_명1 AS (
    SELECT 컬럼1, 컬럼2
    FROM 테이블1
    WHERE 조건1
),
임시_테이블_명2 AS (
    SELECT 컬럼A, 컬럼B
    FROM 테이블2
    WHERE 조건2
)
SELECT *
FROM 임시_테이블_명1
JOIN 임시_테이블_명2 ON 임시_테이블_명1.공통컬럼 = 임시_테이블_명2.다른공통컬럼
WHERE 최종_조건;
```

* 여러 개의 CTE를 쉼표(`,`)로 구분하여 정의할 수 있습니다.
* 최종 `SELECT` 문은 정의된 CTE를 참조하여 결과를 도출합니다.

---

* 쿼리 내에서 동일한 결과를 여러 번 참조해야 하는 경우, 해당 결과를 CTE로 정의해두면 코드를 반복해서 작성할 필요 없이 CTE의 이름으로 간편하게 참조할 수 있습니다
* `WITH` 절을 사용하여 분석 함수의 결과를 임시 테이블로 만들고, 이를 다시 메인 쿼리에서 활용하여 더 복잡한 분석 작업을 수행할 수 있습니다.
* 일부 데이터베이스 시스템 (예: PostgreSQL, SQL Server)에서는 `WITH RECURSIVE` 구문을 사용하여 조직 구조, 댓글 스레드 등 계층적인 데이터를 처리하는 재귀적 쿼리를 작성할 수 있습니다. CTE 내에서 자기 자신을 참조하여 계층 구조를 탐색하고 원하는 결과를 얻을 수 있습니다.

**`WITH` 절의 장점**

* **가독성 향상:** 복잡한 쿼리를 논리적인 단위로 분리하여 이해하기 쉽게 만듭니다.
* **재사용성:** 동일한 하위 쿼리 결과를 여러 번 참조할 때 코드 중복을 줄입니다.
* **유지보수 용이성:** 쿼리의 특정 부분을 수정해야 할 때 해당 CTE만 변경하면 됩니다.
* **재귀 쿼리 지원 (일부 DBMS):** 계층적 데이터 처리를 위한 강력한 기능을 제공합니다.

**주의사항**

* CTE는 해당 쿼리 내에서만 유효한 임시 테이블입니다. 쿼리 실행이 완료되면 자동으로 소멸됩니다.
* CTE의 이름은 쿼리 내에서 고유해야 합니다.

실무에서 사용하는 페이지네이션(Pagination) 방식은 크게 그 두 가지로 나눌 수 있습니다.

### 페이지네이션의 두 가지 핵심 방식

1. 오프셋 기반 페이지네이션 (Offset-based Pagination)

"페이지 번호"를 기반으로 동작하는 가장 전통적이고 직관적인 방식입니다.

 * **동작 원리**: "앞에서부터 N개를 건너뛰고(`OFFSET`), 그 다음부터 M개를 보여줘(`LIMIT`)."
 * **SQL 예시**: `SELECT * FROM posts LIMIT 10 OFFSET 20;` (3페이지의 10개 게시물)
 * **주요 사용처**: 게시판 하단의 `[1]`, `[2]`, `[3]`, ... 처럼 특정 페이지로 바로 이동해야 하는 UI

2. 커서 기반 페이지네이션 (Cursor-based Pagination)

"마지막으로 봤던 데이터의 다음 위치"를 기반으로 동작하는 방식입니다.

여기서 '커서'는 마지막 데이터의 고유한 식별자(ID 또는 타임스탬프)를 의미합니다. **Keyset Pagination**이라고도 불립니다.

 * **동작 원리**: "마지막으로 봤던 ID 값보다 큰(혹은 작은) 데이터부터 M개를 보여줘 (`LIMIT`)."
 * **SQL 예시**: `SELECT * FROM posts WHERE id > 125 ORDER BY id ASC LIMIT 10;` (ID 125번 게시물 다음 10개)
 * **주요 사용처**: SNS의 '더 보기' 버튼이나 모바일 앱의 '무한 스크롤' 기능, 데이터 배치 처리

| 항목          | **오프셋 (Offset)**                                                               | **커서 (Cursor)**                                                             |
| :---          | :---                                                                              | :---                                                                          |
| **개념**      | **"50페이지를 펴줘"**                                                             | **"책갈피 꽂아둔 곳부터 읽을게"**                                             |
| **장점**      | • 구현이 간단함• 특정 페이지로 바로 이동 가능                                     | • **데이터 변경에 안정적 (중복/누락 없음)**• 뒤로 갈수록 **성능 저하가 없음** |
| **단점**      | • **데이터 변경 시 중복/누락 발생 위험**• 뒤 페이지로 갈수록 **성능 급격히 저하** | • '다음 페이지'만 가능 (특정 페이지로 바로 점프 불가)• 구현이 조금 더 복잡함  |
| **추천 상황** | 데이터 변경이 거의 없는 간단한 게시판                                             | **실시간 데이터 처리, 무한 스크롤, 데이터 배치(Batch) 작업**                  |

---

- offset 기반 pagination 의 경우, 대상 범위의 row 가 중간에 조작된다면 데이터 누락/ 중복 등 추적하기 힘든 문제가 생길 수 있다.
- offset 은 내부적으로 N개의 row 를 전부 스캔한 뒤 그 결과를 버리기 때문에 offset 의 갯수가 점점 커질수록 성능 부담이 된다.
  - 이 부하는 index 로 정렬된 상태라 해도 여전하다. 테이블을 스캔해버리는 것.

## 데이터베이스 Boolean 컬럼

### Q1. `bool` 컬럼에는 일반적으로 인덱스를 걸지 않는 것이 권장되나요?

> [!NOTE]
> **네, 일반적인 분포(예: 50:50)에서는 권장하지 않습니다.**
- **이유:** `bool` 타입은 가질 수 있는 값이 2개뿐이라 카디널리티(고유값의 수)가 매우 낮습니다. 데이터가 균등하게 분포하면 선택도가 50%에 육박합니다.
  - 지역 코드(대륙 등), 성별 값등 Cardinality 가 낮은 컬럼에 해당
- **DBMS 동작:** MySQL 옵티마이저는 조회 범위가 전체 데이터의 약 15~20%를 넘으면 인덱스를 타는 비용이 더 크다고 판단하여 **Full Table Scan(FTS)**을 수행합니다. 따라서 인덱스를 생성해도 쓰기 오버헤드만 발생하고 사용되지 않습니다.

boolean 인덱스가 유용한 케이스가 있습니다 — 편향(skew)이 심할 때입니다:

- 예: is_deleted = true가 전체의 0.1%인 경우 → WHERE is_deleted = true 쿼리에서 인덱스가 매우 효과적.
- 이 경우 partial index(PostgreSQL) 또는 함수형/필터 조건과 결합한 인덱스가 더 적합합니다. MySQL은 partial index를 지원하지 않으므로, 보통 복합 인덱스의 선두 컬럼으로 두거나 generated column을 활용합니다.


# 26.07.01

## [How To Corrupt An SQLite Database File](https://www.sqlite.org/howtocorrupt.html)

SQLite 데이터베이스는 손상(Corruption)에 매우 강한 저항력을 가지고 있습니다. 트랜잭션 중간에 애플리케이션 크래시, 운영체제 크래시, 혹은 정전이 발생하더라도, 다음번에 데이터베이스 파일에 액세스할 때 부분적으로 기록된 트랜잭션이 자동으로 롤백됩니다. 이 복구 프로세스는 완전 자동화되어 있으며 사용자나 애플리케이션 측에서 어떠한 조치도 취할 필요가 없습니다.

비록 SQLite가 데이터베이스 손상에 강할지라도 완전히 면역된 것은 아닙니다. 본 문서에서는 SQLite 데이터베이스가 손상될 수 있는 다양한 경로를 설명합니다.

* **불량 스레드/프로세스의 파일 덮어쓰기:** SQLite 데이터베이스는 일반 디스크 파일이므로 다른 프로세스가 무단으로 접근해 덮어쓸 수 있습니다. 특히 닫힌 파일 디스크립터(File Descriptor)를 잘못 재사용하여 덮어쓰거나, 트랜잭션이 활성 상태일 때 비정상적인 방식으로 백업(단, `sqlite3_rsync`, `VACUUM INTO`, Backup API는 안전)하는 경우, 또는 시스템 복구에 필수적인 핫 저널(Hot Journal) 파일을 삭제·분리하는 경우 손상이 발생합니다.
* **파일 잠금(Locking) 문제:** 동시성 제어를 위해 파일 잠금 메커니즘에 의존합니다. 그러나 네트워크 파일시스템(NFS 등)의 잠금 구현 결함, 다중 스레드 환경에서 특정 스레드의 `close()` 호출이 프로세스 내 동일 파일의 POSIX 자문 잠금(Advisory Lock)을 모두 해제해 버리는 디자인 결함, 동일 앱 내에 서로 다른 SQLite 라이브러리가 중복 링크된 경우 잠금 메커니즘이 오작동하여 파일이 손상될 수 있습니다. `fork()` 후 부모의 커넥션을 자식 프로세스에서 공유하는 행위 역시 심각한 잠금 문제를 야기합니다.
* **동기화(Sync) 실패:** 데이터 일관성을 위해 OS에 `fsync()` 등의 동기화 요청을 보내지만, 대부분의 소비자용 디스크(특히 USB 플래시 메모리)는 성능 수치를 높이기 위해 쓰기가 완료되지 않았음에도 완료되었다고 거짓 응답을 보냅니다. 이때 정전이 발생하면 파일이 손상될 수 있습니다. (WAL 모드를 사용하면 이 위험을 크게 낮출 수 있습니다.) 개발자가 `PRAGMA synchronous=OFF` 등의 설정을 통해 고의로 동기화를 끌 때도 정전 시 손상 위험이 극대화됩니다.
* **메모리 오염 및 OS 결함:** SQLite는 애플리케이션과 동일한 메모리 주소 공간을 공유하므로, 앱의 잘못된 포인터나 버퍼 오버런이 SQLite의 내부 구조를 오염시킬 수 있으며 메모리 맵 I/O(`mmap`) 사용 시 이 위험이 더 커집니다. 또한 QNX의 `mmap()` 구현 버그나 운영체제 자체의 파일시스템 결함으로 인해 손상될 수도 있습니다.
* **설정 오류 및 SQLite 자체 버그:** `PRAGMA journal_mode=OFF` 사용 중 크래시가 나거나 `schema_version`을 강제로 수정하는 등 보호 기능을 끄는 설정 오류가 원인이 될 수 있습니다. 드물게 SQLite 자체의 과거 버그(WAL 모드 동시 쓰기 경쟁 조건, 중첩 트랜잭션의 세컨더리 저널 바운더리 체크 오류 등)로 인해 발생하기도 했으나, 철저한 테스트를 통해 발견 즉시 지속적으로 수정되어 왔습니다.

---

### 1. 불량 스레드나 프로세스에 의한 파일 덮어쓰기

SQLite 데이터베이스 파일은 평범한 디스크 파일입니다. 즉, 어떤 프로세스든 파일을 열고 무작위 데이터(Garbage)로 덮어쓸 수 있다는 뜻입니다. SQLite 라이브러리 측에서는 이를 방어하기 위해 할 수 있는 일이 없습니다.

#### 1.1. 파일 디스크립터가 닫힌 후에도 계속 사용되는 경우

파일에 대해 파일 디스크립터가 열려 있다가, 해당 파일 디스크립터가 닫힌 후 SQLite 데이터베이스로 다시 열린 여러 사례를 보았습니다. 나중에 다른 스레드가 원래 파일이 이미 닫혔다는 사실을 깨닫지 못하고 이전 파일 디스크립터에 계속 쓰 작업을 수행했습니다. 그러나 파일 디스크립터가 SQLite에 의해 다시 열렸기 때문에, 원래 파일로 들어가야 했던 정보가 SQLite 데이터베이스의 일부를 덮어쓰게 되어 데이터베이스 손상으로 이어졌습니다.
이러한 문제의 한 예는 2013년 8월 30일경 Fossil DVCS의 정식 저장소에서 발생했습니다. 당시 `sqlite3_open_v2()`를 호출하기 전에 파일 디스크립터 2번(표준 에러)이 (stunnel에 의해 선점된 것으로 추정되지만) 잘못 닫혀서, 저장소 데이터베이스 파일에 사용된 파일 디스크립터가 2번이 되었습니다. 이후 애플리케이션 버그로 인해 `assert()` 문이 `write(2, ...)`를 호출하여 에러 메시지를 내보내게 되었습니다. 하지만 파일 디스크립터 2번이 이제 데이터베이스 파일에 연결되어 있었기 때문에 에러 메시지가 데이터베이스의 일부를 덮어썼습니다. 이러한 종류의 문제를 방어하기 위해 SQLite 버전 3.8.1(2013-10-17) 이후 버전은 데이터베이스 파일에 낮은 번호의 파일 디스크립터를 사용하는 것을 거부합니다. (자세한 내용은 `SQLITE_MINIMUM_FILE_DESCRIPTOR`를 참조하십시오.)
닫힌 파일 디스크립터 사용으로 인해 발생한 손상의 또 다른 사례는 2014년 8월 12일 페이스북 엔지니어들의 블로그 포스트를 통해 보고되었습니다.
이 오류의 또 다른 사례는 2019년 7월 11일 Fossil에서 보고되었습니다. 디버깅 출력을 위해 파일 디스크립터가 열렸다가 닫힌 후 SQLite에 의해 다시 열렸으나, 디버깅 로직이 원래 파일 디스크립터에 계속 쓰기를 수행한 경우입니다. 버그 리포트와 수정 링크는 포럼 토론을 참조하십시오.

#### 1.2. 트랜잭션이 활성 상태인 동안 백업 또는 복원

백그라운드에서 자동 백업을 실행하는 시스템은 SQLite 데이터베이스 파일이 트랜잭션 중간에 있을 때 백업 복사본을 만들려고 시도할 수 있습니다. 그렇게 되면 백업 복사본에 일부는 오래된 콘텐츠가, 일부는 새로운 콘텐츠가 포함될 수 있어 손상된 상태가 됩니다.
SQLite 데이터베이스의 백업 복사본을 안전하게 만드는(즉, 올바르고 손상되지 않은 백업을 생성하는) 여러 가지 안전한 접근 방식이 있습니다. 순서에 상관없이 나열하면 다음과 같습니다.

1. `sqlite3_rsync` 유틸리티 프로그램(SQLite 3.47.0(2024-10-21) 이후 버전에서 사용 가능)은 대역폭 효율적인 프로토콜을 사용하여 SSH를 통해 가동 중인 SQLite의 복사본을 만듭니다.
2. `VACUUM INTO filename` 명령은 SQLite 데이터베이스의 현재 상태를 별도의 파일로 복사합니다.
3. 백업 API는 SQLite 데이터베이스의 일관된 복사본을 만들 수 있는 C 언어 인터페이스입니다.

위의 접근 방식은 가동 중인(Live) 데이터베이스에서도 작동합니다. 또한 복사가 진행되는 동안 진행 중인 트랜잭션이 없다면 SQLite 데이터베이스 파일의 복사본을 만드는 것도 안전합니다. 만약 이전의 쓰기 트랜잭션이 실패했다면, 롤백 저널(`*-journal` 파일)이나 쓰기 시점 로그(`*-wal` 파일)를 데이터베이스 파일 자체와 함께 복사하는 것이 중요합니다.

#### 1.3. 핫 저널(Hot Journal) 삭제

SQLite는 보통 모든 콘텐츠를 단일 디스크 파일에 저장합니다. 그러나 트랜잭션을 수행하는 동안 크래시나 정전 후 데이터베이스를 복구하는 데 필요한 정보는 보조 저널 파일에 저장됩니다. 이러한 저널 파일을 "핫 저널"이라고 부릅니다. 저널 파일은 원래 데이터베이스 파일 이름에 `-journal` 또는 `-wal` 접미사가 추가된 이름을 가집니다.
SQLite가 크래시나 정전으로부터 복구하려면 이 저널 파일들을 반드시 확인해야 합니다. 크래시나 정전 후에 핫 저널 파일이 이동, 삭제 또는 이름 변경되면 자동 복구가 작동하지 않으며 데이터베이스가 손상될 수 있습니다.
이 문제의 또 다른 징후는 8+3 파일 이름의 일관되지 않은 사용으로 인해 발생하는 데이터베이스 손상입니다.

#### 1.4. 데이터베이스 파일과 핫 저널의 잘못된 매칭

이전 예시는 더 일반적인 문제의 구체적인 사례입니다. SQLite 데이터베이스의 상태는 데이터베이스 파일과 저널 파일 모두에 의해 제어됩니다. 정지 상태(Quiescent state)에서는 저널 파일이 존재하지 않으며 데이터베이스 파일만 중요합니다. 하지만 저널 파일이 존재한다면, 손상을 피하기 위해 데이터베이스와 함께 유지해야 합니다. 다음과 같은 행위는 모두 손상으로 이어질 가능성이 높습니다.

* 두 개의 서로 다른 데이터베이스 간에 저널 파일을 교체하는 행위
* 저널 파일을 다른 저널 파일로 덮어쓰는 행위
* 저널 파일을 한 데이터베이스에서 다른 데이터베이스로 이동하는 행위
* 저널을 복사하지 않고 데이터베이스 파일만 복사하는 행위
* 원래 데이터베이스와 관련된 핫 저널을 삭제하지 않고, 데이터베이스 파일을 다른 파일로 덮어쓰는 행위

### 2. 파일 잠금(Locking) 문제

SQLite는 동시 프로세스 간의 액세스를 조율하기 위해 데이터베이스 파일과 쓰기 시점 로그(WAL) 파일에 파일 잠금을 사용합니다. 조율이 없다면 두 스레드나 프로세스가 동시에 데이터베이스 파일에 호환되지 않는 변경을 시도할 수 있으며, 이는 데이터베이스 손상으로 이어집니다.

#### 2.1. 잠금 구현이 손상되었거나 누락된 파일시스템

SQLite는 문서에 명시된 대로 잠금을 수행하기 위해 하위 파일시스템에 의존합니다. 그러나 일부 파일시스템은 잠금 로직에 버그가 있어 잠금이 항상 설명된 대로 작동하지 않습니다. 이는 네트워크 파일시스템, 특히 NFS에서 더욱 그렇습니다. 잠금 원시 함수(Primitives)에 버그가 있는 파일시스템에서 SQLite를 사용하고, 둘 이상의 스레드나 프로세스가 동시에 동일한 데이터베이스에 액세스하려고 하면 데이터베이스 손상이 발생할 수 있습니다.

#### 2.2. 별도의 스레드가 `close()`를 수행하여 POSIX 자문 잠금이 취소되는 경우

유닉스 플랫폼에서 SQLite가 사용하는 기본 잠금 메커니즘은 POSIX 자문 잠금(Advisory Locking)입니다. 불행히도 POSIX 자문 잠금에는 오용과 실패를 유발하기 쉬운 디자인상의 특이점이 있습니다. 특히, POSIX 자문 잠금을 보유하고 있는 파일 디스크립터와 동일한 프로세스 내의 어떤 스레드든 다른 파일 디스크립터를 사용하여 해당 잠금을 무효화할 수 있습니다. 특히 치명적인 문제 중 하나는 `close()` 시스템 호출이 프로세스 내의 모든 스레드와 모든 파일 디스크립터에 대해 동일한 파일에 걸려 있는 모든 POSIX 자문 잠금을 취소한다는 점입니다.
예를 들어, 멀티스레드 프로세스에 동일한 데이터베이스 파일에 대한 별도의 SQLite 데이터베이스 커넥션을 가진 둘 이상의 스레드가 있다고 가정해 보겠습니다. 이때 세 번째 스레드가 나타나 SQLite 라이브러리를 사용하지 않고 독자적으로 동일한 데이터베이스 파일에서 무언가를 읽고자 합니다. 아마도 이 세 번째 스레드는 데이터베이스의 백업 복사본을 만들고 싶어 하거나, 단순히 파일 형식을 식별하기 위해 처음 16바이트를 읽어 이것이 정말 SQLite 데이터베이스인지 확인하려고 할 수 있습니다. 이유가 무엇이든 세 번째 스레드는 `open()`, `read()`, 그리고 `close()`를 수행합니다. 사람들은 이것이 무해할 것이라고 생각할 것입니다. 하지만 `close()` 시스템 호출은 다른 모든 스레드가 데이터베이스에 유지하고 있던 잠금을 해제하는 결과를 초래합니다. 다른 스레드들은 자신들의 잠금이 방금 파기되었다는 사실을 알 방법이 없으며(POSIX는 이를 확인할 수 있는 메커니즘을 제공하지 않음), 따라서 잠금이 여전히 유효하다는 가정하에 계속 작동합니다. 이는 둘 이상의 스레드나 프로세스가 동시에 데이터베이스에 쓰기를 시도하게 만들어 데이터베이스 손상을 초래할 수 있습니다.
참고로 둘 이상의 스레드가 SQLite 라이브러리를 사용하여 동일한 SQLite 데이터베이스 파일에 액세스하는 것은 완전히 안전합니다. SQLite의 유닉스 드라이버는 POSIX 자문 잠금의 특이점을 알고 있으며 이를 우회하여 작동합니다. 이 문제는 스레드가 SQLite 라이브러리를 우회하여 데이터베이스 파일에 직접 읽기를 시도할 때만 발생합니다.
SQLite 버전 3.51.0(2025-11-04)부터 SQLite는 `close()`에 의해 깨지는 잠금으로 인한 문제를 피하기 위해 추가적인 방어책을 구현합니다. 이 새로운 방어책은 데이터베이스가 WAL 모드이고 다중 프로세스에서 액세스할 때 도움이 됩니다. 하지만 이것이 만병통치약은 아닙니다. 손상을 방지하기 위해 개발자는 다른 스레드에서라도 하나 이상의 데이터베이스 커넥션이 열려 있는 동안에는 SQLite 데이터베이스 파일에 대해 `close()`를 절대 사용하지 않도록 주의해야 합니다.

#### 2.3. 동일한 애플리케이션에 중복 링크된 여러 개의 SQLite 카피

이전 섹션에서 언급했듯이 SQLite는 POSIX 자문 잠금의 특이점을 우회하기 위한 조치를 취합니다. 이 우회 조치의 일부로 열려 있는 SQLite 데이터베이스 파일의 전역 목록(뮤텍스로 보호됨)을 유지합니다. 그러나 동일한 애플리케이션에 여러 개의 SQLite 카피가 링크되어 있으면 이 전역 목록의 인스턴스가 여러 개 존재하게 됩니다. SQLite 라이브러리의 한 카피를 사용하여 열린 데이터베이스 커넥션은 다른 카피를 사용하여 열린 데이터베이스 커넥션을 알지 못하므로, POSIX 자문 잠금 특이점을 우회할 수 없게 됩니다. 한 커넥션에서의 `close()` 작업이 자신도 모르게 다른 데이터베이스 커넥션의 잠금을 해제하여 데이터베이스 손상으로 이어질 수 있습니다.
위의 시나리오는 가설처럼 들릴 수 있습니다. 하지만 SQLite 개발자들은 정확히 이 버그를 가진 채 출시된 상용 제품을 최소 한 개 이상 알고 있습니다. 해당 벤더는 리눅스와 맥에서 발생하는 간헐적인 데이터베이스 손상 문제를 추적하는 데 도움을 요청하기 위해 SQLite 개발자들을 찾아왔습니다. 문제는 결국 애플리케이션이 두 개의 별도 SQLite 카피와 링크되어 있다는 사실로 밝혀졌습니다. 해결책은 애플리케이션 빌드 절차를 수정하여 두 개가 아닌 단 하나의 SQLite 카피만 링크하도록 변경하는 것이었습니다.

#### 2.4. 서로 다른 잠금 프로토콜을 사용하는 두 프로세스

유닉스 플랫폼에서 SQLite가 사용하는 기본 잠금 메커니즘은 POSIX 자문 잠금이지만 다른 옵션도 있습니다. `sqlite3_open_v2()` 인터페이스를 사용하여 대체 `sqlite3_vfs`를 선택함으로써, 애플리케이션은 특정 파일시스템에 더 적합할 수 있는 다른 잠금 프로토콜을 사용할 수 있습니다. 예를 들어, POSIX 자문 잠금을 지원하지 않는 NFS 파일시스템에서 실행되어야 하는 애플리케이션의 경우 닷파일(dot-file) 잠금을 선택할 수 있습니다.
동일한 데이터베이스 파일에 대한 모든 커넥션이 동일한 잠금 프로토콜을 사용하는 것이 중요합니다. 한 애플리케이션은 POSIX 자문 잠금을 사용하고 다른 애플리케이션은 닷파일 잠금을 사용한다면, 두 애플리케이션은 서로의 잠금을 보지 못하므로 데이터베이스 액세스를 조율할 수 없게 되며, 결과적으로 데이터베이스 손상이 발생할 수 있습니다.

#### 2.5. 사용 중인 데이터베이스 파일의 연결 해제(Unlink) 또는 이름 변경

두 프로세스가 동일한 데이터베이스 파일에 대한 커넥션을 열어둔 상태에서, 한 프로세스가 커넥션을 닫고 파일 링크를 해제(Unlink)한 다음, 동일한 이름으로 그 자리에 새 데이터베이스 파일을 생성하고 새 파일을 다시 열면 두 프로세스는 이름만 같고 서로 다른 데이터베이스 파일과 통신하게 됩니다. (이는 파일이 읽기 및 쓰기용으로 여전히 열려 있는 동안 파일 링크 해제를 허용하는 포식스 및 포식스 유사 시스템에서만 가능합니다. 윈도우에서는 이런 일이 발생할 수 없습니다.) 롤백 저널과 WAL 파일은 데이터베이스 파일의 이름을 기반으로 하므로, 두 개의 서로 다른 데이터베이스 파일이 동일한 롤백 저널 또는 WAL 파일을 공유하게 됩니다. 데이터베이스 중 하나에 대한 롤백 또는 복구 작업이 다른 데이터베이스의 콘텐츠를 사용할 수 있어 손상이 발생합니다. 데이터베이스 파일이 열려 있는 동안 이름을 변경하고 이전 이름으로 새 파일을 생성하는 경우에도 유사한 문제가 발생합니다.
즉, 열려 있는 데이터베이스 파일의 링크를 해제하거나 이름을 변경하는 행위는 정의되지 않았으며 아마도 원치 않는 동작을 초래합니다.
SQLite 버전 3.7.17(2013-05-20)부터 유닉스 OS 인터페이스는 데이터베이스 파일이 여전히 사용 중인 동안 링크 해제되면 에러 로그에 `SQLITE_WARNING` 메시지를 보냅니다.

#### 2.6. 동일한 파일에 대한 다중 링크

단일 데이터베이스 파일에 다중 링크(하드 링크 또는 소프트 링크)가 있다는 것은 파일이 여러 이름을 가지고 있다는 말과 같습니다. 둘 이상의 프로세스가 서로 다른 이름을 사용하여 데이터베이스를 열면 서로 다른 롤백 저널과 WAL 파일을 사용하게 됩니다. 즉, 한 프로세스가 크래시되면 다른 프로세스는 적절한 저널을 잘못된 위치에서 찾게 되므로 진행 중인 트랜잭션을 복구할 수 없게 됩니다.
즉, 두 개 이상의 이름을 가진 데이터베이스 파일을 열고 사용하는 것은 정의되지 않았으며 아마도 원치 않는 동작을 초래합니다.
SQLite 버전 3.7.17(2013-05-20)부터 유닉스 OS 인터페이스는 데이터베이스 파일에 다중 하드 링크가 있는 경우 에러 로그에 `SQLITE_WARNING` 메시지를 보냅니다.
SQLite 버전 3.10.0(2016-01-06)부터 유닉스 OS 인터페이스는 심볼릭 링크를 해석하고 정식 이름(Canonical name)으로 데이터베이스 파일을 열려고 시도합니다. 버전 3.10.0 이전에는 심볼릭 링크를 통해 데이터베이스 파일을 여는 것이 다중 하드 링크를 가진 데이터베이스 파일을 여는 것과 유사하여 정의되지 않은 동작을 초래했습니다.

#### 2.7. `fork()`를 거쳐 열린 데이터베이스 커넥션을 전달하는 행위

SQLite 데이터베이스 커넥션을 연 다음 `fork()`를 수행하고, 자식 프로세스에서 해당 데이터베이스 커넥션을 사용하려고 시도하지 마십시오. 온갖 종류의 잠금 문제가 발생할 것이며 쉽게 데이터베이스 손상으로 이어질 수 있습니다. SQLite는 그러한 동작을 지원하도록 설계되지 않았습니다. 자식 프로세스에서 사용되는 모든 데이터베이스 커넥션은 부모로부터 상속받는 것이 아니라 자식 프로세스에서 새로 열어야 합니다.
부모 프로세스에서 열린 커넥션이라면 자식 프로세스에서 `sqlite3_close()`를 호출하는 것조차 하지 마십시오. 하위 파일 디스크립터를 닫는 것은 안전하지만, `sqlite3_close()` 인터페이스는 부모 프로세스의 밑단에서 콘텐츠를 삭제하는 정리 활동을 호출할 수 있어 에러와 데이터베이스 손상까지 초래할 수 있습니다.

### 3. 동기화(Sync) 실패

데이터베이스 파일이 항상 일관성을 유지하도록 보장하기 위해, SQLite는 가끔 운영체제에 보류 중인 모든 쓰기를 영구 저장소로 플러시(Flush)하도록 요청한 다음 그 플러시가 완료되기를 기다립니다. 이는 유닉스에서는 `fsync()` 시스템 호출을, 윈도우에서는 `FlushFileBuffers()`를 사용하여 수행됩니다. 우리는 보류 중인 쓰기의 이 플러시 작업을 "동기화(Sync)"라고 부릅니다.
사실, 원자적(Atomic)이고 일관된 쓰기에만 관심이 있고 지속성(Durable) 있는 쓰기를 포기할 용의가 있다면, 동기화 작업이 콘텐츠가 영구 매체에 완전히 저장될 때까지 기다릴 필요는 없습니다. 대신 동기화 작업을 I/O 장벽(Barrier)으로 생각할 수 있습니다. 동기화 이전에 발생하는 모든 쓰기가 동기화 이후에 발생하는 어떤 쓰기보다 먼저 완료되는 한, 데이터베이스 손상은 발생하지 않습니다. 동기화가 실제 동기화가 아닌 I/O 장벽으로 작동하는 경우, 정전이나 시스템 크래시로 인해 이전에 커밋된 하나 이상의 트랜잭션이 롤백될 수 있지만(ACID의 "지속성" 속성 위반), 데이터베이스는 적어도 일관성을 계속 유지할 것이며 이는 대부분의 사람들이 중요하게 생각하는 부분입니다.

#### 3.1. 동기화 요청을 준수하지 않는 디스크 드라이브

불행히도 대부분의 소비자 등급 대용량 저장 장치는 동기화에 대해 거짓말을 합니다. 디스크 드라이브는 콘텐츠가 실제로 산화물(Oxide) 표면에 기록되기 전인 트랙 버퍼에 도달하는 즉시 영구 매체에 안전하게 저장되었다고 보고합니다. 이는 디스크 드라이브가 더 빨리 작동하는 것처럼 보이게 만듭니다(이는 제조업체가 무역 잡지에서 좋은 벤치마크 숫자를 보여주기 위해 매우 중요합니다). 그리고 공정하게 말해서, 트랙 버퍼가 실제로 산화물 표면에 기록되기 전에 전력 손실이나 하드 리셋이 없다면 이 거짓말은 보통 아무런 해를 끼치지 않습니다. 그러나 전력 손실이나 하드 리셋이 실제로 발생하고, 그 결과 동기화 이후에 기록된 콘텐츠가 산화물 표면에 도달한 반면 동기화 이전에 기록된 콘텐츠가 여전히 트랙 버퍼에 남아 있게 되면 데이터베이스 손상이 발생할 수 있습니다.
USB 플래시 메모리 스틱은 동기화 요청에 대해 특히 악질적인 거짓말쟁이로 보입니다. USB 메모리 스틱에 있는 SQLite 데이터베이스에 큰 트랜잭션을 커밋해 보면 이를 쉽게 확인할 수 있습니다. `COMMIT` 명령은 상대적으로 빠르게 반환되어 메모리 스틱이 운영체제에 말했고 운영체제가 SQLite에 모든 콘텐츠가 영구 저장소에 안전하게 있다고 말했음을 나타내지만, 메모리 스틱 끝에 있는 LED는 몇 초 동안 계속 깜박일 것입니다. LED가 여전히 깜박이는 동안 메모리 스틱을 뽑으면 빈번하게 데이터베이스 손상이 발생합니다.
참고로 SQLite는 동기화 요청 상태에 대해 운영체제와 하드웨어가 말하는 것을 무엇이든 믿어야 합니다. SQLite가 둘 중 어느 쪽이 거짓말을 하고 있으며 쓰기가 순서 없이 발생하고 있는지 감지할 수 있는 방법은 없습니다. 그러나 WAL 모드의 SQLite는 기본 롤백 저널 모드보다 비순차적 쓰기에 대해 훨씬 더 관대합니다. WAL 모드에서 동기화 작업 실패가 데이터베이스 손상을 일으킬 수 있는 유일한 시점은 체크포인트(Checkpoint) 작업 중입니다. `COMMIT` 동안의 동기화 실패는 지속성 상실을 초래할 수 있지만 데이터베이스 파일 손상으로 이어지지는 않습니다. 따라서 동기화 작업 실패로 인한 데이터베이스 손상에 대한 한 가지 방어선은 SQLite를 WAL 모드로 사용하고 체크포인트를 가능한 한 드물게 수행하는 것입니다.

#### 3.2. 프라그마(Pragma)를 사용하여 동기화 비활성화

인격 유지를 돕기 위해 SQLite가 수행하는 동기화 작업은 런타임에 synchronous 프라그마를 사용하여 비활성화할 수 있습니다. `PRAGMA synchronous=OFF`로 설정하면 모든 동기화 작업이 생략됩니다. 이는 SQLite가 더 빨리 실행되는 것처럼 보이게 하지만, 운영체제가 쓰기 순서를 자유롭게 재조정할 수 있게 하므로 모든 콘텐츠가 영구 저장소에 도달하기 전에 전력 실패나 하드 리셋이 발생하면 데이터베이스 손상이 발생할 수 있습니다.
최대의 신뢰성과 데이터베이스 손상에 대한 견고성을 위해, SQLite는 항상 기본 synchronous 설정인 `FULL`로 실행되어야 합니다.

### 4. 디스크 드라이브 및 플래시 메모리 실패

디스크 드라이브나 플래시 메모리 실패로 인해 파일 콘텐츠가 변경되면 SQLite 데이터베이스가 손상될 수 있습니다. 매우 드물지만 디스크는 가끔 섹터 중간에 비트를 반전(Bit flip)시키기도 합니다.

#### 4.1. 전원 안전 기능이 없는 플래시 메모리 컨트롤러

일부 플래시 메모리 컨트롤러에서는 쓰기 중에 전원이 차단되면 웨어 레벨링(Wear-leveling) 로직이 무작위 파일시스템 손상을 일으킬 수 있다고 합니다. 이는 예를 들어 전력 손실 당시에 열려 있지조차 않았던 파일의 중간에 무작위적인 변경이 나타나는 방식으로 나타날 수 있습니다. 따라서 예를 들어 전력 손실이 발생했을 때 디바이스가 플래시 메모리의 MP3 파일에 콘텐츠를 쓰고 있었다면, 데이터베이스가 전력 손실 당시에 사용 중이 아니었음에도 불구하고 SQLite 데이터베이스가 손상되는 결과가 발생할 수 있습니다.

#### 4.2. 가짜 용량 USB 스틱

유통 중인 사기성 USB 스틱 중에는 높은 용량(예: 8GB)을 가지고 있다고 보고하지만 실제로는 훨씬 적은 양(예: 1GB)만 저장할 수 있는 것들이 많습니다. 이러한 장치에 쓰기를 시도하면 종종 무관한 파일이 덮어써집니다. 따라서 사기성 플래시 메모리 장치를 사용하면 쉽게 데이터베이스 손상으로 이어질 수 있습니다. "fake capacity usb"와 같은 인터넷 검색은 이 문제에 대한 많은 충격적인 정보를 찾아줄 것입니다.

### 5. 메모리 오염

SQLite는 자신이 서비스하는 애플리케이션과 동일한 주소 공간에서 실행되는 C 라이브러리입니다. 즉, 애플리케이션의 잘못된 포인터(Stray pointer), 버퍼 오버런, 힙 오염 또는 기타 오작동이 내부 SQLite 데이터 구조를 오염시키고 궁극적으로 손상된 데이터베이스 파일로 이어질 수 있음을 의미합니다. 보통 이러한 종류의 문제는 데이터베이스 손상이 발생하기 전에 세그폴트(Segfault)로 나타나지만, 애플리케이션 코드 오류가 SQLite를 미묘하게 오작동하게 만들어 패닉을 일으키기보다 데이터베이스 파일을 손상시킨 사례들이 있었습니다.
메모리 맵 I/O(`mmap`)를 사용할 때 메모리 오염 문제는 더욱 심각해집니다. 데이터베이스 파일의 전체 또는 일부가 애플리케이션의 주소 공간에 매핑되어 있을 때, 매핑된 공간의 임의의 부분을 덮어쓰는 잘못된 포인터는 애플리케이션이 후속 `write()` 시스템 호출을 수행할 필요 없이 즉시 데이터베이스 파일을 손상시킵니다.

### 6. 기타 운영체제 문제

때때로 운영체제가 표준적이지 않은 동작을 보여 문제를 일으킬 수 있습니다. 때로는 이 비표준 동작이 의도된 것일 수도 있고, 때로는 구현상의 실수일 수도 있습니다. 그러나 어쨌든 운영체제가 SQLite가 기대하는 방식과 다르게 작동한다면 데이터베이스 손상의 가능성이 존재합니다.

#### 6.1. 리눅스 스레드(LinuxThreads)

일부 이전 버전의 리눅스는 스레드 지원을 위해 LinuxThreads 라이브러리를 사용했습니다. LinuxThreads는 Pthreads와 유사하지만 POSIX 자문 잠금 처리와 관련하여 미묘하게 다릅니다. SQLite 버전 2.2.3부터 3.6.23까지는 런타임에 LinuxThreads가 사용되고 있음을 인식하고 LinuxThreads의 비표준 동작을 우회하기 위해 적절한 조치를 취했습니다. 그러나 대부분의 현대 리눅스 구현은 Pthreads의 더 새롭고 올바른 NPTL 구현을 사용합니다. SQLite 버전 3.7.0(2010-07-21)부터는 NPTL의 사용이 가정됩니다. 아무런 확인도 하지 않습니다. 따라서 최신 버전의 SQLite는 LinuxThreads를 사용하는 이전 리눅스 시스템에서 실행되는 멀티스레드 애플리케이션에서 사용될 경우 미묘하게 오작동하여 데이터베이스 파일을 손상시킬 수 있습니다.

#### 6.2. QNX에서 `mmap()`의 실패

QNX에서 `mmap()`과 관련된 몇 가지 미묘한 문제가 존재하여, 단일 파일 디스크립터에 대해 두 번째 `mmap()` 호출을 하면 첫 번째 `mmap()` 호출에서 얻은 메모리가 제로화(Zeroed)될 수 있습니다. 유닉스의 SQLite는 WAL 모드에서 트랜잭션 조율을 위한 공유 메모리 영역을 생성하기 위해 `mmap()`을 사용하며 대규모 트랜잭션의 경우 `mmap()`을 여러 번 호출합니다. QNX `mmap()`은 해당 시나리오에서 데이터베이스 파일을 손상시키는 것으로 입증되었습니다. QNX 엔지니어들은 이 문제를 인지하고 해결책을 모색하고 있습니다. 여러분이 이 글을 읽을 때쯤에는 문제가 이미 해결되었을 수도 있습니다.
QNX에서 실행할 때는 메모리 맵 I/O를 절대 사용하지 않는 것이 권장됩니다. 또한 WAL 모드를 사용하려면 애플리케이션이 공유 메모리 없이 WAL을 사용하기 위해 배타적 잠금(Exclusive locking) 모드를 채택하는 것이 권장됩니다.

#### 6.3. 파일시스템 손상

SQLite 데이터베이스는 평범한 디스크 파일이므로 파일시스템의 모든 오작동은 데이터베이스를 손상시킬 수 있습니다. 현대 운영체제의 파일시스템은 매우 신뢰할 수 있지만 오류는 여전히 발생합니다. 예를 들어 2013년 10월 1일, 호스트 컴퓨터가 파일시스템 계층에 문제가 있는 리눅스 커널의 불안정한 빌드로 이동한 지 며칠 만에 Tcl/Tk용 위키를 보유한 SQLite 데이터베이스가 손상되었습니다. 그 사건에서 파일시스템은 결국 머신을 사용할 수 없을 정도로 심하게 손상되었지만, 문제의 가장 이른 징후는 손상된 SQLite 데이터베이스였습니다.

### 7. SQLITE 설정 오류

SQLite에는 데이터베이스 손상에 대한 많은 빌트인 보호 기능이 있습니다. 하지만 이러한 보호 기능 중 다수는 설정 옵션에 의해 비활성화될 수 있습니다. 보호 기능이 비활성화되면 데이터베이스 손상이 발생할 수 있습니다.
다음은 SQLite의 빌트인 보호 메커니즘을 비활성화하는 예시입니다.

* `PRAGMA synchronous=OFF`로 설정하면 운영체제 크래시나 정전이 발생할 경우 데이터베이스가 손상될 수 있습니다. (다만 이 설정은 애플리케이션 크래시로 인한 손상으로부터는 안전합니다.)
* 다른 데이터베이스 커넥션이 열려 있는 동안 `PRAGMA schema_version`을 변경하는 행위
* `PRAGMA journal_mode=OFF` 또는 `PRAGMA journal_mode=MEMORY`를 사용하고 쓰기 트랜잭션 중간에 애플리케이션 크래시가 발생하는 경우
* `PRAGMA writable_schema=ON`으로 설정한 다음 신중하게 하지 않고 DML 문을 사용하여 데이터베이스 스키마를 변경하면 데이터베이스를 완전히 읽을 수 없게 만들 수 있습니다.

### 8. SQLITE의 버그

SQLite는 버그가 최대한 없도록 매우 신중하게 테스트됩니다. 모든 SQLite 버전에 대해 수행되는 수많은 테스트 중에는 이러한 이벤트 동안 데이터베이스 손상이 발생하지 않음을 검증하기 위해 정전, I/O 에러, 메모리 부족(OOM) 에러를 시뮬레이션하는 테스트가 있습니다. 또한 SQLite는 심각한 문제없이 약 20억 개의 활성 배포로 필드에서 입증되었습니다.
그럼에도 불구하고 어떤 소프트웨어도 100% 완벽하지는 않습니다. SQLite에는 데이터베이스 손상을 일으킬 수 있었던 과거의 몇 가지 버그(현재는 수정됨)가 있었습니다. 그리고 아직 발견되지 않은 버그가 몇 개 더 남아 있을 수도 있습니다. SQLite의 광범위한 테스트와 널리 퍼진 사용으로 인해 데이터베이스 손상을 초래하는 버그는 매우 모호한 경향이 있습니다. 애플리케이션이 SQLite 버그를 만날 가능성은 작습니다. 이를 설명하기 위해 2009년 4월 1일부터 2013년 4월 15일까지의 4년 동안 SQLite에서 발견된 모든 데이터베이스 손상 버그에 대한 설명이 아래에 제공됩니다. 이 설명은 독자에게 테스트 절차를 빠져나가 릴리스에 포함되는 SQLite 버그의 종류에 대한 직관적인 감각을 줄 것입니다.

#### 8.1. WAL 모드 데이터베이스에 쓸 때의 경쟁 조건 (Race Condition)

동일한 WAL 모드 데이터베이스에 두 개 이상의 데이터베이스 커넥션이 별도의 스레드나 프로세스에서 열려 있고, 두 커넥션이 동시에 쓰기를 시도하거나 체크포인트를 실행하려고 하면 데이터베이스 파일을 손상시킬 수 있는 경쟁 조건이 존재합니다. 이것이 WAL 리셋(WAL-reset) 버그입니다. 이는 3.7.0부터 3.51.2까지의 모든 SQLite 버전에 존재했습니다. 자세한 내용은 WAL 리셋 버그 문서를 참조하십시오.

#### 8.2. 만료된 표현식 인덱스 (Stale Expression Indexes)

"표현식 인덱스"는 표현식이나 `VIRTUAL` 생성 컬럼을 참조하는 인덱스입니다. 표현식 인덱스는 결정론적(Deterministic) 함수만 사용해야 합니다. 그러나 데이터베이스를 다른 플랫폼으로 이동하거나 SQLite 버전을 변경할 때, 결정론적이라고 여겨졌던 이러한 함수 중 하나가 출력을 미묘하게 변경하는 경우가 있습니다. 이로 인해 인덱스가 손상된 것처럼 보일 수 있습니다. 자세한 내용은 만료된 표현식 인덱스 문서를 참조하십시오.

#### 8.3. 데이터베이스 축소로 인한 허위 손상 보고

데이터베이스가 SQLite 버전 3.7.0 이상에 의해 기록된 다음, 데이터베이스 파일의 크기가 감소하는 방식으로 SQLite 버전 3.6.23 이하에 의해 다시 기록되면, 다음번에 SQLite 버전 3.7.0이 데이터베이스 파일에 액세스할 때 데이터베이스 파일이 손상되었다고 보고할 수 있습니다. 그러나 데이터베이스 파일은 실제로 손상된 것이 아닙니다. 버전 3.7.0이 손상 감지에 너무 과도하게 열성적이었을 뿐입니다.
이 문제는 2011년 2월 20일에 수정되었습니다. 이 수정은 SQLite 버전 3.7.6(2011-04-12)에 처음 나타납니다.

#### 8.4. 롤백 모드와 WAL 모드 간의 전환에 따른 손상

하나의 프로세스나 스레드에서 SQLite 데이터베이스를 WAL 모드로 전환하고 나가는 것을 반복하고 그 전환 사이에 `VACUUM` 명령을 실행하면, 데이터베이스 파일을 열어둔 다른 프로세스나 스레드가 데이터베이스가 변경되었다는 사실을 놓치게 만들 수 있습니다. 그 두 번째 프로세스나 스레드는 만료된 캐시를 사용하여 데이터베이스를 수정하려고 시도할 수 있으며 데이터베이스 손상을 일으킬 수 있습니다.
이 문제는 내부 테스트 중에 발견되었으며 필드에서는 관찰된 적이 없습니다. 이 문제는 2011년 1월 27일에 수정되었으며 버전 3.7.5에 반영되었습니다.

#### 8.5. 잠금을 획득하는 동안의 I/O 에러가 손상으로 이어짐

WAL 모드에서 공유 메모리에 대한 특정 잠금을 획득하려고 시도하는 동안 운영체제가 I/O 에러를 반환하면, SQLite가 캐시를 리셋하는 데 실패할 수 있으며, 이로 인해 후속 쓰기가 시도될 경우 데이터베이스 손상으로 이어질 수 있습니다.
참고로 이 문제는 잠금을 획득하려는 시도가 I/O 에러를 초래한 경우에만 발생합니다. 잠금이 단순히 승인되지 않은 경우(다른 스레드나 프로세스가 이미 충돌하는 잠금을 보유하고 있기 때문에)에는 손상이 절대 발생하지 않습니다. 우리는 공유 메모리에 파일 잠금을 얻으려고 시도하는 동안 I/O 에러로 실패할 운영체제를 알지 못합니다. 따라서 이는 실제 문제라기보다는 이론적인 문제입니다. 말할 필요도 없이, 이 문제는 필드에서 관찰된 적이 없습니다. 이 문제는 I/O 에러를 시뮬레이션하는 테스트 하네스에서 SQLite의 스트레스 테스트를 수행하는 동안 발견되었습니다.
이 문제는 SQLite 버전 3.7.3을 위해 2010년 9월 20일에 수정되었습니다.

#### 8.6. 프리 페이지 리스트(Free Page List)에서 데이터베이스 페이지가 누수됨

SQLite 데이터베이스에서 콘텐츠가 삭제되면 더 이상 사용되지 않는 페이지가 프리 리스트에 추가되고 후속 인서트에 의해 추가되는 콘텐츠를 보유하기 위해 재사용됩니다. 버전 3.6.16부터 3.7.2까지 존재했던 SQLite의 버그는 `incremental_vacuum`이 사용될 때 프리 리스트에서 페이지가 사라지게 만들 수 있었습니다. 이는 데이터 손실을 일으키지는 않습니다. 하지만 데이터베이스 파일이 필요 이상으로 커지는 결과를 초래합니다. 그리고 `integrity_check` 프라그마가 프리 리스트에서 페이지가 누락되었다고 보고하게 만듭니다.
이 문제는 SQLite 버전 3.7.2를 위해 2010년 8월 23일에 수정되었습니다.

#### 8.7. 3.6과 3.7의 교대 쓰기에 따른 손상

SQLite 버전 3.7.0은 SQLite 데이터베이스 파일 형식에 여러 가지 새로운 향상된 기능(WAL을 포함하되 이에 국한되지 않음)을 도입했습니다. 3.7.0 릴리스는 이러한 새로운 기능들을 검증하는 정비 릴리스였습니다. 우리는 문제를 발견할 것으로 예상했고 실망하지 않았습니다.
데이터베이스가 원래 SQLite 버전 3.7.0을 사용하여 생성된 다음, 데이터베이스 파일의 크기가 증가하는 방식으로 SQLite 버전 3.6.23.1에 의해 기록되고, 다시 SQLite 버전 3.7.0에 의해 기록되면 데이터베이스 파일이 손상될 수 있었습니다.
이 문제는 SQLite 버전 3.7.1을 위해 2010년 8월 4일에 수정되었습니다.

#### 8.8. 윈도우 시스템의 복구 시 경쟁 조건

SQLite 버전 3.7.16.2는 윈도우 시스템의 잠금 로직에 있는 미묘한 경쟁 조건을 수정합니다. 이전 프로세스가 트랜잭션 중간에 크래시되어 데이터베이스 파일에 복구가 필요하고 둘 이상의 프로세스가 동시에 해당 데이터베이스를 열려고 시도할 때, 경쟁 조건으로 인해 그 프로세스 중 하나가 복구가 이미 완료되었다는 잘못된 표시를 받아 복구를 먼저 실행하지 않고 데이터베이스 파일을 계속 사용하게 될 수 있습니다. 해당 프로세스가 파일에 쓰면 파일이 손상될 수 있습니다. 이 경쟁 조건은 2004년으로 거슬러 올라가는 윈도우용 SQLite의 모든 이전 버전에 분명히 존재해 왔습니다. 하지만 경쟁 창이 매우 좁았습니다. 현실적으로 말해서, 두 개의 분리된 코어에서 동시에 복구를 실행하기 위해 두 개의 프로세스를 시작하는 고성능 멀티코어 머신이 필요합니다. 이 결함은 윈도우 시스템에만 있었고 포식스 OS 인터페이스에는 영향을 미치지 않았습니다.

#### 8.9. 중첩 트랜잭션에 사용되는 세컨더리 저널의 바운더리 밸류 에러 (Boundary Value Error)

`SAVEPOINT`를 사용하여 중첩 트랜잭션(Nested transaction)이 시작될 때, SQLite는 내부 트랜잭션을 롤백해야 할 경우를 대비하여 중첩 트랜잭션에 대한 변경 사항을 추적하기 위해 세컨더리 롤백 저널(Secondary rollback journal)을 사용합니다. 세컨더리 저널은 프로그램 크래시나 정전으로 인한 손상으로부터 데이터베이스를 보호하는 데 관여하지 않습니다. 세컨더리 저널은 중첩 트랜잭션의 내부 트랜잭션을 롤백할 때만 활성화됩니다.
이러한 세컨더리 저널은 메모리에 유지되거나 디스크의 임시 파일로 유지될 수 있습니다. 기본 동작은 디스크에 저장하는 것입니다. 하지만 이는 `-DSQLITE_TEMP_STORE` 컴파일 타임 옵션을 사용하거나 런타임에 `PRAGMA temp_store` 문을 사용하여 변경할 수 있습니다. 이 버그는 세컨더리 저널이 메모리에 유지될 때만 발생합니다.
SQLite 버전 3.35.0(2021-03-12)에서는 새로운 최적화가 추가되어 SQLite가 세컨더리 저널을 메모리에 유지할 때 더 적은 메모리를 사용하게 되었습니다. 불행하게도 새로운 로직의 바운더리 체크가 잘못 코딩되었습니다. `<` 연산자여야 했던 것이 `<=` 연산자로 코딩되었습니다. 이 에러는 세컨더리 저널이 롤백될 경우 일시적으로 일치하지 않는 상태로 진입하게 만들 수 있습니다. 추가적인 변경 사항이 생성되고 결국 외부 트랜잭션이 커밋되면 데이터베이스가 일치하지 않는 상태로 남을 수 있습니다.
이 문제는 퍼저(Fuzzer)를 사용하여 SQLite에서 버그를 찾으려고 시도하던 독립 연구원에 의해 발견되었습니다. 퍼저는 세컨더리 저널의 내부 상태를 검증하는 데 사용되는 `assert()` 문에서 실패를 발견했습니다. 이 버그는 SQLite에서 `assert()` 문을 집중적으로 사용하지 않았더라면 수년 동안 발견되지 않은 채 지나갔을 정도로 충분히 모호한 코너 케이스였습니다.

### dbt(Data Build Tool) 개념 이해

- [Understanding ELT extract, load, transform  dbt Labs](https://www.getdbt.com/blog/extract-load-transform)

dbt의 역할을 명확히 이해하려면, 전체 데이터 파이프라인(ELT)에서 **dbt가 '무엇을 하지 않는가'**를 아는 것이 가장 중요합니다.

| 레이어 (Layer) | 주요 역할 | 대표 도구 | dbt와의 차이점 (dbt가 아닌 이유) |
| :--- | :--- | :--- | :--- |
| **1. 추출/적재 (E & L)** | 외부(DB, API, 앱)에서 원본 데이터를 가져와 데이터 웨어하우스에 **'그대로(Raw)' 부어 넣음** | Fivetran, Airbyte, Stitch | dbt는 **데이터를 밖에서 가져오지 않습니다.** 데이터 웨어하우스 안에 이미 들어온 데이터만 다룹니다. |
| **2. 저장/연산 (Storage/Compute)** | 대용량 데이터를 저장하고, SQL 쿼리를 받아 **실제 물리적인 연산을 수행함** | Snowflake, BigQuery, Redshift | dbt 자체에는 데이터 저장소나 연산 엔진이 없습니다. 연산은 웨어하우스가 하며, dbt는 똑똑한 쿼리를 '작성해서 던져줄' 뿐입니다. |
| **3. 변환 (T - dbt의 영역)** | 웨어하우스 내의 Raw 데이터를 조인, 필터링, 집계하여 **비즈니스 목적에 맞는 테이블/뷰로 가공함** | **dbt**, Dataform | dbt는 오직 이 'T' 영역에서만 활동합니다. 데이터를 분석하기 좋게 잘 다듬는(Build) 공장장 역할입니다. |
| **4. 오케스트레이션 (Orchestration)** | 데이터 추출 -> 적재 -> 변환의 **전체 파이프라인 스케줄링 및 모니터링** | Apache Airflow, Dagster, Prefect | dbt는 모델 간의 순서(DAG)는 알지만, "매일 아침 9시에 Fivetran을 돌리고 끝나면 dbt를 돌려라" 같은 외부 시스템 통제는 Airflow가 합니다. |
| **5. 분석 및 시각화 (BI)** | 최종 가공된 데이터를 바탕으로 대시보드를 만들고 **의사결정을 도움** | Tableau, Looker, Metabase | dbt는 그래프를 그리지 않습니다. BI 도구가 무거운 쿼리 없이 빠르게 그래프를 그릴 수 있도록 미리 계산된 밥상을 차려줍니다. |


전통적인 ETL 방식에서는 데이터를 추출해서 별도의 서버에서 변환한 뒤 DB에 밀어넣었습니다. 하지만 클라우드 데이터 웨어하우스(BigQuery 등)의 연산 능력이 워낙 강력해지면서, 일단 무식하게 데이터를 다 밀어 넣고(EL), **웨어하우스 내부에서 SQL로 데이터를 요리(T)하는 것이 대세**가 되었습니다.
dbt는 바로 이 **"웨어하우스 내부의 SQL 기반 데이터 요리(Transformation)"** 과정을 소프트웨어 개발하듯 우아하고 체계적으로 관리할 수 있게 해주는 도구입니다.

### Jinja의 역할과 유래 (dbt 맥락에서)

Jinja(정확히는 Jinja2)는 원래 **Python 웹 프레임워크(예: Flask, Django)에서 동적인 웹페이지(HTML)를 만들기 위해 개발된 템플릿 엔진**입니다. HTML의 고정된 틀(Template) 안에 `{{ user.name }}`이나 `{% for item in items %}` 같은 코드를 삽입하면, 파이썬 서버가 사용자의 상황에 맞게 데이터를 채워 넣어 최종 HTML 문서로 렌더링해 주는 역할을 했습니다.

**dbt에서의 역할: "정적인 SQL에 프로그래밍 날개를 달아주다"**
SQL은 데이터를 조회하는 데 최적화되어 있지만, 기본적으로 **정적(Static)**인 언어입니다. 변수를 선언하거나, for문을 돌리거나, 함수(Macro)를 만들어 재사용하는 것이 매우 어렵거나 불가능합니다. dbt는 Jinja를 도입하여 이 한계를 돌파했습니다. (즉, HTML을 동적으로 만들던 Jinja로 **SQL을 동적으로 생성**하는 것입니다.)

**구체적인 역할과 예시:**
1.  **의존성 관리 (`{{ ref() }}`):**
    테이블 이름을 하드코딩하지 않고 Jinja 함수를 사용해 참조합니다. dbt는 이를 컴파일 타임에 분석해 어떤 테이블을 먼저 만들어야 하는지(실행 순서) 알아냅니다.
2.  **반복문 (`{% for %}`):**
    결제 수단(신용카드, 계좌이체, 쿠폰 등)에 따라 컬럼을 여러 개(Pivot) 만들어야 할 때, SQL 복붙 없이 for문으로 짧게 처리할 수 있습니다.
    ```sql
    {% set payment_methods = ['credit_card', 'bank_transfer', 'coupon'] %}
    SELECT
        order_id,
        {% for method in payment_methods %}
        SUM(case when payment_method = '{{ method }}' then amount else 0 end) as {{ method }}_amount{% if not loop.last %},{% endif %}
        {% endfor %}
    FROM raw_payments
    GROUP BY 1
    ```
3.  **환경 분리 및 조건문 (`{% if %}`):**
    개발(Dev) 환경에서는 데이터가 너무 많아 테스트가 느릴 때, Jinja 조건문을 활용해 "개발 환경일 때만 최근 3일 치 데이터만 가져와라"라는 로직을 SQL에 심을 수 있습니다.
4.  **매크로(Macro):**
    자주 쓰는 SQL 패턴(예: 센트 단위를 달러로 변환)을 파이썬 함수처럼 만들어 두고 모든 SQL 모델에서 불러와 재사용(`{{ cents_to_dollars('amount') }}`)할 수 있습니다.

## [MySQL :: MySQL 8.0 Reference Manual :: 12.10.1 Unicode Character Sets](https://dev.mysql.com/doc/refman/8.0/en/charset-unicode-sets.html)

> MySQL implements the xxx_unicode_ci collations according to the Unicode Collation Algorithm (UCA) described at http://www.unicode.org/reports/tr10/.

### 일본어 캐릭터 이름 컬럼 콜레이션(Collation) 설계

일본어 사용자의 정상적인 이름 작명(히라가나/가타카나, 탁점, 요음/촉음 구분)은 허용하면서, 전/반각 눈속임을 통한 사칭을 막으려면 콜레이션 서픽스(`_ai`/`_as`/`_ci`/`_cs`/`_ks`)를 정확히 이해해야 합니다.

| 서픽스 | 의미 | 일본어에서의 역할 | 기본값(`_ai_ci`) 동작 |
| :--- | :--- | :--- | :--- |
| **`_as`** (Accent-sensitive) | 악센트 구분 | UCA에서 탁점(゛)·반탁점(゜)을 알파벳 악센트(e vs é)와 동일 수준으로 취급. `_as` 없으면 ハハ = ババ = パパ | `_ai`: 탁점 무구분 → UNIQUE 컬럼에서 동시 생성 불가 |
| **`_cs`** (Case-sensitive) | 대소문자 구분 | 일본어에서는 **요음/촉음(작은 가나 っ, ょ 등)과 일반 가나** 구분을 담당 | `_ci`: つ = っ 로 인식 |
| **`_ks`** (Kana-sensitive) | 가나 구분 | 히라가나 あ 와 가타카나 ア 를 다른 글자로 취급. 서픽스가 없으면 동일 문자 취급 | 없음: あ = ア |

### JIS 제4수준 한자 (U+20000 ~ U+2A6DF)

- 유니코드 **보조 평면(Plane 2)** 에 존재하며 UTF-8 인코딩 시 4바이트 필요.
- 과거 `utf8`(별칭 `utf8mb3`)은 3바이트 한계로 저장 시 오류 발생.
- **`utf8mb4`** 는 4바이트 영역을 전부 커버하므로 정상 저장·인식.

### 전각(Zenkaku) / 반각(Hankaku) 취급

유니코드 코드 포인트 자체가 다른 별개의 문자입니다(예: 반각 `1` = `U+0031`, 전각 `１` = `U+FF11`). 스토리지 상으로도 다른 바이트로 저장되지만, 콜레이션이 이를 어떻게 비교하느냐가 문제입니다.

| Collation | 전각 vs 반각 | 사칭 위험 |
| :--- | :--- | :--- |
| `utf8mb4_0900_ai_ci` 등 너비 무구분 | 동일 문자로 인식 (`'1' = '１'` → TRUE) | UNIQUE 제약 시 중복으로 차단됨 → **사칭 방어에 유리** |
| `utf8mb4_bin`, 일본어 전용 `_as_cs` 계열 | 다른 문자로 인식 | 전각으로 타 유저 ID 사칭 가능 |

### 캐릭터 이름 컬럼 최종 제안

```sql
-- 테이블 생성 예시 (MySQL 8.0 이상 기준)
CREATE TABLE characters (
    id INT AUTO_INCREMENT PRIMARY KEY,
    -- 일본어 전용 악센트·대소문자·가나 구분 콜레이션 적용
    char_name VARCHAR(30) CHARACTER SET utf8mb4 COLLATE utf8mb4_ja_0900_as_cs_ks NOT NULL,
    UNIQUE KEY uq_char_name (char_name)
);
```

- **허용:** 히라가나/가타카나 구분(`_ks`), 탁점 유무(`_as`), 요음/촉음 유무(`_cs`) 모두 서로 다른 문자로 처리되어 정상적인 이름 동시 생성 가능.
- **차단:** `utf8mb4_ja_0900_as_cs_ks` 는 너비(width) 는 여전히 무구분이므로, 반각 영문/숫자와 전각 영문/숫자를 동일 문자로 취급 → 전각 사칭 UNIQUE 위반으로 자동 차단.
