* **오류:** `sb.AppendFormat("{1:X04}...", arg1, arg2...)` 호출 시 예외 발생.
* **원인:** 1.  포맷 플레이스홀더(`{3}`, `{4}`) 개수보다 전달된 인자가 적음.
2.  C# 포맷 인덱스는 **1이 아닌 0부터 시작**해야 하는데 `{1}`부터 시작함.

### 8위. Equals와 GetHashCode의 불일치 (1)

* **오류:** `GetHashCode`에서는 `Legs` 속성을 사용해 해시를 계산하지만, `Equals`에서는 이를 비교하지 않음.
* **원인:** 특히 이 사례에서는 `Dictionary`를 만들 때 자기 자신의 컬렉션을 다시 참조하여 "컬렉션의 요소가 같은 컬렉션에 있는지" 확인하는 무의미한 로직이 포함되었습니다.

```cs
public override int GetHashCode()
{
  unchecked
  {
    var hashCode = Definition.GetHashCode();
    var arr = new int[Legs.Count];
    for (int i = 0; i < Legs.Count; i++)
    {
      arr[i] = Legs[i].GetHashCode();
    }

    Array.Sort(arr);

    for (int i = 0; i < arr.Length; i++)
    {
      hashCode = (hashCode * 397) ^ arr[i];
    }

    return hashCode;
  }
}

public override bool Equals(object obj)
{
    ....

    return Equals((OptionStrategyDefinitionMatch) obj);
}
```

### 7위. 속성이 누락된 Equals 구현

* **오류:** `Equals` 메서드에서 `Min` 속성만 두 번 비교하고 `Max` 비교를 누락함.
* **분석:** `Equals(Min, other.Min) && Equals(Min, other.Min)`과 같이 작성된 코드입니다. `GetHashCode`에는 `Max`가 포함되어 있어 분석기가 불일치를 감지했습니다.

### 6위. 도달할 수 없는 비트 연산 코드 (Switch Pattern)

### 5위. 값 형식(Struct)의 ReferenceEquals 사용

* **오류:** `struct StackValue`의 `Equals` 메서드 내에서 `ReferenceEquals(this, obj)` 호출.
* **원인:** `this`가 값 형식이므로 `ReferenceEquals`에 전달되는 순간 **박싱(Boxing)**이 발생합니다. 힙에 새로 생성된 참조는 절대 같을 수 없으므로 이 비교는 항상 `false`를 반환합니다.

### 4위. 익명 함수를 이용한 이벤트 구독 해제 실패

* **오류:** `Event -= () => Action();` 형태의 코드.
* **원인:** 익명 함수는 선언될 때마다 **새로운 델리게이트 인스턴스**를 생성합니다. 구독할 때의 인스턴스와 해제할 때의 인스턴스가 다르므로 구독 해제가 작동하지 않습니다.

```cs
private static void SubscribeImmutablePathsInitialized()
{
  NotifyOnScopingReadiness?.Invoke();

  FileClassifier.Shared.OnImmutablePathsInitialized -= () =>
    NotifyOnScopingReadiness?.Invoke();
}
```

### 3위. 연산자 우선순위 혼동 (?? vs +)

* **오류:** `base.Size + nullableObj?.Size ?? 0`
* **원인:** `+` 연산자가 `??`보다 우선순위가 높습니다. 따라서 `(base.Size + null) ?? 0`이 되어 왼쪽 항이 `null`이면 결과가 항상 `0`이 됩니다. 의도한 대로라면 괄호를 써서 `base.Size + (nullableObj?.Size ?? 0)`이 되어야 합니다.

```cs
public override int Size =>   base.Size
                            + ChangeViewMessages?.Values.GetVarSize() ?? 0
                            + 1 + PrepareRequestMessage?.Size ?? 0
                            + PreparationHash?.Size ?? 0
                            + PreparationMessages?.Values.GetVarSize() ?? 0
                            + CommitMessages?.Values.GetVarSize() ?? 0;
```

### 2위. 논리 패턴 매칭의 함정 (not 0 or 1)

* **오류:** `itemCount is not 0 or 1`
* **원인:** 개발자는 "0도 아니고 1도 아님"을 의도했겠지만, C# 컴파일러는 이를 **`(is not 0) or (is 1)`**로 해석합니다. 1인 경우에도 참이 되는 논리적 오류입니다. 정확하게는 `is not (0 or 1)`로 써야 합니다.

```cs
protected void ChangeMode(OmnibarMode? oldMode, OmnibarMode newMode)

  var modeSeparatorWidth = 
    itemCount is not 0 or 1 
      ? _modesHostGrid.Children[1] is FrameworkElement frameworkElement
        ? frameworkElement.ActualWidth
        : 0
      : 0;
  ....
}
```

## [IO is no longer the bottleneck](https://stoppels.ch/2022/11/27/io-is-no-longer-the-bottleneck.html)

Ben Hoyt's blog refers to an [earlier post](https://benhoyt.com/writings/count-words/) which includes a faster C version of the word frequency counter. I compiled optimized.c with GCC 12, using -O3 -march=native flags, and ran it on the 425MB input file (100 copies of the King James Version of the Bible).

이 글은 "현대 시스템에서 데이터 처리의 병목은 디스크 읽기(I/O)가 아니라 CPU 성능"이라는 주장을 검증하기 위해, 단어 수 세기(Word Count) 프로그램을 극한으로 최적화해 본 과정을 담고 있습니다. (2022. 11. 27)

### 1. 배경: 현대 디스크의 속도

최근 벤 호이트(Ben Hoyt)는 "순차 읽기(Sequential Read) 속도는 엄청나게 빨라진 반면, CPU 속도는 정체되었기 때문에 I/O가 병목이라는 믿음은 더 이상 사실이 아니다"라고 주장했습니다.

* **저자의 측정 결과:** * **Cold Cache(디스크에서 직접 읽기):** 약 1.6 GB/s
* **Warm Cache(메모리에 로드된 상태):** 약 12.8 GB/s

### 2. 최적화된 C 코드의 한계

벤 호이트가 작성한 최적화된 C 언어 단어 빈도 계산 코드를 테스트해 본 결과, 웜 캐시 상태에서도 **278 MB/s**라는 실망스러운 속도가 나왔습니다. 디스크 읽기 속도(1.6 GB/s)에 한참 못 미치는 수치입니다.

* **원인 분석:** 루프 내에 많은 분기문(Branch)이 있어 컴파일러의 **자동 벡터화(Auto-vectorization)**를 방해하고 있었습니다.
* **개선 시도:** 소문자 변환 로직을 루프 밖으로 빼는 등 약간의 수정을 거쳐 **330 MB/s**까지 끌어올렸으나, 여전히 디스크 속도보다 5배나 느렸습니다.

### 3. 리눅스 기본 도구 `wc -w`의 성능

가장 단순한 작업인 단어 수 세기 전용 도구 `wc`는 어떨까요?

* **결과:** **245.2 MB/s**로 생각보다 매우 느렸습니다.
* **이유:** `wc`는 단순 공백(' ')뿐만 아니라 탭, 줄 바꿈, 로케일별 특수 문자 등을 모두 처리해야 하므로 범용적인 대신 느립니다.

### 4. 극한의 최적화: AVX2 벡터화

저자는 지난 10년간 비약적으로 발전한 CPU의 기능을 활용하기 위해 **SIMD(AVX2)**를 직접 사용하여 코드를 작성하기로 했습니다.

* **핵심 기술:**
1. **256비트 레지스터:** 한 번에 32개의 문자를 처리.
2. **VPCMPEQB:** 레지스터 내에서 공백 문자를 찾아 마스크(Mask)를 생성.
3. **PMOVMSKB & 비트 트릭:** 생성된 마스크를 32비트 정수로 변환한 뒤, 비트 연산(`ffs` 등)을 통해 단어의 시작과 끝을 빠르게 계산.
4. **루프 언롤링(Unrolling):** 루프를 4배로 풀어 한 번에 128바이트의 데이터를 처리.

- 결과적으로 Warm Cache 기준 **1.45 GB/s** CPU가 거의 100% 사용됨

### ### 결론

* **I/O는 더 이상 병목이 아니다:** 단일 스레드에서 가장 빠른 벡터화 명령어를 사용하여 단어 수만 세는 단순 작업을 수행하더라도, 겨우 **Cold Cache 디스크 읽기 속도(1.6 GB/s)의 약 90%** 정도만 따라잡을 수 있었습니다.
* **웜 캐시 기준:** CPU 처리 속도(1.45 GB/s)는 메모리에 올라온 데이터 읽기 속도(12.8 GB/s)의 **약 11%** 수준에 불과합니다.
* **시사점:** 이제 하드웨어(NVMe SSD 등)의 발전 속도가 소프트웨어와 CPU의 처리 능력을 앞질렀습니다. 복잡한 로직이 들어가는 프로그램이라면 병목은 십중팔구 I/O가 아닌 CPU에서 발생할 것입니다.

## [개발자들이 맥 쓰는 3가지 이유 (비싼데도) - YouTube](https://www.youtube.com/watch?v=nkWdveHFrVI)

이 영상은 단순히 디자인이나 배터리 효율 때문이 아니라, 개발자들이 맥(macOS)을 선호할 수밖에 없는 **운영체제(OS)의 아키텍처적 본질**에 대해 설명하고 있습니다.

**1. 완성도 높은 '네이티브 유닉스(Unix)' 환경**

* **서버와의 일관성:** macOS는 태생이 유닉스 기반입니다. 따라서 개발자들이 실제로 서비스를 운영하는 리눅스 서버와 시스템 핵심 동작 방식이 거의 같습니다 `[00:00:21]`.
* **네이티브 아키텍처:** 윈도우는 WSL(Windows Subsystem for Linux)을 통해 격차를 줄이려 하지만, 이는 결국 호환성을 위한 '레이어'일 뿐입니다. 반면 맥은 시스템 코어부터 유닉스 도구와 동작 방식을 네이티브로 갖추고 있어 훨씬 강력하고 일관된 개발 환경을 제공합니다 `[00:00:44]`.

**2. 모듈화된 설정 관리 체계 (레지스트리 없음)**

## [nfd2nfc/docs/README.ko.md at main · elgar328/nfd2nfc · GitHub](https://github.com/elgar328/nfd2nfc/blob/main/docs/README.ko.md)

NFD 파일명을 실시간으로 감지해 NFC로 변환하는 macOS 도구입니다. 다른 OS와 파일을 주고받을 때 생기는 파일명 깨짐 문제를 해결합니다.

유니코드에는 한글, 악센트 문자(é, ü, ñ), 일본어(が, ぱ) 같은 조합 문자를 표현하는 두 가지 정규화 방식이 있습니다:

- **NFC(Normal Form Composed)** (조합형): 하나의 코드포인트 — `가` = `U+AC00`
- **NFD(Normal Form Decomposed)** (분해형): 기본 문자 + 결합 문자 — `가` = `U+1100 U+1161`

macOS는 파일명을 NFD로 저장하지만, Windows와 Linux는 NFC를 사용합니다. 이 차이 때문에 다른 OS와 파일을 주고받으면 다음과 같은 문제가 생깁니다:

- 한글이 자모로 풀어져서 표시됨 (예: 한글 → ㅎㅏㄴㄱㅡㄹ)
- 악센트 문자가 잘못 표시되거나 정렬이 깨짐
- Git이 같은 파일을 변경된 것으로 인식
- 클라우드 동기화나 압축 해제 시 경로를 찾지 못함

**nfd2nfc**는 TUI와 백그라운드 감시 서비스를 통해 이 문제를 자동으로 해결합니다.

### 오픈소스의 성장과 구조적 한계

오픈소스 프로젝트는 대개 작고 단순한 형태로 시작해 점진적으로 확장된다. 사용자가 늘어나면 기능이 추가되고, 기존 사용자들을 위해 하위 호환성이 유지된다. 이러한 과정 속에서 프로젝트는 점점 거대한 구조로 성장한다. 이것이 오랫동안 오픈소스의 강점이었다. 누구나 기여할 수 있고, 축적된 개선이 자산이 되었다.

그러나 시간이 지나면 문제가 발생한다. 기반 인프라는 낡고, 구조적 부채는 쌓이며, 초기 설계의 한계가 드러난다. 프로젝트는 이미 거대한 배가 되어 방향을 틀기 어려워진다. 이를 해결하기 위해 두 가지 방식이 등장한다. 하나는 기존 코드를 기반으로 다른 길을 가는 포크(fork)이고, 다른 하나는 API 계약만 유지한 채 내부를 완전히 다시 구현하는 전면 재작성(total rewrite)이다. 예를 들어 Redpanda는 Kafka와 호환되지만 내부는 완전히 새로 작성된 사례다. 또한 S3 호환 API들이 널리 퍼진 것 역시 동일한 현상이다.

### 상업적 오픈소스의 인센티브 충돌

완전히 비영리적인 오픈소스 프로젝트라면 포크나 재작성은 자연스러운 흐름일 수 있다. 누군가 다른 방향으로 가더라도 본질적인 이해관계 충돌이 없다. 그러나 상업적 오픈소스는 다르다. 겉으로는 이타적 가치와 공동체 기여를 강조하지만, 동시에 기업으로서 생존과 수익이라는 현실적 인센티브를 갖고 있다.

많은 기업이 장기적 신뢰를 위해 단기적 이익을 포기하기도 한다. 하지만 완전히 이타적일 수는 없다. 명확한 API와 좋은 문서, 강력한 테스트는 사용자 신뢰를 높이지만, 동시에 경쟁자가 이를 기반으로 재구현하기 쉽게 만든다. 바로 이 지점에서 긴장이 발생한다.

### Cloudflare와 Next.js 사례

이 현상은 Cloudflare와 Next.js 사례에서 잘 드러난다. Next.js는 오랜 시간에 걸쳐 문서화, API 계약 정의, 테스트 스위트 구축에 막대한 투자를 해왔다. 이는 사용자 신뢰를 구축하는 핵심 자산이었다.

그러나 Cloudflare는 AI를 활용해 Next.js와 호환되는 대안을 빠르게 구축했다. 기존 사용자 기반이 없기 때문에 하위 호환성 부담도 없었다. 최신 API 표면만 지원하면 되었고, 내부 구현은 더 가볍고 현대적인 도구 위에 재구성할 수 있었다. 테스트 통과율 94%라는 수치는 문서 이상의 가치가 테스트에 있음을 보여준다. 말로 설명된 기능보다, 자동화된 검증 코드가 훨씬 더 강력한 재현 도구가 된 것이다.

### SQLite의 선택: 테스트는 비공개

흥미로운 대비 사례로 SQLite가 있다. SQLite는 핵심 엔진 코드는 오픈소스로 공개하지만, 방대한 테스트 스위트는 공개하지 않는다. 공식 문서에 따르면 테스트 코드는 수천만 줄에 달하며, 이는 핵심 소스 코드보다 훨씬 크다.

이 전략은 일부에게 오픈소스 정신에 어긋나는 것으로 보일 수 있다. 그러나 다른 관점에서는 이것이 바로 프로젝트의 지속 가능성을 지키는 해자다. AI 시대에는 테스트가 곧 동작 정의서이자 품질 보증 체계이기 때문이다. 이를 공개하면 복제 비용이 급격히 낮아질 수 있다.

### 새로운 해자: 테스트와 계약

AI 시대에 코드 자체는 점점 상품화되고 있다. 대신 진정한 자산은 다음 세 가지로 이동한다.

첫째, 명확하게 정의된 API 계약.
둘째, 동작을 정확히 검증하는 테스트 스위트.
셋째, 인터페이스 표면과 그 일관성.

과거에는 코드를 직접 작성하고 이해하는 능력이 경쟁력이었다. 이제는 코드가 계약을 얼마나 정확히 만족하는지가 핵심이다. 그리고 그 계약을 가장 엄격하게 표현하는 것이 바로 테스트다.

### 오픈소스의 미래에 대한 질문

결국 상업적 오픈소스는 선택의 기로에 서 있다. API와 테스트를 완전히 공개하여 생태계를 확장할 것인가, 아니면 일부를 해자로 남겨 지속 가능성을 확보할 것인가. AI가 복제 비용을 낮출수록 이 고민은 더욱 심화될 것이다.

이 글의 핵심 메시지는 단순하다.
AI 시대에는 코드가 아니라 테스트가 가장 강력한 자산이 될 수 있다. 그리고 상업적 오픈소스는 그 사실을 점점 더 의식하게 될 것이다.

## C# struct default constructor

C#의 **`struct`**(값 타입)와 **기본 생성자(Parameterless Constructor)** 동작 방식은 C# 버전(특히 C# 10.0 전후)에 따라 큰 차이가 있습니다. **필드 초기화와 `default` 키워드의 관계**를 정리해 드립니다.

⚠️ 개발자가 "필드에서 초기화했으니 절대 `null`이 아닐 거야"라고 확신하고 코드를 짰을 때 **`NullReferenceException`**이 발생할 수 있습니다.

```csharp
public struct MyData
{
    public object Protector = new(); // C# 10+ 가능
}

// ... 

MyData[] list = new MyData[5];
// list[0].Protector 는 null입니다! 생성자가 호출되지 않았기 때문입니다.

```

### "어떻게 생성하느냐"에 따라 다릅니다.

`public object Obj = new();`와 같이 필드에서 직접 초기화(Field Initializer)를 하더라도, **어떤 방식으로 구조체를 만드느냐**에 따라 결과가 달라집니다.

| 생성 방식            | 코드 예시                     | 결과 (`Obj` 멤버)                    |
| ---                  | ---                           | ---                                  |
| **`new MyStruct()`** | `var s = new MyStruct();`     | **`new object()`** (인스턴스 할당됨) |
| **`default` 키워드** | `var s = default(MyStruct);`  | **`null`**                           |
| **배열 할당**        | `var arr = new MyStruct[10];` | **`null`** (모든 요소)               |

### 왜 이런 차이가 발생하나요?

#### **A. C# 10.0 미만 (구형 규격)**

과거에는 `struct` 내부에서 필드를 직접 초기화하거나 매개변수가 없는 생성자를 정의하는 것 자체가 **불가능**했습니다. 모든 구조체는 강제로 모든 비트를 `0`으로 미는 `default` 동작만 수행했기 때문입니다.

#### **B. C# 10.0 이상 (현대적 규격)**

이제 구조체에서도 **명시적 기본 생성자**와 **필드 초기화**가 허용됩니다.

* `new MyStruct()`를 호출하면 C# 컴파일러는 사용자가 정의한 필드 초기화 로직(또는 기본 생성자)을 실행합니다. 따라서 이때는 `new object()`가 할당됩니다.
* 하지만 **`default`**나 **배열 할당**은 성능 최적화를 위해 메모리 영역을 단순히 `0`으로 초기화(Zero-init)합니다. 이 과정에서는 **생성자 로직이 호출되지 않으므로** 참조 타입인 `object`는 `null`이 됩니다.

## dotnet-dump cli

`dotnet-dump`는 .NET 프로세스의 덤프를 수집하고 분석할 때 사용하는 필수 CLI 도구입니다. 특히 리눅스 환경이나 컨테이너 내부에서 발생한 메모리 누수, CPU 과부하 등을 진단할 때 유용합니다.

### 1. 수집 및 관리 명령어

| 명령어                            | 설명                                               | 비고                                                |
| :---                              | :---                                               | :---                                                |
| `dotnet-dump collect -p <PID>`    | 특정 프로세스의 덤프를 생성합니다.                 | `--type` 옵션으로 Full/Mini 선택 가능 (기본값 Full) |
| `dotnet-dump analyze <DUMP_FILE>` | 생성된 덤프 파일을 분석 모드로 엽니다.             | REPL 환경으로 진입                                  |
| `dotnet-dump ps`                  | 덤프를 뜰 수 있는 .NET 프로세스 목록을 확인합니다. |                                                     |

### 2. 분석 모드(`analyze`) 내부 주요 명령어
덤프 파일을 연 후 내부 REPL 환경에서 자주 사용하는 명령어들입니다.

#### **메모리 분석 관련**
* **`dumpheap -stat`**
    * 힙(Heap) 메모리에 있는 객체 통계를 출력합니다. 어떤 타입의 객체가 메모리를 많이 차지하는지 확인할 때 가장 먼저 사용합니다.
* **`dumpheap -mt <MethodTable_Address>`**
    * 특정 타입의 모든 인스턴스를 나열합니다. `-stat`에서 발견된 의심스러운 객체의 주소를 확인할 때 씁니다.
* **`gcroot <Object_Address>`**
    * 특정 객체가 왜 메모리에서 해제되지 않고 남아있는지(참조 체인)를 추적합니다. **메모리 누수 진단의 핵심**입니다.
* **`eeheap -gc`**
    * GC 힙의 구조와 세대별(Gen 0, 1, 2, LOH) 메모리 사용량을 확인합니다.

#### **스레드 및 호출 스택 관련**
* **`clrstack`**
    * 현재 스레드의 관리 코드(Managed Code) 호출 스택을 보여줍니다.
* **`threads`**
    * 프로세스 내의 모든 관리형 스레드 목록을 출력합니다.
* **`setthread <ID>`**
    * 분석할 스레드를 전환합니다.
* **`clrstack -all`**
    * 모든 스레드의 호출 스택을 한꺼번에 출력합니다. **데드락(Deadlock) 분석** 시 유용합니다.

#### **상태 및 변수 조회**
* **`dumpobj <Object_Address>`**
    * 특정 객체의 필드 값과 내부 데이터를 상세히 조회합니다.
* **`dumplocal`**
    * 현재 프레임의 지역 변수들을 확인합니다. (최적화된 빌드에서는 제한적일 수 있습니다.)

### 3. 실무 진단 시나리오 예시

#### **A. 메모리 누수가 의심될 때**
1.  `dumpheap -stat`으로 비정상적으로 개수가 많은 클래스 식별
2.  `dumpheap -mt <MT>`로 해당 객체 주소 하나 확보
3.  `gcroot <Address>`로 어떤 루트(Static 변수, Event Handler 등)가 붙들고 있는지 확인

#### **B. CPU 사용율이 100%일 때**
1.  `threads`로 활동량이 많은 스레드 확인
2.  `setthread`로 해당 스레드 이동 후 `clrstack`으로 현재 무한 루프나 무거운 연산을 수행 중인 메서드 확인

### 4. 팁: `dotnet-gcdump`와의 차이
* **`dotnet-dump`**: 프로세스의 전체 상태를 담기 때문에 파일 크기가 크지만, 로컬 변수나 스택 정보 등 모든 정보를 분석할 수 있습니다.
* **`dotnet-gcdump`**: 메모리 힙 그래프만 가볍게 수집합니다. 파일이 매우 작고 Visual Studio에서 열기 편하지만, 스택 정보는 알 수 없습니다.

#### 1. 특정 타입 이름으로 검색
정확한 전체 네임스페이스를 모르더라도 **부분 일치(Partial Match)** 검색이 가능합니다.

```bash
# 'UserService'가 포함된 모든 타입의 통계 확인
> dumpheap -stat -type UserService

# 특정 네임스페이스 하위의 객체들만 확인
> dumpheap -stat -type MyApp.Models.
```

#### 2. 분석 흐름 (이름 검색 -> 인스턴스 조회)
보통 이름을 통해 **MethodTable(MT)** 주소를 알아낸 뒤 상세 분석으로 들어갑니다.

* `dotnet-dump` 내의 검색은 기본적으로 대소문자를 구분하므로 주의해야 합니다.

1.  **통계 확인:** `dumpheap -stat -type MyClass`
    * 출력 결과의 가장 왼쪽 열에 있는 **MT (MethodTable)** 주소를 복사합니다.
2.  **해당 타입의 모든 인스턴스 주소 나열:** `dumpheap -mt <MT_주소>`
    * 실제 메모리에 할당된 객체들의 개별 주소가 출력됩니다.
3.  **특정 객체 내부 값 보기:** `dumpobj <객체_주소>`

## [High Availability vs. Fault Tolerance: Key Differences - The Couchbase Blog](https://www.couchbase.com/blog/ko/high-availability-vs-fault-tolerance/)

### 1. 고가용성 vs 내결함성 핵심 비교

두 전략 모두 시스템의 가동 시간을 극대화하는 것이 목적이지만, 장애 발생 시의 **'연속성'** 수준에서 차이가 납니다.

| 구분            | 고가용성 (High Availability)                   | 내결함성 (Fault Tolerance)                       |
| :---            | :---                                           | :---                                             |
| **목표**        | 장애 시 **신속한 복구**를 통한 다운타임 최소화 | 장애 시에도 **중단 없는** 운영 보장              |
| **작동 방식**   | 장애 감지 후 예비 시스템으로 전환 (Failover)   | 이중화된 시스템이 동시에 작동하여 즉각 대응      |
| **핵심 기술**   | 로드 밸런서, 자동 장애 조치, 모니터링          | 합의 알고리즘(Raft, Paxos), 자가 복구, 동기 복제 |

### 고가용성 (HA)
* **특징:** "99.999%(파이브 나인스)" 가동률을 지향하며, 단일 장애 지점(SPOF)을 제거하는 데 집중합니다.
* **주요 사례:** 전자상거래(매출 유지), 헬스케어(데이터 접근), 클라우드 서비스(SLA 준수).
* **한계:** 복제 과정에서 데이터 일관성(Consistency)과 가용성(Availability) 사이의 트레이드오프가 발생할 수 있습니다.
* **HA 특화 도구:** 로드 밸런서(Nginx, HAProxy), Route 53(DNS 장애 조치), 모니터링(Prometheus, Datadog).

### 내결함성 (FT)
* **특징:** 일부 구성 요소가 완전히 고장 나더라도 성능 저하 없이 전체 기능을 유지합니다.
* **주요 사례:** 금융 거래(결제 무결성), 항공우주(안전 직결), 국방(중단 없는 명령 체계).
* **한계:** 동일한 작업을 수행하는 중복 자원이 항상 가동되어야 하므로 리소스 낭비와 높은 구축 비용이 발생합니다.
* **FT 특화 도구:** 분산 합의 알고리즘(Zookeeper, etcd), 이벤트 스트리밍(Kafka - 메시지 버퍼링을 통한 데이터 손실 방지).

---

* **공통 도구:** NoSQL 데이터베이스(Couchbase, DynamoDB, MongoDB), 쿠버네티스(Kubernetes).

### "스펙트럼"의 차이

이 둘은 '연속성'이라는 스펙트럼 상에 놓여 있습니다.

- Low Availability: 장애 나면 **수동**으로 복구 (엔지니어 호출)
- High Availability (HA): 장애 나면 **자동**으로 예비 서버 가동 (수 초~수 분 내 복구)
- Fault Tolerance (FT): 장애가 나도 서비스는 미동도 없이 계속됨 (0초 복구)

### Q. 서비스 다운 시, 예비 서비스가 아니라 다운된 서비스를 재시작 후 밀린 메세지 큐를 처리하면서 상태 복구한다면 HA 인가 (이중화 없음)

질문하신 과정은 **"이중화(Redundancy)가 결여된 자가 복구형 고가용성(Self-healing HA)"** 전략입니다. 

그것도 **고가용성(HA)**의 범주에 포함됩니다. 다만, 아키텍처 패턴 중에서는 **'자가 복구(Self-Healing)'** 또는 **'콜드 리부트(Cold Reboot)'** 기반의 HA 전략으로 분류할 수 있습니다.

#### 1. "이중화가 없는데 왜 HA인가요?"

고가용성(HA)의 정의는 '단일 장애 지점을 제거'하는 것뿐만 아니라, **'장애 발생 시 서비스 가동 중지 시간(Downtime)을 최소화하여 가용성을 100%에 가깝게 유지하는 능력'** 전체를 의미하기 때문입니다.

프로세스가 자동으로 다시 살아나서 큐에 쌓인 데이터를 처리하기 시작한다면, 전체 시스템 관점에서는 **"잠시 지연은 있었지만 서비스가 중단되지 않고 유지됨"**으로 간주됩니다.

#### 2. 이 구조의 핵심: 탄력성 (Resilience)

사용자가 말씀하신 구조는 **메시지 큐(Message Queue)**가 중간에서 '상태(State)'와 '데이터'를 보호해주는 완충 역할을 하고 있습니다.

* **데이터 보존:** 서비스 프로세스는 죽었지만, 처리해야 할 '일(Message)'은 큐에 안전하게 저장되어 있습니다.
* **상태 복구:** 재시작 후 큐의 메시지를 다시 읽으며 장애 이전의 상태를 재구성합니다.

이것은 이중화(Redundancy)를 통한 즉각적인 전환은 아니지만, **시스템의 탄력성(Resilience)**을 극대화하여 최종적으로 가용성을 확보하는 방식입니다.

실무에서는 **쿠버네티스(Kubernetes)**의 'Liveness Probe'가 컨테이너를 자동으로 재시작해주는 메커니즘이 바로 이 원리를 기반으로 하며, 현대적인 클라우드 네이티브 환경에서 매우 보편적으로 사용되는 HA 기법 중 하나입니다.

### `less` 명령어의 유래와 등장 배경

`less`는 유닉스 및 유닉스 계열 시스템에서 텍스트 파일의 내용을 확인하기 위해 사용하는 **터미널 페이저(terminal pager)**입니다. 이 프로그램은 이전의 표준 페이저였던 `more`의 한계를 극복하기 위해 만들어졌습니다.

### 1. 시대적 배경: `more`의 한계
`less`가 등장하기 전, 유닉스 사용자들은 **`more`**라는 명령어를 사용했습니다. 당시 `more`는 이름 그대로 파일의 내용을 **"더 보여주는(more)"** 기능에 충실했지만, 몇 가지 결정적인 단점이 있었습니다.

* **단방향 이동:** `more`는 파일의 앞부분에서 뒷부분으로만 이동할 수 있었습니다. 이미 지나간 텍스트를 다시 보기 위해 위로 스크롤하는 기능이 없었거나 매우 제한적이었습니다.
* **전체 로딩 방식:** 대용량 파일을 읽을 때 파일 전체를 메모리에 올리거나 끝까지 읽어야 하는 경우가 있어 실행 속도가 느려질 때가 많았습니다.

### 2. `less`의 탄생과 위트 있는 이름
1983년에서 1985년 사이, **마크 누델만(Mark Nudelman)**은 `more`의 불편함을 해결하고자 새로운 페이저를 개발했습니다.

* **언어유희:** `less`라는 이름은 **"less is more"** (적은 것이 더 많은 것이다)라는 유명한 격언에서 따온 것입니다.
* **유래:** 당시 동료들이 "반대 방향(역방향)으로도 이동할 수 있는 `more`가 필요하다"고 농담처럼 이야기했고, 이에 마크 누델만은 `more`의 반대말인 `less`를 프로그램 이름으로 정했습니다. 유닉스 커뮤니티에서는 이를 두고 **"less is more, more or less"**라는 농담을 즐기기도 합니다.

개별 옵션을 매번 입력하기 번거로우므로, 보통 아래와 같이 조합하여 환경 변수로 등록해 사용합니다.

## Q: git 1인 개인 private 저장소라도 api key 등 민감 정보를 저장하는 것이 안전한지?

A: 보안은 **"현재의 비공개 상태"**를 믿는 것이 아니라, **"실수로 노출되어도 안전한 구조"**를 만드는 것에서 시작됩니다. 지금 즉시 민감 정보를 코드에서 분리하고 히스토리를 정리하는 것을 권장합니다.

결론부터 말씀드리면, **개인 1인용 Private 저장소라 할지라도 비밀번호나 API Key를 직접 커밋하는 것은 매우 위험한 습관입니다.** 보안 업계에서는 이를 "잠재적 사고(Incidental Exposure)"의 전단계로 간주합니다.

### 1. 계정 탈취 및 유출 위험
* **계정 해킹:** GitHub/GitLab 등 플랫폼 계정이 탈취될 경우, 공격자는 즉시 모든 Private 저장소의 소스 코드를 복제합니다. 이때 포함된 API Key는 그대로 공격자의 자산이 됩니다.
* **로컬 PC 분실/해킹:** 개발 장비를 분실하거나 로컬 환경이 악성코드에 감염되면, `.git` 히스토리에 남은 모든 민감 정보가 유출됩니다.

### 2. Git 히스토리의 영속성
* Git은 파일의 현재 상태뿐만 아니라 **과거의 모든 변경 이력**을 저장합니다. 나중에 코드를 수정해서 키를 삭제하더라도, 이전 커밋 기록(History)에는 해당 값이 그대로 남아 있습니다. 이를 완전히 지우려면 `git filter-repo`나 `BFG Repo-Cleaner` 같은 도구로 히스토리를 강제로 재작성해야 하는 번거로움이 발생합니다.

### 3. 실수에 의한 Public 전환
* 작업 중에 저장소 설정을 실수로 **Public으로 변경**하거나, 팀 프로젝트로 전환하기 위해 협업자를 초대하는 순간 민감 정보가 공개됩니다. 특히 Public으로 전환되는 찰나에 스캐닝 봇(Bot)들이 1초도 안 되어 키를 수집해 가기 때문에 즉각적인 금전적 피해(클라우드 비용 폭탄 등)로 이어질 수 있습니다.
