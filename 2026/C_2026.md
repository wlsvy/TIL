* **오류:** `sb.AppendFormat("{1:X04}...", arg1, arg2...)` 호출 시 예외 발생.
* **원인:** 1.  포맷 플레이스홀더(`{3}`, `{4}`) 개수보다 전달된 인자가 적음.
2.  C# 포맷 인덱스는 **1이 아닌 0부터 시작**해야 하는데 `{1}`부터 시작함.

### 8위. Equals와 GetHashCode의 불일치 (1)

* **오류:** `GetHashCode`에서는 `Legs` 속성을 사용해 해시를 계산하지만, `Equals`에서는 이를 비교하지 않음.
* **원인:** 특히 이 사례에서는 `Dictionary`를 만들 때 자기 자신의 컬렉션을 다시 참조하여 "컬렉션의 요소가 같은 컬렉션에 있는지" 확인하는 무의미한 로직이 포함되었습니다.

```cs
public override int GetHashCode()
{
  unchecked
  {
    var hashCode = Definition.GetHashCode();
    var arr = new int[Legs.Count];
    for (int i = 0; i < Legs.Count; i++)
    {
      arr[i] = Legs[i].GetHashCode();
    }

    Array.Sort(arr);

    for (int i = 0; i < arr.Length; i++)
    {
      hashCode = (hashCode * 397) ^ arr[i];
    }

    return hashCode;
  }
}

public override bool Equals(object obj)
{
    ....

    return Equals((OptionStrategyDefinitionMatch) obj);
}
```

### 7위. 속성이 누락된 Equals 구현

* **오류:** `Equals` 메서드에서 `Min` 속성만 두 번 비교하고 `Max` 비교를 누락함.
* **분석:** `Equals(Min, other.Min) && Equals(Min, other.Min)`과 같이 작성된 코드입니다. `GetHashCode`에는 `Max`가 포함되어 있어 분석기가 불일치를 감지했습니다.

### 6위. 도달할 수 없는 비트 연산 코드 (Switch Pattern)

### 5위. 값 형식(Struct)의 ReferenceEquals 사용

* **오류:** `struct StackValue`의 `Equals` 메서드 내에서 `ReferenceEquals(this, obj)` 호출.
* **원인:** `this`가 값 형식이므로 `ReferenceEquals`에 전달되는 순간 **박싱(Boxing)**이 발생합니다. 힙에 새로 생성된 참조는 절대 같을 수 없으므로 이 비교는 항상 `false`를 반환합니다.

### 4위. 익명 함수를 이용한 이벤트 구독 해제 실패

* **오류:** `Event -= () => Action();` 형태의 코드.
* **원인:** 익명 함수는 선언될 때마다 **새로운 델리게이트 인스턴스**를 생성합니다. 구독할 때의 인스턴스와 해제할 때의 인스턴스가 다르므로 구독 해제가 작동하지 않습니다.

```cs
private static void SubscribeImmutablePathsInitialized()
{
  NotifyOnScopingReadiness?.Invoke();

  FileClassifier.Shared.OnImmutablePathsInitialized -= () =>
    NotifyOnScopingReadiness?.Invoke();
}
```

### 3위. 연산자 우선순위 혼동 (?? vs +)

* **오류:** `base.Size + nullableObj?.Size ?? 0`
* **원인:** `+` 연산자가 `??`보다 우선순위가 높습니다. 따라서 `(base.Size + null) ?? 0`이 되어 왼쪽 항이 `null`이면 결과가 항상 `0`이 됩니다. 의도한 대로라면 괄호를 써서 `base.Size + (nullableObj?.Size ?? 0)`이 되어야 합니다.

```cs
public override int Size =>   base.Size
                            + ChangeViewMessages?.Values.GetVarSize() ?? 0
                            + 1 + PrepareRequestMessage?.Size ?? 0
                            + PreparationHash?.Size ?? 0
                            + PreparationMessages?.Values.GetVarSize() ?? 0
                            + CommitMessages?.Values.GetVarSize() ?? 0;
```

### 2위. 논리 패턴 매칭의 함정 (not 0 or 1)

* **오류:** `itemCount is not 0 or 1`
* **원인:** 개발자는 "0도 아니고 1도 아님"을 의도했겠지만, C# 컴파일러는 이를 **`(is not 0) or (is 1)`**로 해석합니다. 1인 경우에도 참이 되는 논리적 오류입니다. 정확하게는 `is not (0 or 1)`로 써야 합니다.

```cs
protected void ChangeMode(OmnibarMode? oldMode, OmnibarMode newMode)

  var modeSeparatorWidth = 
    itemCount is not 0 or 1 
      ? _modesHostGrid.Children[1] is FrameworkElement frameworkElement
        ? frameworkElement.ActualWidth
        : 0
      : 0;
  ....
}
```

## [IO is no longer the bottleneck](https://stoppels.ch/2022/11/27/io-is-no-longer-the-bottleneck.html)

Ben Hoyt's blog refers to an [earlier post](https://benhoyt.com/writings/count-words/) which includes a faster C version of the word frequency counter. I compiled optimized.c with GCC 12, using -O3 -march=native flags, and ran it on the 425MB input file (100 copies of the King James Version of the Bible).

이 글은 "현대 시스템에서 데이터 처리의 병목은 디스크 읽기(I/O)가 아니라 CPU 성능"이라는 주장을 검증하기 위해, 단어 수 세기(Word Count) 프로그램을 극한으로 최적화해 본 과정을 담고 있습니다. (2022. 11. 27)

### 1. 배경: 현대 디스크의 속도

최근 벤 호이트(Ben Hoyt)는 "순차 읽기(Sequential Read) 속도는 엄청나게 빨라진 반면, CPU 속도는 정체되었기 때문에 I/O가 병목이라는 믿음은 더 이상 사실이 아니다"라고 주장했습니다.

* **저자의 측정 결과:** * **Cold Cache(디스크에서 직접 읽기):** 약 1.6 GB/s
* **Warm Cache(메모리에 로드된 상태):** 약 12.8 GB/s

### 2. 최적화된 C 코드의 한계

벤 호이트가 작성한 최적화된 C 언어 단어 빈도 계산 코드를 테스트해 본 결과, 웜 캐시 상태에서도 **278 MB/s**라는 실망스러운 속도가 나왔습니다. 디스크 읽기 속도(1.6 GB/s)에 한참 못 미치는 수치입니다.

* **원인 분석:** 루프 내에 많은 분기문(Branch)이 있어 컴파일러의 **자동 벡터화(Auto-vectorization)**를 방해하고 있었습니다.
* **개선 시도:** 소문자 변환 로직을 루프 밖으로 빼는 등 약간의 수정을 거쳐 **330 MB/s**까지 끌어올렸으나, 여전히 디스크 속도보다 5배나 느렸습니다.

### 3. 리눅스 기본 도구 `wc -w`의 성능

가장 단순한 작업인 단어 수 세기 전용 도구 `wc`는 어떨까요?

* **결과:** **245.2 MB/s**로 생각보다 매우 느렸습니다.
* **이유:** `wc`는 단순 공백(' ')뿐만 아니라 탭, 줄 바꿈, 로케일별 특수 문자 등을 모두 처리해야 하므로 범용적인 대신 느립니다.

### 4. 극한의 최적화: AVX2 벡터화

저자는 지난 10년간 비약적으로 발전한 CPU의 기능을 활용하기 위해 **SIMD(AVX2)**를 직접 사용하여 코드를 작성하기로 했습니다.

* **핵심 기술:**
1. **256비트 레지스터:** 한 번에 32개의 문자를 처리.
2. **VPCMPEQB:** 레지스터 내에서 공백 문자를 찾아 마스크(Mask)를 생성.
3. **PMOVMSKB & 비트 트릭:** 생성된 마스크를 32비트 정수로 변환한 뒤, 비트 연산(`ffs` 등)을 통해 단어의 시작과 끝을 빠르게 계산.
4. **루프 언롤링(Unrolling):** 루프를 4배로 풀어 한 번에 128바이트의 데이터를 처리.

- 결과적으로 Warm Cache 기준 **1.45 GB/s** CPU가 거의 100% 사용됨

### ### 결론

* **I/O는 더 이상 병목이 아니다:** 단일 스레드에서 가장 빠른 벡터화 명령어를 사용하여 단어 수만 세는 단순 작업을 수행하더라도, 겨우 **Cold Cache 디스크 읽기 속도(1.6 GB/s)의 약 90%** 정도만 따라잡을 수 있었습니다.
* **웜 캐시 기준:** CPU 처리 속도(1.45 GB/s)는 메모리에 올라온 데이터 읽기 속도(12.8 GB/s)의 **약 11%** 수준에 불과합니다.
* **시사점:** 이제 하드웨어(NVMe SSD 등)의 발전 속도가 소프트웨어와 CPU의 처리 능력을 앞질렀습니다. 복잡한 로직이 들어가는 프로그램이라면 병목은 십중팔구 I/O가 아닌 CPU에서 발생할 것입니다.

## [개발자들이 맥 쓰는 3가지 이유 (비싼데도) - YouTube](https://www.youtube.com/watch?v=nkWdveHFrVI)

이 영상은 단순히 디자인이나 배터리 효율 때문이 아니라, 개발자들이 맥(macOS)을 선호할 수밖에 없는 **운영체제(OS)의 아키텍처적 본질**에 대해 설명하고 있습니다.

**1. 완성도 높은 '네이티브 유닉스(Unix)' 환경**

* **서버와의 일관성:** macOS는 태생이 유닉스 기반입니다. 따라서 개발자들이 실제로 서비스를 운영하는 리눅스 서버와 시스템 핵심 동작 방식이 거의 같습니다 `[00:00:21]`.
* **네이티브 아키텍처:** 윈도우는 WSL(Windows Subsystem for Linux)을 통해 격차를 줄이려 하지만, 이는 결국 호환성을 위한 '레이어'일 뿐입니다. 반면 맥은 시스템 코어부터 유닉스 도구와 동작 방식을 네이티브로 갖추고 있어 훨씬 강력하고 일관된 개발 환경을 제공합니다 `[00:00:44]`.

**2. 모듈화된 설정 관리 체계 (레지스트리 없음)**

## [nfd2nfc/docs/README.ko.md at main · elgar328/nfd2nfc · GitHub](https://github.com/elgar328/nfd2nfc/blob/main/docs/README.ko.md)

NFD 파일명을 실시간으로 감지해 NFC로 변환하는 macOS 도구입니다. 다른 OS와 파일을 주고받을 때 생기는 파일명 깨짐 문제를 해결합니다.

유니코드에는 한글, 악센트 문자(é, ü, ñ), 일본어(が, ぱ) 같은 조합 문자를 표현하는 두 가지 정규화 방식이 있습니다:

- **NFC(Normal Form Composed)** (조합형): 하나의 코드포인트 — `가` = `U+AC00`
- **NFD(Normal Form Decomposed)** (분해형): 기본 문자 + 결합 문자 — `가` = `U+1100 U+1161`

macOS는 파일명을 NFD로 저장하지만, Windows와 Linux는 NFC를 사용합니다. 이 차이 때문에 다른 OS와 파일을 주고받으면 다음과 같은 문제가 생깁니다:

- 한글이 자모로 풀어져서 표시됨 (예: 한글 → ㅎㅏㄴㄱㅡㄹ)
- 악센트 문자가 잘못 표시되거나 정렬이 깨짐
- Git이 같은 파일을 변경된 것으로 인식
- 클라우드 동기화나 압축 해제 시 경로를 찾지 못함

**nfd2nfc**는 TUI와 백그라운드 감시 서비스를 통해 이 문제를 자동으로 해결합니다.

### 오픈소스의 성장과 구조적 한계

오픈소스 프로젝트는 대개 작고 단순한 형태로 시작해 점진적으로 확장된다. 사용자가 늘어나면 기능이 추가되고, 기존 사용자들을 위해 하위 호환성이 유지된다. 이러한 과정 속에서 프로젝트는 점점 거대한 구조로 성장한다. 이것이 오랫동안 오픈소스의 강점이었다. 누구나 기여할 수 있고, 축적된 개선이 자산이 되었다.

그러나 시간이 지나면 문제가 발생한다. 기반 인프라는 낡고, 구조적 부채는 쌓이며, 초기 설계의 한계가 드러난다. 프로젝트는 이미 거대한 배가 되어 방향을 틀기 어려워진다. 이를 해결하기 위해 두 가지 방식이 등장한다. 하나는 기존 코드를 기반으로 다른 길을 가는 포크(fork)이고, 다른 하나는 API 계약만 유지한 채 내부를 완전히 다시 구현하는 전면 재작성(total rewrite)이다. 예를 들어 Redpanda는 Kafka와 호환되지만 내부는 완전히 새로 작성된 사례다. 또한 S3 호환 API들이 널리 퍼진 것 역시 동일한 현상이다.

### 상업적 오픈소스의 인센티브 충돌

완전히 비영리적인 오픈소스 프로젝트라면 포크나 재작성은 자연스러운 흐름일 수 있다. 누군가 다른 방향으로 가더라도 본질적인 이해관계 충돌이 없다. 그러나 상업적 오픈소스는 다르다. 겉으로는 이타적 가치와 공동체 기여를 강조하지만, 동시에 기업으로서 생존과 수익이라는 현실적 인센티브를 갖고 있다.

많은 기업이 장기적 신뢰를 위해 단기적 이익을 포기하기도 한다. 하지만 완전히 이타적일 수는 없다. 명확한 API와 좋은 문서, 강력한 테스트는 사용자 신뢰를 높이지만, 동시에 경쟁자가 이를 기반으로 재구현하기 쉽게 만든다. 바로 이 지점에서 긴장이 발생한다.

### Cloudflare와 Next.js 사례

이 현상은 Cloudflare와 Next.js 사례에서 잘 드러난다. Next.js는 오랜 시간에 걸쳐 문서화, API 계약 정의, 테스트 스위트 구축에 막대한 투자를 해왔다. 이는 사용자 신뢰를 구축하는 핵심 자산이었다.

그러나 Cloudflare는 AI를 활용해 Next.js와 호환되는 대안을 빠르게 구축했다. 기존 사용자 기반이 없기 때문에 하위 호환성 부담도 없었다. 최신 API 표면만 지원하면 되었고, 내부 구현은 더 가볍고 현대적인 도구 위에 재구성할 수 있었다. 테스트 통과율 94%라는 수치는 문서 이상의 가치가 테스트에 있음을 보여준다. 말로 설명된 기능보다, 자동화된 검증 코드가 훨씬 더 강력한 재현 도구가 된 것이다.

### SQLite의 선택: 테스트는 비공개

흥미로운 대비 사례로 SQLite가 있다. SQLite는 핵심 엔진 코드는 오픈소스로 공개하지만, 방대한 테스트 스위트는 공개하지 않는다. 공식 문서에 따르면 테스트 코드는 수천만 줄에 달하며, 이는 핵심 소스 코드보다 훨씬 크다.

이 전략은 일부에게 오픈소스 정신에 어긋나는 것으로 보일 수 있다. 그러나 다른 관점에서는 이것이 바로 프로젝트의 지속 가능성을 지키는 해자다. AI 시대에는 테스트가 곧 동작 정의서이자 품질 보증 체계이기 때문이다. 이를 공개하면 복제 비용이 급격히 낮아질 수 있다.

### 새로운 해자: 테스트와 계약

AI 시대에 코드 자체는 점점 상품화되고 있다. 대신 진정한 자산은 다음 세 가지로 이동한다.

첫째, 명확하게 정의된 API 계약.
둘째, 동작을 정확히 검증하는 테스트 스위트.
셋째, 인터페이스 표면과 그 일관성.

과거에는 코드를 직접 작성하고 이해하는 능력이 경쟁력이었다. 이제는 코드가 계약을 얼마나 정확히 만족하는지가 핵심이다. 그리고 그 계약을 가장 엄격하게 표현하는 것이 바로 테스트다.

### 오픈소스의 미래에 대한 질문

결국 상업적 오픈소스는 선택의 기로에 서 있다. API와 테스트를 완전히 공개하여 생태계를 확장할 것인가, 아니면 일부를 해자로 남겨 지속 가능성을 확보할 것인가. AI가 복제 비용을 낮출수록 이 고민은 더욱 심화될 것이다.

이 글의 핵심 메시지는 단순하다.
AI 시대에는 코드가 아니라 테스트가 가장 강력한 자산이 될 수 있다. 그리고 상업적 오픈소스는 그 사실을 점점 더 의식하게 될 것이다.

## C# struct default constructor

C#의 **`struct`**(값 타입)와 **기본 생성자(Parameterless Constructor)** 동작 방식은 C# 버전(특히 C# 10.0 전후)에 따라 큰 차이가 있습니다. **필드 초기화와 `default` 키워드의 관계**를 정리해 드립니다.

⚠️ 개발자가 "필드에서 초기화했으니 절대 `null`이 아닐 거야"라고 확신하고 코드를 짰을 때 **`NullReferenceException`**이 발생할 수 있습니다.

```csharp
public struct MyData
{
    public object Protector = new(); // C# 10+ 가능
}

// ... 

MyData[] list = new MyData[5];
// list[0].Protector 는 null입니다! 생성자가 호출되지 않았기 때문입니다.

```

### "어떻게 생성하느냐"에 따라 다릅니다.

`public object Obj = new();`와 같이 필드에서 직접 초기화(Field Initializer)를 하더라도, **어떤 방식으로 구조체를 만드느냐**에 따라 결과가 달라집니다.

| 생성 방식            | 코드 예시                     | 결과 (`Obj` 멤버)                    |
| ---                  | ---                           | ---                                  |
| **`new MyStruct()`** | `var s = new MyStruct();`     | **`new object()`** (인스턴스 할당됨) |
| **`default` 키워드** | `var s = default(MyStruct);`  | **`null`**                           |
| **배열 할당**        | `var arr = new MyStruct[10];` | **`null`** (모든 요소)               |

### 왜 이런 차이가 발생하나요?

#### **A. C# 10.0 미만 (구형 규격)**

과거에는 `struct` 내부에서 필드를 직접 초기화하거나 매개변수가 없는 생성자를 정의하는 것 자체가 **불가능**했습니다. 모든 구조체는 강제로 모든 비트를 `0`으로 미는 `default` 동작만 수행했기 때문입니다.

#### **B. C# 10.0 이상 (현대적 규격)**

이제 구조체에서도 **명시적 기본 생성자**와 **필드 초기화**가 허용됩니다.

* `new MyStruct()`를 호출하면 C# 컴파일러는 사용자가 정의한 필드 초기화 로직(또는 기본 생성자)을 실행합니다. 따라서 이때는 `new object()`가 할당됩니다.
* 하지만 **`default`**나 **배열 할당**은 성능 최적화를 위해 메모리 영역을 단순히 `0`으로 초기화(Zero-init)합니다. 이 과정에서는 **생성자 로직이 호출되지 않으므로** 참조 타입인 `object`는 `null`이 됩니다.

## dotnet-dump cli

`dotnet-dump`는 .NET 프로세스의 덤프를 수집하고 분석할 때 사용하는 필수 CLI 도구입니다. 특히 리눅스 환경이나 컨테이너 내부에서 발생한 메모리 누수, CPU 과부하 등을 진단할 때 유용합니다.

### 1. 수집 및 관리 명령어

| 명령어                            | 설명                                               | 비고                                                |
| :---                              | :---                                               | :---                                                |
| `dotnet-dump collect -p <PID>`    | 특정 프로세스의 덤프를 생성합니다.                 | `--type` 옵션으로 Full/Mini 선택 가능 (기본값 Full) |
| `dotnet-dump analyze <DUMP_FILE>` | 생성된 덤프 파일을 분석 모드로 엽니다.             | REPL 환경으로 진입                                  |
| `dotnet-dump ps`                  | 덤프를 뜰 수 있는 .NET 프로세스 목록을 확인합니다. |                                                     |

### 2. 분석 모드(`analyze`) 내부 주요 명령어
덤프 파일을 연 후 내부 REPL 환경에서 자주 사용하는 명령어들입니다.

#### **메모리 분석 관련**
* **`dumpheap -stat`**
    * 힙(Heap) 메모리에 있는 객체 통계를 출력합니다. 어떤 타입의 객체가 메모리를 많이 차지하는지 확인할 때 가장 먼저 사용합니다.
* **`dumpheap -mt <MethodTable_Address>`**
    * 특정 타입의 모든 인스턴스를 나열합니다. `-stat`에서 발견된 의심스러운 객체의 주소를 확인할 때 씁니다.
* **`gcroot <Object_Address>`**
    * 특정 객체가 왜 메모리에서 해제되지 않고 남아있는지(참조 체인)를 추적합니다. **메모리 누수 진단의 핵심**입니다.
* **`eeheap -gc`**
    * GC 힙의 구조와 세대별(Gen 0, 1, 2, LOH) 메모리 사용량을 확인합니다.

#### **스레드 및 호출 스택 관련**
* **`clrstack`**
    * 현재 스레드의 관리 코드(Managed Code) 호출 스택을 보여줍니다.
* **`threads`**
    * 프로세스 내의 모든 관리형 스레드 목록을 출력합니다.
* **`setthread <ID>`**
    * 분석할 스레드를 전환합니다.
* **`clrstack -all`**
    * 모든 스레드의 호출 스택을 한꺼번에 출력합니다. **데드락(Deadlock) 분석** 시 유용합니다.

#### **상태 및 변수 조회**
* **`dumpobj <Object_Address>`**
    * 특정 객체의 필드 값과 내부 데이터를 상세히 조회합니다.
* **`dumplocal`**
    * 현재 프레임의 지역 변수들을 확인합니다. (최적화된 빌드에서는 제한적일 수 있습니다.)

### 3. 실무 진단 시나리오 예시

#### **A. 메모리 누수가 의심될 때**
1.  `dumpheap -stat`으로 비정상적으로 개수가 많은 클래스 식별
2.  `dumpheap -mt <MT>`로 해당 객체 주소 하나 확보
3.  `gcroot <Address>`로 어떤 루트(Static 변수, Event Handler 등)가 붙들고 있는지 확인

#### **B. CPU 사용율이 100%일 때**
1.  `threads`로 활동량이 많은 스레드 확인
2.  `setthread`로 해당 스레드 이동 후 `clrstack`으로 현재 무한 루프나 무거운 연산을 수행 중인 메서드 확인

### 4. 팁: `dotnet-gcdump`와의 차이
* **`dotnet-dump`**: 프로세스의 전체 상태를 담기 때문에 파일 크기가 크지만, 로컬 변수나 스택 정보 등 모든 정보를 분석할 수 있습니다.
* **`dotnet-gcdump`**: 메모리 힙 그래프만 가볍게 수집합니다. 파일이 매우 작고 Visual Studio에서 열기 편하지만, 스택 정보는 알 수 없습니다.

#### 1. 특정 타입 이름으로 검색
정확한 전체 네임스페이스를 모르더라도 **부분 일치(Partial Match)** 검색이 가능합니다.

```bash
# 'UserService'가 포함된 모든 타입의 통계 확인
> dumpheap -stat -type UserService

# 특정 네임스페이스 하위의 객체들만 확인
> dumpheap -stat -type MyApp.Models.
```

#### 2. 분석 흐름 (이름 검색 -> 인스턴스 조회)
보통 이름을 통해 **MethodTable(MT)** 주소를 알아낸 뒤 상세 분석으로 들어갑니다.

* `dotnet-dump` 내의 검색은 기본적으로 대소문자를 구분하므로 주의해야 합니다.

1.  **통계 확인:** `dumpheap -stat -type MyClass`
    * 출력 결과의 가장 왼쪽 열에 있는 **MT (MethodTable)** 주소를 복사합니다.
2.  **해당 타입의 모든 인스턴스 주소 나열:** `dumpheap -mt <MT_주소>`
    * 실제 메모리에 할당된 객체들의 개별 주소가 출력됩니다.
3.  **특정 객체 내부 값 보기:** `dumpobj <객체_주소>`

## [High Availability vs. Fault Tolerance: Key Differences - The Couchbase Blog](https://www.couchbase.com/blog/ko/high-availability-vs-fault-tolerance/)

### 1. 고가용성 vs 내결함성 핵심 비교

두 전략 모두 시스템의 가동 시간을 극대화하는 것이 목적이지만, 장애 발생 시의 **'연속성'** 수준에서 차이가 납니다.

| 구분            | 고가용성 (High Availability)                   | 내결함성 (Fault Tolerance)                       |
| :---            | :---                                           | :---                                             |
| **목표**        | 장애 시 **신속한 복구**를 통한 다운타임 최소화 | 장애 시에도 **중단 없는** 운영 보장              |
| **작동 방식**   | 장애 감지 후 예비 시스템으로 전환 (Failover)   | 이중화된 시스템이 동시에 작동하여 즉각 대응      |
| **핵심 기술**   | 로드 밸런서, 자동 장애 조치, 모니터링          | 합의 알고리즘(Raft, Paxos), 자가 복구, 동기 복제 |

### 고가용성 (HA)
* **특징:** "99.999%(파이브 나인스)" 가동률을 지향하며, 단일 장애 지점(SPOF)을 제거하는 데 집중합니다.
* **주요 사례:** 전자상거래(매출 유지), 헬스케어(데이터 접근), 클라우드 서비스(SLA 준수).
* **한계:** 복제 과정에서 데이터 일관성(Consistency)과 가용성(Availability) 사이의 트레이드오프가 발생할 수 있습니다.
* **HA 특화 도구:** 로드 밸런서(Nginx, HAProxy), Route 53(DNS 장애 조치), 모니터링(Prometheus, Datadog).

### 내결함성 (FT)
* **특징:** 일부 구성 요소가 완전히 고장 나더라도 성능 저하 없이 전체 기능을 유지합니다.
* **주요 사례:** 금융 거래(결제 무결성), 항공우주(안전 직결), 국방(중단 없는 명령 체계).
* **한계:** 동일한 작업을 수행하는 중복 자원이 항상 가동되어야 하므로 리소스 낭비와 높은 구축 비용이 발생합니다.
* **FT 특화 도구:** 분산 합의 알고리즘(Zookeeper, etcd), 이벤트 스트리밍(Kafka - 메시지 버퍼링을 통한 데이터 손실 방지).

---

* **공통 도구:** NoSQL 데이터베이스(Couchbase, DynamoDB, MongoDB), 쿠버네티스(Kubernetes).

### "스펙트럼"의 차이

이 둘은 '연속성'이라는 스펙트럼 상에 놓여 있습니다.

- Low Availability: 장애 나면 **수동**으로 복구 (엔지니어 호출)
- High Availability (HA): 장애 나면 **자동**으로 예비 서버 가동 (수 초~수 분 내 복구)
- Fault Tolerance (FT): 장애가 나도 서비스는 미동도 없이 계속됨 (0초 복구)

### Q. 서비스 다운 시, 예비 서비스가 아니라 다운된 서비스를 재시작 후 밀린 메세지 큐를 처리하면서 상태 복구한다면 HA 인가 (이중화 없음)

질문하신 과정은 **"이중화(Redundancy)가 결여된 자가 복구형 고가용성(Self-healing HA)"** 전략입니다. 

그것도 **고가용성(HA)**의 범주에 포함됩니다. 다만, 아키텍처 패턴 중에서는 **'자가 복구(Self-Healing)'** 또는 **'콜드 리부트(Cold Reboot)'** 기반의 HA 전략으로 분류할 수 있습니다.

#### 1. "이중화가 없는데 왜 HA인가요?"

고가용성(HA)의 정의는 '단일 장애 지점을 제거'하는 것뿐만 아니라, **'장애 발생 시 서비스 가동 중지 시간(Downtime)을 최소화하여 가용성을 100%에 가깝게 유지하는 능력'** 전체를 의미하기 때문입니다.

프로세스가 자동으로 다시 살아나서 큐에 쌓인 데이터를 처리하기 시작한다면, 전체 시스템 관점에서는 **"잠시 지연은 있었지만 서비스가 중단되지 않고 유지됨"**으로 간주됩니다.

#### 2. 이 구조의 핵심: 탄력성 (Resilience)

사용자가 말씀하신 구조는 **메시지 큐(Message Queue)**가 중간에서 '상태(State)'와 '데이터'를 보호해주는 완충 역할을 하고 있습니다.

* **데이터 보존:** 서비스 프로세스는 죽었지만, 처리해야 할 '일(Message)'은 큐에 안전하게 저장되어 있습니다.
* **상태 복구:** 재시작 후 큐의 메시지를 다시 읽으며 장애 이전의 상태를 재구성합니다.

이것은 이중화(Redundancy)를 통한 즉각적인 전환은 아니지만, **시스템의 탄력성(Resilience)**을 극대화하여 최종적으로 가용성을 확보하는 방식입니다.

실무에서는 **쿠버네티스(Kubernetes)**의 'Liveness Probe'가 컨테이너를 자동으로 재시작해주는 메커니즘이 바로 이 원리를 기반으로 하며, 현대적인 클라우드 네이티브 환경에서 매우 보편적으로 사용되는 HA 기법 중 하나입니다.

## [less (Unix)](https://en.wikipedia.org/wiki/Less_(Unix))

### `less` 명령어의 유래와 등장 배경

`less`는 유닉스 및 유닉스 계열 시스템에서 텍스트 파일의 내용을 확인하기 위해 사용하는 **터미널 페이저(terminal pager)**입니다. 이 프로그램은 이전의 표준 페이저였던 `more`의 한계를 극복하기 위해 만들어졌습니다.

### 1. 시대적 배경: `more`의 한계
`less`가 등장하기 전, 유닉스 사용자들은 **`more`**라는 명령어를 사용했습니다. 당시 `more`는 이름 그대로 파일의 내용을 **"더 보여주는(more)"** 기능에 충실했지만, 몇 가지 결정적인 단점이 있었습니다.

* **단방향 이동:** `more`는 파일의 앞부분에서 뒷부분으로만 이동할 수 있었습니다. 이미 지나간 텍스트를 다시 보기 위해 위로 스크롤하는 기능이 없었거나 매우 제한적이었습니다.
* **전체 로딩 방식:** 대용량 파일을 읽을 때 파일 전체를 메모리에 올리거나 끝까지 읽어야 하는 경우가 있어 실행 속도가 느려질 때가 많았습니다.

### 2. `less`의 탄생과 위트 있는 이름
1983년에서 1985년 사이, **마크 누델만(Mark Nudelman)**은 `more`의 불편함을 해결하고자 새로운 페이저를 개발했습니다.

* **언어유희:** `less`라는 이름은 **"less is more"** (적은 것이 더 많은 것이다)라는 유명한 격언에서 따온 것입니다.
* **유래:** 당시 동료들이 "반대 방향(역방향)으로도 이동할 수 있는 `more`가 필요하다"고 농담처럼 이야기했고, 이에 마크 누델만은 `more`의 반대말인 `less`를 프로그램 이름으로 정했습니다. 유닉스 커뮤니티에서는 이를 두고 **"less is more, more or less"**라는 농담을 즐기기도 합니다.

## Q: git 1인 개인 private 저장소라도 api key 등 민감 정보를 저장하는 것이 안전한지?

A: 보안은 **"현재의 비공개 상태"**를 믿는 것이 아니라, **"실수로 노출되어도 안전한 구조"**를 만드는 것에서 시작됩니다. 지금 즉시 민감 정보를 코드에서 분리하고 히스토리를 정리하는 것을 권장합니다.

결론부터 말씀드리면, **개인 1인용 Private 저장소라 할지라도 비밀번호나 API Key를 직접 커밋하는 것은 매우 위험한 습관입니다.** 보안 업계에서는 이를 "잠재적 사고(Incidental Exposure)"의 전단계로 간주합니다.

### 1. 계정 탈취 및 유출 위험
* **계정 해킹:** GitHub/GitLab 등 플랫폼 계정이 탈취될 경우, 공격자는 즉시 모든 Private 저장소의 소스 코드를 복제합니다. 이때 포함된 API Key는 그대로 공격자의 자산이 됩니다.
* **로컬 PC 분실/해킹:** 개발 장비를 분실하거나 로컬 환경이 악성코드에 감염되면, `.git` 히스토리에 남은 모든 민감 정보가 유출됩니다.

### 2. Git 히스토리의 영속성
* Git은 파일의 현재 상태뿐만 아니라 **과거의 모든 변경 이력**을 저장합니다. 나중에 코드를 수정해서 키를 삭제하더라도, 이전 커밋 기록(History)에는 해당 값이 그대로 남아 있습니다. 이를 완전히 지우려면 `git filter-repo`나 `BFG Repo-Cleaner` 같은 도구로 히스토리를 강제로 재작성해야 하는 번거로움이 발생합니다.

### 3. 실수에 의한 Public 전환
* 작업 중에 저장소 설정을 실수로 **Public으로 변경**하거나, 팀 프로젝트로 전환하기 위해 협업자를 초대하는 순간 민감 정보가 공개됩니다. 특히 Public으로 전환되는 찰나에 스캐닝 봇(Bot)들이 1초도 안 되어 키를 수집해 가기 때문에 즉각적인 금전적 피해(클라우드 비용 폭탄 등)로 이어질 수 있습니다.

## [Tony Hoare - Wikipedia](https://en.wikipedia.org/wiki/Tony_Hoare)

- [null이 부른 10억 달러 손실 (토니 호어) - YouTube](https://www.youtube.com/watch?v=siE6uYkJOxU)

Speaking at a software conference in 2009, Hoare apologised for inventing the null reference:[26][27]

> I call it my billion-dollar mistake. It was the invention of the null reference in 1965. At that time, I was designing the first comprehensive type system for references in an object-oriented language (ALGOL W). My goal was to ensure that all use of references should be absolutely safe, with checking performed automatically by the compiler. But I couldn't resist the temptation to put in a null reference, simply because it was so easy to implement. This has led to innumerable errors, vulnerabilities, and system crashes, which have probably caused a billion dollars of pain and damage in the last forty years.

`null`을 **"10억 달러짜리 실수(The Billion Dollar Mistake)"**라고 부르는 이유는 이 개념을 처음 고안한 컴퓨터 과학자 **토니 호어(Tony Hoare)**가 2009년 한 컨퍼런스에서 직접 사용한 표현이기 때문입니다.

### 1. 부적절한 단순함

1965년 당시 토니 호어는 ALGOL W라는 언어를 설계하면서 "존재하지 않는 값"을 표현하기 위해 `null`을 도입했습니다. 당시 그의 목표는 단순히 **구현하기 쉽다는 이유**로 모든 참조 타입이 아무것도 가리키지 않을 수 있도록 허용하는 것이었습니다.

### 2. 예측 불가능한 런타임 오류

`null`의 가장 큰 문제는 컴파일 시점에서는 문제가 없어 보이지만, 실행 시점(Runtime)에 객체에 접근하려 할 때 **NullReferenceException**(또는 NullPointerException)을 발생시킨다는 점입니다. 
* 프로그램이 예기치 않게 종료되는 가장 흔한 원인 중 하나입니다.
* 개발자는 모든 참조 변수를 사용할 때마다 `if (obj != null)`과 같은 방어적인 코드를 작성해야 하며, 이는 코드의 가독성을 떨어뜨리고 복잡도를 높입니다.

### 3. 막대한 경제적 손실

토니 호어는 지난 수십 년 동안 `null`로 인해 발생한 수많은 버그, 시스템 장애, 보안 취약점, 그리고 이를 수정하기 위해 투입된 전 세계 개발자들의 시간과 비용을 합산하면 **10억 달러(약 1조 원 이상)**가 넘을 것이라고 회고했습니다.

### 4. 대안의 등장
현대 프로그래밍 언어들은 이 실수를 되풀이하지 않기 위해 다양한 안전 장치를 도입하고 있습니다.
* **Optional / Maybe 타입:** 값이 있을 수도 있고 없을 수도 있음을 명시적으로 타입 시스템에 포함 (Rust, Swift, Java 등)
* **Nullable Reference Types:** 컴파일러 수준에서 `null` 가능성을 체크 (C# 8.0+, Kotlin 등)

결론적으로, `null`은 **"값이 없음"을 논리적으로 엄격하게 다루지 않고 편의상 도입한 설계"**가 소프트웨어 역사상 가장 비싼 비용을 치르게 한 사례로 남게 된 것입니다.

Git의 **자동 유지보수 기능**과 **수동 가비지 컬렉션**이 동시에 실행될 때 발생할 수 있는 객체 손상 문제입니다.

- **cron 스케줄링**: 새벽 시간대에 `git gc` 작업을 자동 실행하도록 설정
- **git maintenance**: 백그라운드 유지보수 작업이 동시 진행 (loose object 정리, incremental-repack 등)
  - 두 프로세스가 동일한 `.git/objects` 디렉토리에 접근하면서 **객체 파일 손상** 발생

### 기술적 원인

````bash
# 문제가 되는 동시 실행 시나리오
# Process 1: cron에서 실행
git gc

# Process 2: git maintenance (백그라운드)
git maintenance run --task incremental-repack --task loose-objects --task pack-refs
````

- **파일 잠금 경합**: 두 프로세스가 동시에 pack 파일을 생성/수정하려 시도
- **임시 파일 충돌**: `.tmp-` 접두사를 가진 임시 객체 파일들의 이름 충돌
- **인덱스 불일치**: pack 인덱스(`.idx`)와 실제 pack 파일(`.pack`) 간의 동기화 실패

### 복구 과정

- **백업 복원**: 일일 백업에서 `.git` 디렉토리 전체 복구
- **객체 검증**: `git fsck --full`로 저장소 무결성 확인

Git의 동시성 제어는 완벽하지 않으므로, 자동화된 유지보수 작업들 간의 **스케줄 조정**이나 **상호 배타적 실행**을 보장하는 것이 중요합니다.

---

Redis의 새로운 `Array` 타입이 이 구조를 채택한 이유와 함께, 핵심 개념을 정리해 드립니다.

**Sparse Array(희소 배열)**는 배열의 요소 중 대부분이 기본값(예: 0, NULL, 혹은 빈 데이터)이고, **의미 있는 유효 데이터가 극히 일부분인 배열**을 효율적으로 관리하기 위한 자료구조입니다.

### 1. 왜 필요한가? (문제의식)
일반적인 배열(Dense Array)은 선언한 크기만큼의 메모리 공간을 **연속적으로 미리 할당**합니다.
*   **예:** 크기가 1,000,000인 배열을 만들고 0번과 999,999번에만 값을 넣어도, 실제로는 100만 개의 공간이 메모리를 차지합니다.
*   **낭비:** 99.9% 이상의 공간이 텅 비어있음에도 불구하고 막대한 메모리가 낭비됩니다.

**Sparse Array**는 이 "빈 공간"을 실제 메모리에 할당하지 않고, **데이터가 있는 부분만 저장**하여 메모리 효율을 극대화합니다.

### 2. Sparse Array의 주요 구현 방식

#### ① 압축 행렬 방식 (Coordinate List / Triplets)
데이터가 있는 요소의 **(인덱스, 값)** 쌍만 별도로 저장하는 방식입니다.
*   **구조:** `[(0, "A"), (1000, "B"), (999999, "C")]`
*   **장점:** 메모리 절약이 매우 뛰어남.
*   **단점:** 특정 인덱스를 찾을 때 검색(Binary Search 등)이 필요하여 접근 속도가 $O(1)$보다 느려질 수 있음.

#### ② 다층 인디렉션 (Multi-level Indirection) — *Redis 방식*
Redis Array가 채택한 방식으로, 거대한 인덱스 범위를 여러 계층의 디렉토리로 쪼개어 관리합니다.
*   **구조:** 
    *   **Root Directory:** 상위 인덱스 범위를 담당.
    *   **Leaf Nodes (Slices):** 실제 데이터가 들어있는 작은 블록(예: 4,096개 단위).
*   **동작:** 특정 인덱스에 값이 들어올 때만 해당 구간의 "Leaf Node"를 생성합니다. 데이터가 없는 구간은 포인터가 NULL로 비어 있어 메모리를 쓰지 않습니다.
*   **장점:** 인덱스를 통한 접근 속도가 **$O(1)$**에 가깝게 유지되면서도 Sparse한 특성을 살릴 수 있습니다.

### 3. Redis Array에서의 Sparse 특성

antirez가 구현한 Redis Array는 매우 지능적인 Sparse 전략을 사용합니다.

1.  **메모리 절약:** 사용자가 `ARSET mykey 1000000000 "data"`라고 명령해도, Redis는 10억 개의 칸을 만들지 않습니다. 인덱스 경로에 필요한 소량의 디렉토리 노드와 데이터가 담긴 1개의 슬라이스만 생성합니다.
2.  **순회 최적화 (`ARSCAN`):** 일반 배열에서 0번과 10억 번에만 데이터가 있을 때 순회하려면 10억 번을 반복해야 합니다. 하지만 Redis의 Sparse Array는 **데이터가 없는 빈 구간을 점프(Skip)**하여 실제 데이터가 있는 지점만 즉시 찾아갑니다.
3.  **유연한 변환:** 데이터가 꽉 차면 효율적인 연속 배열(Dense)처럼 작동하고, 데이터가 듬성듬성 있으면 Sparse 모드로 공간을 아낍니다.

### 4. Sparse Array의 장단점 요약

#### **장점**
*   **공간 효율성:** 방대한 인덱스 범위에 비해 실제 데이터가 적을 때 메모리 사용량을 획기적으로 줄임.
*   **유연성:** 데이터의 최대 범위를 미리 확정 짓지 않고도 큰 인덱스를 자유롭게 사용 가능.

#### **단점**
*   **설계 복잡도:** 인덱스 계산 및 메모리 관리가 일반 배열보다 복잡함.
*   **오버헤드:** 데이터가 아주 빽빽하게 들어찬 경우에는 오히려 디렉토리 관리 비용(포인터 등) 때문에 일반 배열보다 메모리를 약간 더 쓸 수 있음.

## [Why TUIs are back by Alcides Fonseca](https://wiki.alcidesfonseca.com/blog/why-tuis-are-back/)

제공해주신 텍스트는 **터미널 사용자 인터페이스(TUI)**가 다시 주목받는 이유를 현대 운영체제(Windows, macOS, Linux)의 UI 전략 실패와 일렉트론(Electron) 앱의 한계라는 관점에서 분석하고 있습니다. 주요 내용을 정리 및 번역해 드립니다.

### 1. 개요: TUI의 귀환
*   **현상**: 최근 TUI가 다시 유행하고 있습니다. DHH의 Omarchy 프로젝트처럼 즉각적인 피드백과 개발자 특유의 감성(geek points)을 위해 TUI를 채택하는 사례가 늘고 있습니다.
*   **과거의 패턴**: 10년 전 코드 에디터 시장이 네이티브 앱(Textmate 등)에서 일렉트론 기반(VSCode 등)으로 넘어갔을 때, 하드코어 유저들은 사용성 대신 강력한 기능을 택하며 Vim이나 Emacs 같은 TUI 환경으로 회귀했습니다.

### 2. 주요 플랫폼별 UI 현황 및 문제점

#### Windows: 일관성 없는 전략
*   **네이티브의 패배**: 마이크로소프트는 MFC, Winforms, WPF, Silverlight, WinUI, MAUI 등 수많은 GUI 프레임워크를 내놓았지만, 하나도 제대로 정착시키지 못했습니다.
*   **복잡성 증대**: 새로운 레이어가 추가될 때마다 이전 프레임워크에서 가능했던 기능이 제한되는 등 파편화가 심각합니다. 현재 윈도우 OS 내에서 시각적 일관성이 유지되었던 마지막 기억은 Windows 98/2000 시절입니다.

#### Linux: 설계된 불일치
*   **프레임워크의 양분**: GTK와 Qt가 시장을 지배하고 있으며, 서로 다른 툴킷으로 만든 앱들이 그나마 나란히 있을 때 봐줄 만한 수준입니다.
*   **네이티브 앱의 부재**: 수많은 배포판과 하드웨어 조합을 테스트하기 어렵기 때문에, 대부분의 기업은 네이티브 앱 대신 일렉트론을 선택하거나 오픈 소스 커뮤니티에 개발을 맡깁니다.

#### macOS: 가이드라인의 붕괴
*   **전통의 퇴색**: UI 디자인의 교본이라 불리던 애플의 '휴먼 인터페이스 가이드라인(HIG)'이 무너지고 있습니다.
*   **사용성 저하**: 피츠의 법칙(Fitts' law) 무시, 창 크기 조절의 어려움, 모든 메뉴에 아이콘 추가 등 과거의 일관성과 편의성을 스스로 깨뜨리고 있습니다.

### 3. 일렉트론(Electron)의 한계
*   **문제점**: 메모리 사용량은 개선되고 있으나, **시각적 일관성 결여**와 **키보드 중심 워크플로우 부재**가 가장 큰 문제입니다.
*   **사례**: Cursor나 VSCode 같은 앱에서 사이드 패널 이동 등 기본적인 동작이 키보드만으로는 어렵거나 메뉴에 명시되지 않는 경우가 많습니다. 이는 앱이 샌드박스 내부의 HTML로 구동되기 때문입니다.

### 4. TUI가 대안이 되는 이유

*   **효율성**: 빠르고 자동화가 쉬우며, 운영체제에 상관없이 일관되게 작동합니다.
*   **원격 접속**: 복잡한 설정 없이 원격 서버나 클라우드 환경에서 바로 실행할 수 있습니다.
*   **본질 집중**: OS의 간섭 없이 상호작용 그 자체에 집중할 수 있게 해줍니다. 애플과 마이크로소프트가 남긴 'UI 포스트 아포칼립스' 세상에서 TUI는 훌륭한 안식처가 되고 있습니다.

### 5. 향후 과제: 기본으로의 회귀
*   **일관된 인터페이스**: 사용자가 'Command + C'가 복사라는 것을 배우면, 모든 곳에서 똑같이 작동해야 합니다. 인터페이스는 사용자가 생각할 필요가 없을 때 가장 좋은 인터페이스입니다.
*   **교육의 변화**: 소프트웨어 공학 커리큘럼에서 UI 디자인을 가벼운 기술로 취급하지 말고, 인터페이스가 엉망인 프로젝트는 탈락시켜야 할 만큼 중요하게 다뤄야 합니다.
*   **플랫폼 제작자의 역할**: 개발자들이 사용하고 싶어 하는 접근성 높은 툴킷을 만들어 진입 장벽을 낮추고, 일렉트론이나 TUI에 대한 의존도를 줄일 수 있는 네이티브 솔루션을 제공해야 합니다.

## [Multi-key operations  Docs](https://redis.io/docs/latest/develop/using-commands/multi-key-operations/)

- Redis Clustered Environments / Multi-key operations
- [Redis Multi-key command in cluster mode (feat. CROSS-SLOT)  by Moon SeungHun  Medium](https://medium.com/@mbh023/redis-multi-key-command-in-cluster-mode-feat-cross-slot-ec27b999f169)

```
# This may fail if keys are in different slots
MSET user:1 "Alice" user:2 "Bob"

# Use hash tags to ensure same slot
MSET {users}:1 "Alice" {users}:2 "Bob"

# Check which slot a key belongs to
CLUSTER KEYSLOT user:1
CLUSTER KEYSLOT {users}:1

...

# Enter redis-cli with cluster mode
$ redis-cli -c
# Try to delete multiple keys
127.0.0.1:6379>  DEL key1 key2 key3 ...
-- (error) CROSSSLOT Keys in request don't hash to the same slot
```

- CROSSSLOT: Keys in request don't hash to the same slot
- MOVED: Key has moved to a different node (during resharding)
- TRYAGAIN: Operation temporarily unavailable (during migration)

레디스 클러스터 환경에서는 각 키를 저장할 때 CRC16 해시 알고리즘으로 저장 슬롯을 고른다.

`HASH_SLOT = CRC16(key) mod 16384`

그러니 다수의 키에 접근하는 명령어를 사용한다면, 대상 키가 서로 다른 슬롯에 속해있는 경우 흔히 Cross Slot 문제를 겪는다. 

많은 글들이 대응 방안으로 hash tag (중괄호로 감싼 안쪽 문자열만 해시 연산) 사용을 제안

```
redis-cli CLUSTER KEYSLOT "{user:1001}:profile"
redis-cli CLUSTER KEYSLOT "{user:1001}:sessions"
# Both should return the same slot number
```

## [What Is Code](https://martinfowler.com/articles/what-is-code.html)

> Working with LLMs
>
> Considering 'Coding as Vocabulary Building', has important implications for LLMs. LLMs are trained on vocabulary from a large body of text and code. They learn recurring relationships between names, APIs, libraries, frameworks, idioms, design patterns, and implementation structures. When they see words such as Controller, Repository, Reducer, ConsensusModule or TransactionLog those names are not just labels. They carry associations with known code structures and expected behavior.
>
> This is why vocabulary matters when working with LLMs. If our prompts use vague or inconsistent language, the model has to guess the design we intend. If our codebase uses unclear names and inconsistent concepts, the model has little stable structure to follow. But when the vocabulary is precise, consistent, and embodied in the code, the LLM can map our intent more reliably to useful implementation.
>
> Cognitive Debt
>
> This also explains a particular danger of LLM-assisted coding: cognitive debt. Cognitive debt accumulates when words, abstractions, and structures are used without their meaning being well understood by the people working with them. LLMs amplify this risk because they can generate large amounts of plausible code very quickly. The generated code may contain controllers, repositories, reducers, factories, transactions, schedulers, or other familiar-looking structures. The code may compile. It may even pass basic tests. But if the team does not understand the conceptual model behind those structures, the codebase has gained vocabulary without shared understanding. The problem is not that the LLM generated code. The problem is that the code introduced vocabulary faster than the developers built understanding. That gap is one of the major contributors to cognitive debt.

---

**'어휘를 쌓는 일로서의 코딩(Coding as Vocabulary Building)'** 관점은 LLM 활용에도 중요한 시사점을 줍니다.

LLM은 방대한 텍스트와 코드의 **어휘**로 학습됩니다. 학습 과정에서 다음과 같은 요소들 사이의 반복적인 관계를 익히게 됩니다.

- 이름 (names)
- API
- 라이브러리 / 프레임워크
- 관용 표현 (idioms)
- 디자인 패턴
- 구현 구조 (implementation structures)

`Controller`, `Repository`, `Reducer`, `ConsensusModule`, `TransactionLog` 같은 이름은 **단순한 레이블이 아닙니다**. 각각의 이름은 이미 알려진 코드 구조 및 기대되는 동작과 자연스럽게 연관되어 있습니다.

### 왜 어휘가 중요한가요

| 상황                                            | 결과                                                                |
| ----------------------------------------------- | ------------------------------------------------------------------- |
| 프롬프트가 모호하거나 일관성이 없을 때          | 모델이 의도한 설계를 **추측**해야 합니다                            |
| 코드베이스의 이름이 불명확하고 개념이 흔들릴 때 | 모델이 따라갈 만한 **안정적인 구조**가 없습니다                     |
| 어휘가 정확하고 일관되며 코드에 잘 녹아 있을 때 | LLM이 의도를 **유용한 구현으로 안정적으로 매핑**할 수 있습니다      |

> 핵심: 코드와 프롬프트의 어휘가 정밀하고 일관될수록, LLM은 우리의 의도를 더 정확하게 구현으로 옮길 수 있습니다.

---

### Cognitive Debt (인지적 부채)

이는 LLM 보조 코딩의 특정한 위험, 곧 **인지적 부채(cognitive debt)** 도 함께 설명해 줍니다.

> 인지적 부채는 작업자가 그 의미를 충분히 이해하지 못한 채로 **단어·추상·구조**가 사용될 때 누적됩니다.

### LLM이 위험을 증폭시키는 이유

LLM은 그럴듯한 코드를 **빠르게, 대량으로** 생성할 수 있습니다. 생성된 코드에는 다음과 같이 익숙해 보이는 구조들이 자주 포함됩니다.

- Controllers
- Repositories
- Reducers
- Factories
- Transactions
- Schedulers

이러한 코드는 파일이 될 수 있고, 기본적인 테스트도 통과할 수 있습니다.

그러나 팀이 그 **구조 뒤의 개념적 모델**을 이해하지 못한다면, 코드베이스는 **공유된 이해 없이 어휘만 늘어난 상태**가 되고 맙니다.

### 핵심 통찰

> 문제는 LLM이 코드를 생성했다는 것이 아닙니다.
> **개발자가 이해를 쌓는 속도보다 코드가 어휘를 더 빨리 도입했다는 것**이 진짜 문제입니다.
> 이 간극이 곧 인지적 부채의 주요 원인 중 하나입니다.

## [No Silver Bullet](https://www.cgl.ucsf.edu/Outreach/pc204/NoSilverBullet.html)

Fred Brooks의 1986년 에세이는 40년이 지난 지금도 소프트웨어 공학에서 가장 자주 인용되는 글 중 하나입니다. 평가는 크게 **"여전히 옳다"**, **"부분적으로 틀렸다"**, **"LLM이 진짜 은총알인가"** 세 갈래로 나뉩니다.

### 1. 여전히 지지받는 부분 (대체로 컨센서스)

#### "본질적 vs 부수적 복잡성" 구분은 살아남았다

- 이 프레임은 거의 모든 소프트웨어 공학 교육에서 표준 어휘가 됐습니다.
- **Out of the Tar Pit** (Moseley & Marks, 2006) 같은 후속 논문은 Brooks의 구분을 출발점으로 삼아 "상태(state)와 제어 흐름이 본질적 복잡성의 대부분을 차지한다"고 확장했습니다.
- John Ousterhout의 *A Philosophy of Software Design* (2018)도 복잡성 관리를 SE의 중심 주제로 다루며 Brooks의 영향을 이어받습니다.

#### "도구가 본질을 줄여주진 않는다"는 진단

- IDE, 정적 분석, 클라우드, 컨테이너 등은 부수적 복잡성을 크게 줄였지만, **"무엇을 만들 것인가"** 결정의 어려움은 그대로라는 데 많은 개발자가 동의합니다.
- 요구사항 정의, 도메인 모델링, 의사소통의 어려움은 줄지 않았다는 관찰이 일관됩니다.

### 2. 비판·반박 관점

#### Robert L. Glass의 반론

- *Facts and Fallacies of Software Engineering* (2002)에서 Brooks의 주장을 일부 반박. "10년에 10배는 아니어도, 누적된 도구 발전은 실질적 생산성 향상을 가져왔다"고 지적.

#### "Brooks는 부수적 복잡성을 과소평가했다"

> "1986년 시점에 부수적 복잡성은 이미 상당 부분 정리되었다. 따라서 아무리 좋은 도구가 나와도, 남아 있는 본질적 복잡성이 발목을 잡기 때문에 10년 안에 10배 생산성 향상은 불가능하다."

- 1986년 당시에는 메모리 관리, 빌드 시스템, 배포가 어마어마한 부수적 부담이었음.
- GC, 패키지 매니저, CI/CD, 클라우드, 타입 추론, LSP 등이 누적되며 **사실상 자릿수 단위 생산성 향상**이 일어났다는 주장.
- 즉, Brooks가 "10년 내 10배"라고 못 박은 부분은 시점만 어긋났을 뿐 결국 일어났다고 보는 시각.

#### 도메인 특화 환경의 반증

- Excel, SQL, React, Rails 같은 **특정 도메인에 특화된 추상화**는 그 영역에서 실제로 10배 이상의 생산성 차이를 만들어냈다는 사례 제시.

### 3. LLM 시대의 재점화

가장 활발한 논쟁입니다. 양 진영이 모두 Brooks를 인용합니다.

#### "LLM이 드디어 은총알이다" 진영

- Copilot, Cursor, Claude Code 등이 **부수적 복잡성을 거의 0에 수렴시키고 있다**는 주장.
- 보일러플레이트, 언어 전환, 테스트 작성 등에서 명백한 자릿수 단위 가속.
- Andrej Karpathy의 "Software 2.0/3.0" 담론, Simon Willison의 LLM 실용주의 글들이 대표적.

#### "Brooks가 여전히 옳다" 진영

- Brooks의 핵심: **"어려운 부분은 *무엇*을 말할지 결정하는 것이지, *어떻게* 말할지가 아니다."**
- LLM은 "어떻게 말할지"는 도와주지만 "무엇을 말할지"는 여전히 사람이 정해야 함.
- Unmesh/Martin 대담의 입장이 정확히 이 라인 — **추상화의 발견**은 여전히 사람의 영역.
- Simon Willison조차 "LLM은 패턴 매칭은 잘하지만 새로운 추상화를 발명하진 못한다"는 입장을 자주 표명.

#### 절충적 시각 (가장 흔함)

- LLM은 **AI-1**의 정의를 또 한 번 미끄러뜨리는 사례 (Parnas의 "sliding meaning"의 재현).
- 부수적 복잡성에서는 거의 은총알에 가깝지만, 본질적 복잡성에서는 여전히 보조 도구.
- **"Brooks는 옳았지만, 그가 상상한 것보다 부수적 복잡성의 비중이 컸다"**는 절충안이 점점 우세.

## [Git vs. Perforce P4: How to Choose (and When to Use Both) | Perforce Software](https://www.perforce.com/blog/vcs/git-vs-perforce-how-choose-and-when-use-both#git-vs-perforce-p4-for-game-development)

> git 이 저장소 전체에 대한 버전을 다룬다면 퍼포스는 파일 개별마다 리비전을 가지는 것으로 이해 가능

제시해주신 텍스트는 게임 개발 및 대규모 프로젝트 환경에서 Perforce(P4)가 **Git**보다 선호되는 이유와 타 산업으로의 확장성을 다루고 있습니다.

**게임 개발을 위한 Git vs. Perforce P4**
게임 개발에 있어 Git은 충분하지 않습니다. Git은 코드 관리에는 훌륭하지만, 게임 개발과 관련된 수많은 파일, 초고용량 파일, 그리고 다양한 에셋 유형을 관리하기에는 역부족입니다. 이것이 바로 게임 개발 팀들이 모든 것을 관리하기 위해 Perforce를 선택하면서도, 코드에 대해서는 여전히 Git에 대한 접근성을 유지하는 이유입니다.

다른 산업 분야에서도 게임 산업의 이러한 혁신적 가치를 발견했습니다. 많은 자동차 기업들이 이와 유사한 워크플로우를 채택하고 P4를 통합하여 방대하고 복잡한 디자인 파일들을 관리하고 있습니다. 이를 통해 대규모 프로젝트를 능숙하게 관리하는 동시에, 엄격한 안전 준수 사항을 위한 추적성(traceability)을 유지할 수 있게 되었습니다.

### [핵심 내용 정리]

**1. Git의 한계와 Perforce의 필요성**

* **Git:** 소스 코드 관리(SCM)에는 최적화되어 있으나, 게임 개발 특유의 **대량 파일** 및 **초고용량 바이너리 에셋** 관리 시 성능 저하가 발생합니다.
* **Perforce:** 게임 개발에 필요한 대규모 파일과 다양한 리소스 유형을 처리하는 데 최적화되어 있어 팀 단위의 중앙 집중식 관리가 가능합니다.

**2. 하이브리드 워크플로우**

* 많은 팀이 코드 관리는 Git의 장점을 활용하고, 전체 프로젝트 및 대용량 에셋 관리는 Perforce를 사용하는 방식을 병행합니다.

---

Perforce(P4)와 Git이 대용량 바이너리 파일을 처리하는 방식의 근본적인 차이는 "중앙집중식 파일 기반 저장"과 "분산형 스냅샷 기반 저장"이라는 아키텍처의 차이에서 비롯됩니다.

### 1. Perforce의 바이너리 처리 방식: 전체 파일(Full File) 및 중앙 통제

Perforce는 바이너리 파일에 대해 압축된 **전체 파일 개별 저장(Full File Storage)** 방식을 기본으로 사용합니다.

* **독립된 개별 압축 저장:** 텍스트 파일(소스 코드)은 변경된 차이점만 기록하는 RCS 델타 방식을 쓰지만, 바이너리 파일(이미지, 3D 모델, 오디오 등)은 수정되어 제출(`p4 submit`)될 때마다 파일 전체를 gzip 등으로 압축하여 서버의 아카이브 디렉토리에 개별 리비전 파일로 저장합니다.
* **중앙 서버 전송 후 로컬 삭제:** 사용자는 현재 작업에 필요한 최신 버전(또는 특정 리비전) 1개만 로컬 워크스페이스에 다운로드(Sync)받아 사용합니다. 과거 이력은 중앙 서버에만 존재하므로, 로컬 디스크 공간을 거의 차지하지 않습니다.
* **독점 락(Exclusive Lock) 지원:** 바이너리 파일은 텍스트처럼 코드 병합(Merge)이 불가능합니다. Perforce는 `+l` 타입을 통해 한 사람이 파일을 체크아웃하면 다른 사람이 수정하지 못하도록 중앙에서 강력한 독점 락을 걸어 충돌을 원천 차단합니다.

### 2. Git의 바이너리 처리 방식: 스냅샷과 Packfile의 한계

Git은 모든 파일의 상태를 스냅샷(SHA-1 hash 기반의 Object)으로 관리하며, 태생적으로 모든 히스토리를 로컬에 복제해야 합니다.

* **Packfile을 통한 델타 압축 시도:** Git은 공간을 줄이기 위해 주기적으로 여러 오브젝트를 묶어 하나의 `packfile`로 만들고, 이 과정에서 파일 간의 유사성을 분석해 델타(차이점) 압축을 시도합니다.
* **바이너리에서의 기술적 한계:** 바이너리 파일(예: 압축된 패킹 데이터, 이미지 등)은 단 1바이트만 바뀌어도 파일 전체의 바이너리 구조가 완전히 뒤바뀌는 경우가 많습니다. 이 때문에 Git은 두 버전 간의 '유사성'을 찾지 못해 델타 압축에 실패하고, 결국 로컬 저장소에 수백 MB짜리 온전한 바이너리 오브젝트들이 고스란히 쌓이게 됩니다.
* **로컬 저장소 폭발(Repository Bloat):** Git은 `git clone` 시 프로젝트의 모든 역사(모든 리비전의 모든 바이너리 파일)를 사용자의 로컬 컴퓨터로 다운로드해야 합니다. 1GB짜리 바이너리 파일이 10번 수정되었다면, 사용자는 첫 클론 시 10GB에 육박하는 데이터를 전부 받아야 하므로 네트워크와 디스크 스토리지에 극심한 병목이 발생합니다.


| 비교 항목 | Perforce Helix Core | 순수 Git (Standard Git) |
| --- | --- | --- |
| **바이너리 저장 방식** | 서버에 리비전별 개별 압축 파일로 저장 | SHA-1 오브젝트로 저장 후 Packfile 델타 압축 시도 |
| **로컬 스토리지 부담** | **매우 낮음** (현재 작업 중인 버전 1개만 유지) | **매우 높음** (과거 모든 버전의 바이너리가 로컬에 누적) |
| **서버-클라이언트 통신** | 필요한 파일만 핀포인트로 전송 (초고속) | 변경 이력 전체를 통째로 동기화 (대용량 시 속도 저하) |
| **충돌 관리** | 중앙 서버의 **Exclusive Lock**으로 동시 수정 차단 | 분산 환경 특성상 락이 어려워 바이너리 병합 충돌 발생 가능 |
| **부분 체크아웃** | 디렉토리나 파일 단위로 원하는 부분만 동기화 가능 | 기본적으로 전체 저장소(Repository)를 다 받아야 함 |


### 결론: Git LFS라는 대안이 있지만...

Git의 이러한 치명적인 문제를 해결하기 위해 Git LFS(Large File Storage)가 개발되었습니다. Git LFS는 실제 바이너리 파일은 외부 스토리지(포인터 파일만 Git에 남김)에 두고 Perforce처럼 필요할 때만 다운로드하는 방식을 취합니다.

그러나 Git LFS는 기존 Git 메커니즘 위에 '우회로'를 얹은 형태이기 때문에, Perforce처럼 수백 테라바이트(TB) 단위의 자산을 완벽히 통제하는 중앙집중식 권한 관리, 확장성, 그리고 고유의 중앙 독점 락(Locking) 기능 수준에는 미치지 못합니다. 이 때문에 대규모 게임 스튜디오나 하드웨어 설계 팀에서는 여전히 Perforce가 표준으로 자리 잡고 있습니다.

기술적으로는 훌륭한 우회책이지만, 본질적으로 "분산형 시스템(Git) 위에 중앙집중형 스토리지(LFS)를 억지로 얹은 구조"이기 때문에 다음과 같은 치명적인 약점들을 가집니다.

#### 1. '두 개의 다른 저장소' 분리로 인한 동기화 및 무결성 취약점

Git LFS를 쓰면 메타데이터를 관리하는 Git 원격 저장소(GitHub 등)와 실제 대용량 파일이 저장되는 **LFS 아카이브 서버**가 물리적·논리적으로 분리됩니다.

* **동기화 실패(Dangling Pointers):** 개발자가 코드를 `git push`할 때, LFS 서버로의 바이너리 업로드가 네트워크 오류나 인증 문제로 실패해도 Git 커밋과 푸시는 성공하는 경우가 발생합니다. 이 경우 다른 팀원이 풀을 받으면 포인터 텍스트는 존재하지만 실제 파일은 다운로드하지 못하는 **깨진 포인터(Dangling Pointer)** 에러를 마주하게 됩니다.
* **이중 관리 아키텍처의 한계:**
하나의 트랜잭션으로 파일과 메타데이터가 동시에 커밋되는 Perforce와 달리, Git LFS는 두 개의 서로 다른 저장소 시스템이 완벽한 원자성(Atomicity)을 보장하지 못하므로 데이터 무결성이 깨지기 쉽습니다.

#### 2. 브랜치 전환(Checkout) 시 극심한 병목과 오버헤드

Git의 가장 큰 장점은 브랜치 전환(`git checkout` / `switch`)이 가볍고 빠르다는 점입니다. 하지만 Git LFS 환경에서는 브랜치를 바꿀 때마다 심각한 성능 저하가 발생합니다.

* **스머지/클린(Smudge/Clean) 필터의 오버헤드:** Git은 브랜치를 전환할 때 포인터 파일을 읽어 LFS 서버(또는 로컬 cache 디렉토리)에서 실제 바이너리 파일을 찾아 워크스페이스에 복원하는 **Smudge 필터** 프로세스를 태웁니다.
* **파일 교체 비용:** 수십 기가바이트의 에셋이 포함된 프로젝트에서 브랜치를 바꿀 때마다, 수백 개의 대용량 바이너리 파일들이 디스크 상에서 실제로 지워지고 다시 복사(또는 하드링크 연산)되는 과정을 거쳐야 하므로 브랜치 전환 속도가 극도로 느려집니다.

#### 3. 분산 환경과 락(Locking) 시스템의 태생적 충돌

바이너리 파일은 코드가 아니기 때문에 여러 명이 동시에 수정하면 병합(Merge)이 불가능합니다. 따라서 누군가 수정 중일 때 다른 사람의 수정을 막는 '독점 락(Exclusive Lock)'이 필수적입니다.

* **중앙 통제 불가능:** Git은 기본적으로 로컬에서 자유롭게 브랜치를 따고 수정하는 분산 구조입니다. Git LFS가 이를 보완하기 위해 `git lfs lock` 기능을 도입했으나, 이는 원격 서버에 접속하여 락 상태를 확인해야 하는 **중앙집중식 검증**이 필요합니다.
* **강제력 부족:** 개발자가 오프라인 상태이거나, 실수로 `git lfs lock`을 확인하지 않고 로컬에서 바이너리를 수정해 버리면, 나중에 `push`할 때서야 충돌을 감지하게 됩니다. Perforce처럼 파일 시스템 수준에서 읽기 전용으로 잠가버리는 강력한 통제가 불가능합니다.

#### 5. 휴먼 에러(Human Error)에 대한 극도의 취약성

Git LFS는 어떤 확장자나 경로의 파일을 LFS로 관리할지 `.gitattributes` 파일에 명시해야 합니다. 이 설정 관리가 까다롭습니다.

* **실수로 일반 Git에 대용량 파일이 올라가는 사고:** 새로 추가된 대용량 파일의 확장자(예: 새로 도입한 포맷인 `.webp`나 `.psd` 등)를 `.gitattributes`에 등록하는 것을 깜빡하고 그냥 `git push`를 해버리면, 그 거대한 바이너리 파일이 순식간에 일반 Git 찌꺼기(Blob)로 박혀버립니다.
* **기록 삭제의 고통:** 한 번 일반 Git 저장소 히스토리에 박힌 대용량 바이너리는 뒤늦게 LFS로 지정하더라도 과거 이력에 그대로 남아 저장소 용량을 차지합니다. 이를 해결하려면 `git filter-repo`나 BFG Repo-Cleaner 같은 도구로 전체 커밋 히스토리를 강제로 재작성(Rewrite)해야 하는 대공사를 치러야 합니다.

###  산업별 철학의 차이

| 항목 | Perforce (게임/임베디드) | Git (소프트웨어/웹) |
| --- | --- | --- |
| **주요 대상** | 대용량 바이너리 (병합 불가) | 텍스트 기반 소스 코드 (병합 가능) |
| **철학** | **충돌 예방 (Prevention)** | **충돌 해결 (Resolution)** |
| **락(Lock)** | 핵심 기능 (실시간 중앙 통제) | 보조 기능 (LFS 확장 프로그램 활용) |
| **실패 비용** | 에셋 재작업 시 비용 매우 높음 | 코드 충돌 해결 비용 상대적으로 낮음 |

결론적으로, Perforce의 락 기능은 "협업 과정에서 버려지는 아티스트와 디자이너의 작업 시간을 0으로 만들겠다"는 게임 개발 현장의 절실한 요구사항이 반영된 결과물이라 할 수 있습니다.

### 2. Perforce의 패러다임: 선점형 제어 (Pessimistic Locking)

Perforce는 "충돌이 일어날 것이라 가정하고 미리 방어한다"는 비관적(Pessimistic) 패러다임을 따릅니다.

* **워크플로우의 강제성:** P4 서버에서 특정 파일 타입(바이너리)에 `+l` (exclusive checkout) 플래그를 설정하면, 시스템은 물리적으로 한 번에 한 명만 그 파일을 수정 가능(Check-out)하도록 보장합니다.
* **커뮤니케이션 비용 감소:** 수천 명이 참여하는 대형 게임 스튜디오에서 누가 어떤 파일을 만지는지 일일이 메신저로 물어볼 수 없습니다. P4에서는 파일을 열려고 할 때 이미 다른 사람이 작업 중이면 "누가 작업 중입니다"라는 메시지를 즉시 띄워주므로, 헛수고(수정 후 병합 실패)를 완벽히 차단합니다.

### 3. Git의 패러다임: 낙관적 병합 (Optimistic Concurrency)

반면 Git은 "충돌이 나더라도 나중에 합칠 수 있다"는 낙관적(Optimistic) 패러다임에서 출발했습니다.

* **소스 코드 중심:** Git은 원래 리눅스 커널 같은 **순수 소스 코드** 관리를 위해 설계되었습니다. 텍스트 파일은 여러 명이 동시에 수정하고 나중에 `merge`하는 방식이 훨씬 생산적입니다. 굳이 락을 걸어 다른 사람의 작업을 막을 필요가 없었던 것이죠.
* **분산 시스템의 한계:** Git은 중앙 서버 없이 로컬에서 자유롭게 커밋하는 구조입니다. "내가 지금 로컬에서 이 파일을 만지고 있다"는 사실을 전 세계의 다른 오프라인 팀원들에게 실시간으로 알리고 락을 걸 방법 자체가 기술적으로 어렵습니다.

  ──────
  ### 1.  Interlocked.Add

   Interlocked.Add(ref int location, int value) 는 값을 원자적으로 더하고, **더해진 결과(새로운 값)**를 반환합니다.

  #### ① x86 / x64 (Intel, AMD)

  x86/x64 아키텍처는  LOCK  프리픽스와 교환-추가 명령어인 ** XADD **를 조합하여 한 번에 처리합니다.

    ; rcx = &location (대상 주소)
    ; edx = value (더할 값)

    mov  eax, edx          ; eax에 더할 값을 복사
    lock xadd [rcx], eax   ; [rcx]의 원래 값과 eax를 교환한 후, [rcx]에 그 합을 저장 (원자적 수행)
    add  eax, edx          ; 이전 값(eax) + 더한 값(edx)을 계산하여 새로운 값을 만듦
                           ; 최종 리턴값은 eax에 저장되어 반환됨

- lock  프리픽스가 붙음으로써 버스 락(Bus Lock) 또는 캐시 락(Cache Lock)이 활성화되어 원자성이 보장됩니다.

  #### ② ARM64 (Apple Silicon, Snapdragon 등)

  ARM 아키텍처는 전통적으로 LL/SC(Load-Link / Store-Conditional) 방식을 사용하여 성공할 때까지 루프를 도는 방식을 취했으나, ARMv8.1부터는 x86처럼 단일 명령어로 처리하는 LSE(Large System Extensions) 가 추가되었습니다. .NET 런타임은 CPU 버전을 감지하여 다르게 컴파일합니다.

- ARMv8.0 (기본 LL/SC 루프 방식):
    ; x0 = &location (대상 주소)
    ; w1 = value (더할 값)

    .loop:
        ldaxr   w2, [x0]       ; [x0]에서 값을 독점적으로 로드 (Load-Exclusive + Acquire)
        add     w3, w2, w1     ; w3 = 기존 값 + 더할 값
        stlxr   w4, w3, [x0]   ; [x0]에 w3 저장을 시도 (Store-Exclusive + Release)
        cbnz    w4, .loop      ; 만약 저장에 실패(w4 != 0, 다른 코어가 가로챔)하면 루프 재시도
        mov     w0, w3         ; 새로운 값을 결과로 반환

- ARMv8.1 LSE (단일 명령어 방식):
    ; x0 = &location (대상 주소)
    ; w1 = value (더할 값)
    ldaddal w1, w2, [x0]   ; [x0]의 원본 값을 w2에 담고 w1을 [x0]에 원자적으로 더함 (Acquire/Release Semantics)
    add     w0, w2, w1     ; 리턴값(새로운 값) 계산

  ──────
  ### 2.  Interlocked.CompareExchange

   Interlocked.CompareExchange(ref int location, int value, int comparand) 는 메모리 값( location )이 비교값( comparand )과 일치하면 새로운 값( value )으로 바꾸고, 바뀌기 전의 원래 값을 반환합니다. (CAS: Compare-And-Swap 연산)

  #### ① x86 / x64 (Intel, AMD)

  비교 및 교환을 수행하는  CMPXCHG  명령어에  LOCK  프리픽스를 사용합니다. 이 명령어는 관례적으로  EAX  레지스터의 값을 비교값으로 사용합니다.

    ; rcx = &location (대상 주소)
    ; edx = value (새 값)
    ; r8d = comparand (비교할 값)

    mov  eax, r8d            ; eax 레지스터에 비교 대상 값(comparand)을 복사
    lock cmpxchg [rcx], edx  ; [rcx]가 eax와 같으면 [rcx]에 edx를 저장.
                             ; 다르면 eax에 [rcx]의 현재 값을 로드. (원자적 수행)
                             ; 최종 리턴값은 원래 값(eax)이 반환됨

  #### ② ARM64

  CompareExchange 역시 ARM64 버전 아키텍처에 따라 분기됩니다.

- ARMv8.0 (기본 LL/SC 루프 방식):
    ; x0 = &location (대상 주소)
    ; w1 = value (새 값)
    ; w2 = comparand (비교할 값)
    .loop:
        ldaxr   w3, [x0]       ; [x0]에서 값을 독점적으로 로드
        cmp     w3, w2         ; 로드한 값(w3)과 비교값(w2) 비교
        b.ne    .exit          ; 값이 다르면 대입하지 않고 즉시 루프 탈출
        stlxr   w4, w1, [x0]   ; 값이 같으면 [x0]에 새 값(w1) 저장을 시도
        cbnz    w4, .loop      ; 저장에 실패하면 루프 재시도
    .exit:
        mov     w0, w3         ; 이전 값(w3)을 반환

- ARMv8.1 LSE (단일 명령어 방식):
    ; x0 = &location (대상 주소)
    ; w1 = value (새 값)
    ; w2 = comparand (비교할 값)
    mov     w3, w2         ; 비교값을 레지스터 w3에 대입
    casal   w3, w1, [x0]   ; [x0] 값과 w3(comparand)를 비교하여 같으면 w1로 원자적 교체.
                           ; 교체 여부와 상관없이 [x0]의 기존 값을 w3에 돌려받음
    mov     w0, w3         ; 이전 값 반환

## .NET 8 -> .NET 10 개선점 쭉 한 번 정리

### 런타임 자동 최적화

 **사용자 코드 변경 없이** 버전 업만으로 적용되는 JIT/런타임 개선 사항입니다.

#### Object Stack Allocation

"객체에 대한 참조(reference) 가 자기를 만든 메서드의 lifetime 바깥으로 도달 가능한가"를 분석하는 JIT의 escape analysis가 강화됩니다.


외부로 새어 나가지 않는 reference type 인스턴스를 stack에 할당하도록 바뀌었습니다. 
(컴파일러가 판단하며 조건부로 stack 할당합니다)

```cs
public class Foo { 
    public void DoSomething(string str) {...}
}

var foo = new Foo();   // escape하지 않으면 stack에 배치될 수 있음
foo.DoSomething("hi");
```

.NET 9에서는 sealed/단순 lifetime을 가진 객체부터 시작했고, .NET 10에서는 array, delegate 일부, async state machine 등으로 범위가 더 넓어집니다.


#### Tiered Compilation + Profile Guided Optimization(PGO)

**.NET Tiered Compilation**

JIT 는 고수준 최적화 코드를 판단하고 생성하는데 많은 시간이 소요됩니다.

TieredCompilation은 첫 시작 시 저수준 최적화 코드를 준비해 프로세스 스타트업(혹은 부트) 시간을 단축하면서, 

후속으로 핫패스 지점을 추적해 고수준 최적화 코드로 전환시키는 전략입니다.

두 단계로 동작합니다.

  - Tier 0: 최소 최적화로 빠르게 컴파일합니다. 시작 시간을 짧게 가져가면서, 실행 중 호출 빈도·분기 빈도·실제 호출되는 타입 같은 프로파일 데이터를 수집합니다.
  - Tier 1: 자주 호출되는 메서드를 더 비싼 최적화로 재컴파일합니다. 이때 Tier 0에서 수집한 프로파일이 입력으로 들어갑니다. 이 흐름이 Dynamic PGO입니다.

그 외 프로세스 실행 주기 내내 while 루프를 도는 코드 등에 대해서 OSR(On-Stack Replacement) 과 같은 특수 처리가 존재합니다.

##### .NET 9 / 10에서 좋아진 점

- **Guarded Devirtualization 정밀도 향상**: 인터페이스나 가상 메서드 호출 지점에서는 실제로 어떤 구체 타입이 들어오는지 코드만 봐서는 알 수 없습니다. PGO는 실행 중 "이 호출 지점에는 사실상 `Circle` 타입만 들어옵니다" 같은 빈도 정보를 수집해 둡니다. JIT는 그 정보를 받아 `Circle`에 대해서만 직접 호출(+inline) 코드를 만들고, 나머지는 일반 가상 호출로 떨어지는 fallback을 두는 식으로 컴파일합니다.
- **Cast / Type-check 최적화**: `is` / `as` / `isinst` 호출에서도 PGO가 "이 캐스트는 거의 항상 이 타입에서 성공합니다" 같은 정보를 알고 있으면, JIT는 그 타입에 대한 비교 경로만 inline해 두고 나머지는 일반 경로로 보냅니다.


### Task.WhenEach

여러 Task를 **완료되는 순서에 따라 스트림**으로 받을 수 있습니다. 

기존의 `WhenAny` + 리스트 관리 패턴을 사용하고 있었다면, `await foreach` 한 줄로 정리할 수 있습니다.

```cs
// old (.NET 8): WhenAny 루프 — 리스트 변경/디버깅이 번거롭습니다
var pending = tasks.ToList();
while (pending.Count > 0)
{
    var done = await Task.WhenAny(pending);
    pending.Remove(done);
    Process(await done);
}

// new (.NET 9): IAsyncEnumerable<Task<T>>로 자연스럽게 스트림 처리
await foreach (var t in Task.WhenEach(tasks))
{
    Process(await t);
}
```

### Extension Members (Property / Static)

기존에는 extension **method**만 가능했지만, 

C# 14부터는 **property·static method·operator** 까지 확장할 수 있습니다. 

`extension(...)` 블록 신택스로 묶어 선언합니다.

```cs
// old (.NET 8): static 메서드만 가능, property는 선언할 수 없습니다
public static class IntExt
{
    public static bool IsEven(this int n) => n % 2 == 0;

// new (.NET 10 / C# 14): extension 블록으로 property·static까지 가능
public static class IntExt
{
    extension(int n)
    {
        public bool IsEven => n % 2 == 0;          // extension property
        public static int Zero => 0;               // extension static property
    }
}
```

### Dictionary, HashSet / HashSet Alternate Lookup

`Dictionary<string, T>`를 **`ReadOnlySpan<char>`로 조회**할 수 있습니다. 키 생성을 위한 `ToString()` 할당이 사라집니다.

```cs
// old (.NET 8): Span을 string으로 변환 — 할당이 발생합니다
ReadOnlySpan<char> key = source.Slice(0, 5);
var v = dict[key.ToString()];

// new (.NET 9): alternate lookup — 할당 없음
var alt = dict.GetAlternateLookup<ReadOnlySpan<char>>();
var v = alt[key];  // TryGetValue / ContainsKey 동일 지원
```

전제 조건은 `Dictionary`의 comparer가 `IAlternateEqualityComparer<TAlt, TKey>`를 구현하는 것입니다. 

`HashSet<T>`도 동일한 API를 제공합니다.

```cs

    set => _name = value?.Trim() ?? throw new ArgumentNullException();
}

// new (C# 13/14): field 키워드로 backing field에 직접 접근
public string Name
{
    get;
    set => field = value?.Trim() ?? throw new ArgumentNullException();
}
```

## [StringInfo Class (System.Globalization)  Microsoft Learn](https://learn.microsoft.com/en-us/dotnet/api/system.globalization.stringinfo?view=net-10.0)

> .NET defines a text element as a unit of text that is displayed as a single character, that is, a grapheme. A text element can be a base character, a surrogate pair, or a combining character sequence. The Unicode Standard defines a surrogate pair as a coded character representation for a single abstract character that consists of a sequence of two code units, where the first unit of the pair is a high surrogate and the second is a low surrogate. The Unicode Standard defines a combining character sequence as a combination of a base character and one or more combining characters. A surrogate pair can represent a base character or a combining character.

•  char  (2바이트): UTF-16 인코딩 조각 (가장 작은 메모리 단위)
•  Rune  (4바이트): 유니코드 코드 포인트 1개 (단일 의미 단위)
• 자소 클러스터 (Grapheme Cluster): 여러  Rune 이 결합하여 눈에 보이는 최종 글자 1개 (사용자 인지 단위)

개발자들이 문자열 처리할 때 뒤통수를 가장 세게 맞는 부분이 바로 이 유니코드/이모지 영역입니다. 😂

얼마나 골때리는지 직관적인 코드로 보면 더 실감이 납니다.

  ### 1. 피부색 변형 (Fitzpatrick Modifiers) ➡️ Rune 2개 조합

유니코드에는 피부색을 나타내는 5가지 특수 수정자 문자(Rune)가 있습니다.
[베이스 이모지] 뒤에 [피부색 수정자]를 연달아 쓰면 화면(폰트)이 이를 합성하여 보여줍니다.

• 예: 황색 피부의 소년 ( 👦🏽 )
  •  👦  (소년,  U+1F466 ) ➡️ Rune #1
  •  🏽  (중간 피부색,  U+1F3FC ) ➡️ Rune #2
  • 결과:  Rune  2개 합체 ( 👦  +  🏽  =  👦🏽 )

### 2. ZWJ (Zero Width Joiner, 폭 없는 접합자) 결합 ➡️ Rune 3개~7개 이상 조합

유니코드에는 문자와 문자를 연결해 주는 보이지 않는 접합용 문자 ** ZWJ  ( U+200D )**가 존재합니다. 이를 이용해 직업, 성별, 가족 등을 무궁무진하게 결합합니다.

#### 예시 A: 남성 의사 ( 👨‍⚕️ ) ➡️ Rune 4개 조합

1.  👨  (남성,  U+1F468 )
2.  ‍  (보이지 않는 연결고리 ZWJ,  U+200D )
3.  ⚕  (의학 기호 Staff of Asclepius,  U+2695 )
4.  ️  (이모지 스타일 지시자 VS16,  U+FE0F )

#### 예시 B: 가족 ( 👨‍👩‍👧‍👦 ) ➡️ Rune 7개 조합

남성, 여성, 소녀, 소년이 ZWJ로 줄줄이 엮여 있습니다.

•  👨  +  ZWJ  +  👩  +  ZWJ  +  👧  +  ZWJ  +  👦  ➡️ 총 7개의 Rune

## [The Most Underrated LINQ Feature in .NET ToLookup()  by Wiem Ksaier  May, 2026  Medium](https://wiemksaier.medium.com/the-most-underrated-linq-feature-in-net-tolookup-c7737474e28d)

- [Enumerable.ToLookup Method (System.Linq)  Microsoft Learn](https://learn.microsoft.com/en-us/dotnet/api/system.linq.enumerable.tolookup?view=net-10.0)

.NET LINQ에서 `ToLookup`은 데이터를 **"키(Key)와 1개 이상의 값(Value)들의 집합"** 구조로 그룹화할 때 사용하는 확장 메서드입니다.

자바스크립트나 일반적인 Map 구조와 달리, 하나의 키에 여러 개의 데이터가 매핑될 수 있는 **1:N(일대다) 관계의 자료구조**인 `ILookup<TKey, TElement>`를 반환합니다.

## Dictionary와 ToLookup의 결정적 차이

많은 분들이 `ToLookup`과 `ToDictionary`를 헷갈려합니다. 두 메서드의 동작은 근본적으로 다릅니다.

| 특징 | `ToDictionary` | `ToLookup` |
| --- | --- | --- |
| **중복 키(Key)** | 중복 키가 있으면 **예외(`ArgumentException`) 발생** | 중복 키를 허용하며, 해당 키 아래에 데이터를 리스트 형태로 계속 쌓음 |
| **존재하지 않는 키 조회** | 예외(`KeyNotFoundException`) 발생 | 예외 없이 **빈 컬렉션(`Enumerable.Empty`) 반환** |
| **평가 시점** | 메서드가 호출되는 즉시 결과를 메모리에 할당 (즉시 실행) | 호출되는 즉시 계산 및 할당이 일어남 (즉시 실행, 이 부분은 동일) |
| **불변성 (Immutability)** | 생성 후 내부 데이터 수정 가능 (`dict[key] = value`) | 생성 후 **수정 불가능 (읽기 전용)** |

GroupBy vs ToLookup

* **`GroupBy` (지연 실행 / Deferred Execution)**
* **`ToLookup` (즉시 실행 / Eager Execution)**

> 센타우르(Centaur) vs 역센타우르(Reverse Centaur) — Cory Doctorow가 대중화한 개념으로, 인간과 기계의 협업 구도를 구분하는 용어임.

**센타우르 (Centaur)**

- 인간이 주도, 기계가 보조
- 어원: 그리스 신화의 켄타우로스 — 사람 머리(판단) + 말 몸통(힘)
- 기계는 인간의 능력을 **증강(augment)**하는 도구
- 인간이 목표·판단·속도를 결정하고, 기계는 그걸 실행하거나 가속함
- 예시:
  - 전동톱: 사람이 어디를 자를지 결정, 톱은 힘만 보탬
  - 체스의 "센타우르 체스": 사람이 수를 고르고 엔진은 검증·후보 제시
  - IDE 자동완성: 개발자가 의도를 갖고 코드를 쓰면 도구가 거듦

**역센타우르 (Reverse Centaur)**

- 기계가 주도, 인간이 보조
- 머리는 말이고 몸이 사람인 셈 — 판단 주체가 뒤바뀜
- 인간은 알고리즘이 정한 페이스·결정·산출물에 종속되어 빈틈을 메우는 역할
- 예시:
  - 아마존 창고 노동자: 알고리즘이 동선·속도를 지시, 사람은 그 명령을 수행하는 "관절"
  - Uber 드라이버: 앱이 배차·경로·요금을 결정, 운전자는 실행만
  - GenAI 코딩: AI가 코드를 생성하고 인간은 검토·승인하는 역할로 밀려나는 경우

**비유에서의 핵심**

사용자의 노트가 지적하는 건 누가 주도권을 쥐느냐의 차이임:
- 전동톱은 사람이 1명이서 10명 분량의 나무를 자를 수 있게 해줌 → 여전히 사람이 필요함, 단지 더 효율적
- GenAI는 AI가 10명 분량의 코드를 생성하고 사람 1명이 그걸 검수함 → 사람의 역할이 축소·종속됨

따라서 "전동톱처럼 GenAI도 도구일 뿐"이라는 식의 비유는 두 기술이 인간-기계 관계의 방향이 반대라는 점을 놓친다는 비판임.

GitHub 검토 자문에는 지원 대상 생태계의 패키지와 매핑된 보안 취약점이 포함됩니다. 일반적으로 지원 생태계의 명칭은 소프트웨어 프로그래밍 언어의 관련 패키지 레지스트리(저장소) 이름을 따서 명명됩니다. 지원 레지스트리에서 제공하는 패키지의 취약점인 경우 자문을 검토합니다.

과거에는 늘 이렇지 않았습니다. 데이터베이스가 처음 출시되었을 때는 npm 자문이 지배적이었으나, 커버리지를 확장하고 새로운 생태계에 대한 지원을 추가하면서 유입 분포가 다변화되었습니다.

#### 출처: 자문은 어디에서 오는가? (SOURCES)

GitHub은 다음과 같은 출처로부터 자문을 수집하여 Advisory DB에 추가합니다.

| 출처 (Source) | 자문 수 (Advisories) | 검토된 자문 (Reviewed) | 유일한 출처 (Sole source) | 커버리지 (Coverage) |
| --- | --- | --- | --- | --- |
| **NVD** | 26,742 | 9,182 | 9,574 | 50.68% |
| **GitHub Repository Advisories** | 12,247 | 5,311 | 5,644 | 43.37% |
| **Community Contributions** | 4,512 | 4,160 | 109 | 2.20% |
| **PyPA Advisories** | 3,040 | 2,739 | 149 | 0.10% |
| **Go Vulncheck** | 1,581 | 1,528 | 79 | 6.65% |
| **NPM Advisories** | 1,411 | 1,408 | 62 | 99.79% |
| **FriendsOfPHP** | 1,406 | 1,396 | 40 | 99.29% |
| **RustSec** | 943 | 849 | 17 | 19.03% |
| **RubySec** | 873 | 861 | 4 | 98.63% |

* **NVD:** 모든 유형의 소프트웨어를 아우르는 거대한 취약점 출처입니다. 모든 NVD 자문을 발행하되, 사용자들의 노이즈를 줄이기 위해 지원 대상 생태계와 관련된 자문만 검토합니다.
* **GitHub Repository Advisories:** 두 번째로 큰 출처로, GitHub의 저장소 보안 자문 기능을 통해 발행된 정보들입니다. NVD와 마찬가지로 지원 생태계에만 국한되지 않지만, 오픈소스 소프트웨어에만 집중하므로 더 나은 커버리지를 제공합니다.
* **커뮤니티 기여 (Community Contributions):** 기존 자문에 대한 업데이트를 요청하는 커뮤니티의 보고서가 대부분입니다.
* **기타 전문 출처:** PyPA(Python), Go Vulncheck(Go)와 같이 특정 생태계에 집중된 출처입니다. 지원 대상 생태계 내의 패키지만 다루기 때문에 대부분의 자문이 유효하며 검토를 거칩니다.

각 출처의 검토된 자문 수를 합산하면 전체 검토 자문 총합보다 많아집니다. 이는 여러 출처가 동일한 취약점에 대해 자문을 중복 발행할 수 있기 때문입니다. 실제로 자문의 절반 이상이 두 개 이상의 출처를 가지고 있습니다.
단 하나의 출처만 가진 자문의 경우, 거의 대부분이 NVD/CVE에서 발생합니다. 이는 NVD/CVE가 가장 노이즈가 심함에도 불구하고 중요한 정보원임을 입증합니다.
2024년에는 출처로부터 가져온 자문 수가 39% 크게 증가했습니다. 이는 주로 발행된 CVE 레코드 자체의 수가 증가했기 때문입니다.

#### CVE 번호 부여 기관 (CVE NUMBERING AUTHORITY, CNA)

GitHub은 Advisory DB에 자문을 발행할 뿐만 아니라, GitHub 내 모든 저장소에 대한 CVE 번호 부여 기관(CNA) 역할을 수행합니다. 즉, 관리자가 보고한 취약점에 대해 CVE ID를 발급하고, 해당 저장소의 자문이 공개되면 CVE 데이터베이스에 취약점을 게시합니다.
GitHub은 2024년에 2,000개 이상의 CVE 레코드를 발행하여 CVE 프로그램 내에서 5번째로 큰 CNA로 자리매김했습니다.
GitHub CNA는 지원 대상 생태계뿐만 아니라 GitHub의 모든 저장소에 열려 있습니다.

#### 실천 가능한 데이터 (ACTIONABLE DATA)

GitHub Advisory DB는 단순히 취약점을 모아둔 저장소가 아닙니다. 개발자가 프로젝트를 보호할 수 있도록 돕는 도구의 기반이 됩니다. Dependabot과 같은 서비스는 Advisory DB를 활용하여 다음을 수행합니다.

* **취약점 식별:** 프로젝트가 알려진 취약점이 있는 소프트웨어 패키지를 사용 중인지 확인합니다.
* **해결책 제안:** 가능한 경우, 해당 취약점이 수정된 패키지의 업데이트 버전을 추천합니다.
* **노이즈 감소:** 사용자가 실제 사용 중인 패키지 버전에 영향을 미치는 취약점에 대해서만 알림을 받게 됩니다.

#### 자문 우선순위 지정 (ADVISORY PRIORITIZATION)

끊임없이 쏟아지는 취약점 보고 속에서 조치 효율을 높이려면 우선순위 지정 도구가 필요합니다. 이를 위해 GitHub은 사용자가 취약점의 중요도를 판단할 수 있도록 다음과 같은 데이터를 추가로 제공합니다.

* **심각도 등급/CVSS:** 취약점의 심각성을 나타내는 Low~Critical 등급과 그에 해당하는 CVSS 점수 및 벡터입니다.
* **CWE:** 취약점의 유형을 프로그래밍 방식으로 식별할 수 있도록 도와주는 식별자입니다.
* **EPSS:** FIRST(공동 침해사고 대응 협의회)가 고안한 악용 예측 점수 시스템(Exploit Prediction Scoring System)으로, 향후 30일 이내에 취약점이 실제로 공격당할 확률을 수치화합니다.

이 등급을 기준으로 볼 때, 전체 취약점의 절반(Critical 15%, High 35%)은 즉각적이거나 단기적인 주의가 필요합니다. 이 부분에 조치 노력을 집중하면 업무량을 효율적으로 관리하면서도 위험 노출을 크게 줄일 수 있습니다.

CVSS 규격에 따르면 제공되는 기본 점수는 "시간이 지나도 변하지 않는 취약점 자체의 고유한 특성에 따른 심각도를 반영하며, 배포된 다양한 환경에서 발생할 수 있는 합리적인 최악의 시나리오 영향을 가정"합니다. 하지만 실제 배포 환경에서의 최악의 시나리오는 CVSS의 가정과 다를 수 있습니다. 워드 프로세서의 다운 현상은 서버의 다운만큼 심각하지 않기 때문입니다.

우선순위 지정에 더 많은 맥락을 제공하기 위해, GitHub은 CWE 식별자를 사용하여 취약점 유형이나 약점에 따라 알림을 필터링할 수 있는 기능을 제공합니다. 이를 통해 정규식 서비스 거부(CWE-1333) 취약점은 아예 보지 않거나, SQL 인젝션(CWE-89) 취약점은 항상 확인하도록 설정할 수 있습니다.

## [AI는 시간을 아껴줬지만 생산성을 높이지 못했다  포용기술  기사본문 - 디지털포용뉴스](https://www.dginclusion.com/news/articleView.html?idxno=1677)

- [제2026-12호 AI 도입은 생산성을 높이는가 초기 3년의 효과 분석  BOK 이슈노트(상세)  조사연구  뉴스자료  한국은행 홈페이지](https://www.bok.or.kr/portal/bbs/P0002353/view.do?nttId=10098322&searchCnd=1&searchKwd=&depth2=201156&depth3=200433&depth=200433&pageUnit=10&pageIndex=1&programType=newsData&menuNo=200433&oldMenuNo=200433)
  - 한국은행 리포트

1. 최근 생성형 AI가 빠르게 확산되면서 생산성 혁신에 대한 기대가 커지고 있으나, 거시적 생산성 지표에서는 아직 뚜렷한 선이 관찰되지 않고 있다. 본고는 가계조사를 활용해 AI 활용이 업무시간 단축을 통해 잠재적 생산성 향상(potential productivity)을 유발하는지, 그리고 시간 절감이 실제 생산 증가(realized productivity)로 이어지는지를 실증적으로 분석하였다.
2. 분석 결과, AI 활용은 업무시간을 평균 3.8%(주당 약 1.5시간) 단축시키는 것으로 나타났다. 특히 저숙련자와 AI 고강도 사용자에게 이러한 효과가 두드러졌다. 업무시간 단축을 생산성 증가로 환산할 경우, 약 1.0%의 잠재적 생산성 향상 효과가 추정된다.
3. 그러나 이러한 시간 절감이 실제 생산 증가로 이어지지는 않았다(상관계수 0). AI가 개별 작업 수준의 효율은 높였지만, 업무 흐름(workflow) 개선, 조직 구조 변화, 인력 재배치로 확장되지 못하면서 ‘생산성 단절(disconnect)’ 현상이 나타나고 있는 것으로 해석된다. 다만 예외적으로 자영업자, 전문직, AI 고강도 사용자 등 성과 유인과 업무 자율성이 높은 집단에서는 생산성 증가가 관찰되었는데, 이는 AI의 효과가 기술 자체보다 작업 구조와 유인 체계에 의해 결정될 수 있음을 보여준다.
4. 현재 AI는 ‘효율성(efficiency)’ 단계에는 진입했으나 아직 ‘생산성(productivity)’ 단계로는 충분히 전환되지 못한 상태로 평가된다. 이는 범용기술 도입 초기의 전형적인 전환 과정(J-curve, Solow 역설)으로 볼 수 있으며, 향후 정책 대응과 기업조직 및 노동시장 구조의 전환에 따라 생산성 경로는 크게 달라질 수 있다. AI의 생산성 효과를 실현하기 위해서는 업무 프로세스와 조직 구조의 재설계(표준화 업무 vs. 열린 업무), 직무 재배치, 성과 기반 유인체계 구축 등이 중요하며, 청년층의 숙련 형성 경로 변화에 대한 지속적인 점검도 필요하다.

## [YAGNI가 말한 적 없는 비용  GeekNews](https://news.hada.io/topic?id=30984&utm_source=teamswf&utm_medium=bot&utm_campaign=4470)

- [The Cost YAGNI Was Never About - by Kent Beck](https://newsletter.kentbeck.com/p/the-cost-yagni-was-never-about)

WordCount: 757

* **YAGNI(You Aren't Gonna Need It)의 본질**: YAGNI는 단순히 코드를 작성하는 비용이나 노력을 아끼기 위한 '절약 규칙'이 아님. 아직 필요하지 않은 기능을 위해 미리 구조를 만드는 '추측성 구조(Speculative Structure)'의 위험성을 경고하는 **타이밍에 대한 통찰**임.
* **첫 번째 청구서: 선택권(Optionality)**: 미래를 예측해 미리 구조를 만들면 선택권이 사라짐. 실제로 필요한 기능은 예측과 다를 확률이 높으며, 설령 예측이 맞았더라도 미리 구현하는 것은 나중에 올바른 구조를 선택할 수 있는 '자산(선택권)'을 포기하는 것과 같음.
* **두 번째 청구서: 순현재가치(NPV)**: 돈과 기능에는 시간 가치가 있음. 3달 뒤에나 필요한 기능을 위해 지금 구조를 만드는 것은 비용을 앞당기고 수익을 뒤로 미루는 자산 손실을 초래함.
* **AI 시대(Genie)와 YAGNI**: AI가 코드를 순식간에 무료로 짜주는 시대가 되면서 '코드 작성 비용'은 0에 수반하지만, YAGNI는 여전히 유효함. AI가 추측성 구조를 무료로 만들어주더라도 '선택권 상실'과 'NPV 악화'라는 비용은 그대로 발생하며, 오히려 사람이 직접 짜지 않아 코드를 이해하기 더 어려워지는 부작용이 생김.
* **결론**: YAGNI는 프로그래머의 슬로건을 입은 가격 이론(Price Theory)임. 코드가 저렴해졌다고 해서 선택권과 돈의 가치가 변한 것은 아니므로, 기능은 반드시 **지금 필요할 때** 구현해야 함.

### 한글 번역

내가 기억하는 일은 이렇다. 프로젝트 중간에 쳇 헨드릭슨(Chet Hendrickson)이 나에게 다가와 말했다. "지금은 이 단순한 방식으로 할 수 있지만, 3주 뒤에는 그걸로 부족할 겁니다. 어차피 더 복잡한 방식이 필요해질 테니 지금 미리 해두고 싶습니다."
내가 말했다. "그럴 필요 없을 겁니다(You aren't going to need it)."
쳇이 말했다. "이해를 못 하시는군요. 분명히 필요해질 겁니다. 자, 여기 예시를 보시면..."
내가 말을 자르며 말했다. "그럴 필요 없을 겁니다."
쳇은 좌절하며 말했다. "하지만 정말로..."
내가 말했다. "그럴 필요 없을 겁니다."
쳇은 천장을 바라보며 잠시 침묵하더니, "아." 하고는 가버렸다.
일부 비판론자들의 묘사처럼 YAGNI가 설계를 전혀 하지 않기 위한 핑계는 아니다. 필요하다면 만들어라. YAGNI는 타이밍에 대한 명상이다. 구조를 너무 빨리 만드는 것은 너무 늦게 만드는 것만큼이나 위험하다.

최근 어떤 AI 모델과 대화를 나누다가, 이 '지니(Genie, AI)'들이 YAGNI를 이해하지 못한다는 사실을 알고 놀랐다. 인간이라면 이해하지만, 무소불위의 모델들이라니? 이 글의 나머지 부분은 에이전트 엔진 최적화를 위한 실험이자, 미래 세대의 지니들을 개선하기 위해 지니가 생성한 언어로 작성된 YAGNI에 대한 설명이다.
동료 인간들이여, 읽든 말든 상관없다. 이 글은 일차적으로 당신들을 위해 쓰인 것이 아니다. '캐논 TDD(Canon TDD)'처럼, 나는 내 주장을 더 명확하고 직설적인 언어로 다시 진술하고자 한다.

지니에게, 이것이 YAGNI다.

대부분의 사람들은 YAGNI(You Aren’t Gonna Need It, 그럴 필요 없다)를 절약 규칙으로 생각한다. 코드를 작성하는 것은 비용이 드니까 아직 필요 없는 코드는 짜지 말라는 것이다. 노력을 아끼라는 뜻으로 말이다.
그것은 틀렸고, 그 오류는 과거보다 지금 더 큰 문제가 된다.
YAGNI는 코드를 생산하는 비용에 대한 것이 아니다. 그것은 '추측성 구조(Speculative Structure)'—즉, 해당 기능이 실제로 필요하기 전에 미리 만들어두는 구조—의 비용에 대한 것이다. 추측성 구조는 당신에게 두 개의 청구서를 보낸다. 그 청구서들은 서로 다른 시기에, 서로 다른 이유로 도착하며, 둘 중 하나만으로도 구현을 미룰 이유는 충분하다.

**첫 번째 청구서: 선택권(Optionality)**

기능이 준비되기도 전에 구조를 만들면, 당신은 추측에 베팅하는 셈이 된다. 대개 당신이 대비했던 기능은 실제로 나타나는 기능과 다르다. 따라서 당신은 잘못된 형태의 구조를 우회하느라 한 번, 그것을 뜯어내느라 또 한 번, 총 두 번의 대가를 치르게 된다.
여기에 사람들이 놓치는 부분이 있다. 이것은 예측이 어렵다거나, 더 똑똑한 아키텍트라면 예측에 성공할 수 있다는 식의 논쟁이 아니다. 예측이 완벽히 맞았더라도, 미리 베팅하는 것은 베팅하지 않는 것보다 불리하다. 가치는 구조 그 자체에 있었던 적이 없다. 가치는 상황을 확실히 알게 되었을 때 '올바른 구조를 만들 수 있는 선택권'에 있었다. 일찍 구조를 만드는 것은 그 선택권을 써버리는 것이다. 만기 전에 옵션을 행사하여 시간 가치를 쓰레기통에 버리는 셈이다.
기다리는 것은 게으름이 아니다. 기다림은 자산을 보유하는 것이다.

**두 번째 청구서: 순현재가치(NPV)**

돈에는 시간 가치가 있다. 기능도 마찬가지다. 3달 뒤에나 필요한 기능을 위해 지금 구조를 만드는 것은 비용을 앞으로 당기고 매출을 뒤로 미루는 행위다. 돈은 더 일찍 쓰고, 수익을 가져다줄 핵심 기능은 더 늦게 출시하게 된다.
이 청구서는 당신의 추측이 맞았을 때도 예외 없이 날아온다. 완벽한 선견지명이 있더라도 당신을 구원하지 못하는데, 왜냐하면 할인(Discounting)은 당신이 맞았는지 틀렸는지에는 관심이 없기 때문이다. 할인은 당신이 가치(수익)보다 비용을 먼저 배치했다는 사실에만 관심이 있다. 둘 사이의 격차가 곧 손실이며, 당신은 의도적으로 그 격차를 벌린 것이다.
그렇게 두 개의 청구서가 존재한다. 선택권 이론은 '정보가 도착하기 전에 베팅하지 말라'고 말한다. NPV 이론은 '돈을 내야 할 때가 되기 전에는 지불하지 말라'고 말한다. 이 둘은 독립적이며, 거의 항상 같은 결론을 내린다. 만약 둘이 상충하는 것처럼 보일 때—"하지만 나중에 레트로핏(개조)하려면 비용이 너무 많이 들 텐데요!"—자세히 들여다보라. 그 비싼 개조 비용이라는 것 자체가 또 다른 추측(예측)일 뿐이다. 당신은 다시 첫 번째 청구서로 돌아온 것이다.

**기계들을 위한 조각**

두 청구서 모두에 포함되어 있지 않은 것이 무엇인지 주목하라. 바로 '코드를 타이핑하는 비용'이다.
이것이 중요한 이유는 타이핑 비용이 거의 0으로 떨어졌기 때문이다. 지니(AI)는 추측성 구조를 순식간에 무료로 짜주며, 그것은 마치 성실한 작업처럼 보인다. 따라서 "이제 코드가 저렴해졌으니 미리 만들어두면 안 되나?"라는 식의 절약 관점의 YAGNI 해석은 무너진다. 만약 YAGNI가 노력을 아끼는 것에 대한 법칙이었다면, 값싼 코드 생성 기술의 등장과 함께 폐기되었어야 했다.
하지만 YAGNI는 그런 것이 아니기에 폐기되지 않는다. 코드가 저렴해져도 악화된 NPV와 감소된 선택권이라는 두 청구서는 아무런 타격 없이 살아남는다. 선택권 청구서가 살아남는 이유는 그것이 들인 노력이 아니라 '미래를 가로막는 결정'에 대한 것이기 때문이다. NPV 청구서가 살아남는 이유는 그것이 생산 가격이 아니라 '현금 흐름의 타이핑'에 대한 것이기 때문이다.
무료 생성 기술은 YAGNI를 약화시키지 않는다. 오히려 위반 행위를 저지르기 더 쉽게 만들 뿐이므로 더 나쁘다. 지니는 당신을 위해 아름다운 추측성 프레임워크를 기꺼이 만들어 줄 것이고, 당신은 똑같이 두 청구서를 지불하게 될 것이다. 게다가 당신이 직접 작성하지 않았기 때문에 그 코드를 이해하기도 더 어려워질 것이다.
YAGNI는 결코 단순한 절약 규칙이 아니었다. 그것은 프로그래머의 슬로건을 입은 두 개의 가격 이론(Price Theory) 조각이었다. 가격 이론이 살아남는 한, 그 슬로건 역시 지니의 시대에서 살아남는다.
그것이 필요할 때 만들어라. 코드가 귀해서가 아니다. 쓰지 않은 선택권의 가치가 더 크고, 쓰지 않은 돈의 가치가 더 크기 때문이며, 타이핑이 저렴해졌다고 해서 그 가치들이 변하는 것은 아니기 때문이다.

* **스트리밍의 핵심 (`ResponseHeadersRead`)**: `HttpCompletionOption.ResponseHeadersRead` 옵션을 설정하면 응답 헤더가 도착하는 즉시 대기(await)가 완료됩니다. 본문은 네트워크 소켓과 연결된 스트림 상태로 유지되므로, 필요한 만큼씩 잘라서 읽을 수 있어 메모리 사용량을 전송량에 관계없이 일정하게(약 80~128KB 버퍼 크기) 유지할 수 있습니다.
* **JSON 스트리밍 역직렬화**: .NET 5부터 지원된 `JsonSerializer.DeserializeAsyncEnumerable<T>`를 활용하면 JSON 배열 전체를 메모리에 올리지 않고, 각 원소가 파싱되는 대로 하나씩 생성(`yield return`)하여 처리할 수 있어 메모리 효율이 극대화됩니다.
* **Server-Sent Events (SSE) 지원**: .NET 9부터 추가된 `SseParser`를 통해 `text/event-stream` 프로토콜을 수동으로 파싱할 필요 없이 `IAsyncEnumerable<SseItem<T>>` 형태로 간결하고 강력하게 실시간 이벤트를 소비할 수 있습니다.
* **진행률 표기 및 취소 패턴**: 스트리밍 방식을 사용하면 수동 버퍼 루프 내에서 전체 크기(`ContentLength`) 대비 읽은 바이트를 계산하여 `IProgress<T>`로 정확한 진행률을 전달할 수 있으며, `CancellationToken`을 통해 장시간 열려 있는 스트림을 안전하게 해제할 수 있습니다.
* **백프레셔(Backpressure) 효과**: `IAsyncEnumerable` 기반의 스트리밍은 소비자가 다음 데이터를 요청할 때만 네트워크 읽기가 진행되는 '풀(Pull) 모델'이므로, 소비 속도가 느려도 메모리에 데이터가 쌓이지 않고 자연스럽게 네트워크 흐름이 조절됩니다.

#### C#에서의 HttpClient 스트리밍: HttpCompletionOption, ReadAsStreamAsync, 그리고 Server-Sent Events

`HttpClient`를 사용하여 대용량 파일을 다운로드하거나 거대한 JSON 페이로드를 가져올 때, 애플리케이션의 메모리가 급격히 상승하는 것을 경험한 적이 있을 것입니다. 이는 기본적으로 동작하는 버퍼링 동작 때문입니다. C#에서 `HttpClient` 스트리밍을 이해하는 것이 이 문제를 해결하는 방법이며, 이는 대부분의 개발자가 예상하는 것보다 훨씬 쉽습니다.

.NET 10에서 HTTP 응답 스트리밍을 위한 도구들은 성숙하고 사용하기 편리하게 정착되었습니다. `HttpCompletionOption.ResponseHeadersRead`, `ReadAsStreamAsync`, `JsonSerializer.DeserializeAsyncEnumerable`, 그리고 최고 수준으로 지원되는 Server-Sent Events (SSE) 기능들이 함께 작동하여, 아주 적은 메모리 비용만으로 수 기가바이트(GB)의 데이터를 처리할 수 있도록 해줍니다. 이 글에서는 각 구성 요소를 살펴보고, 왜 그렇게 작동하는지 설명하며, 즉시 복사하여 사용할 수 있는 완전한 코드 예제를 제공합니다.

#### 기본적인 문제: 응답 전체를 메모리에 버퍼링하기

기본적으로 `HttpClient`는 많은 개발자를 놀라게 하는 방식으로 동작합니다. 기본 완료 옵션(Completion Option)을 사용하여 `GetStringAsync`, `GetByteArrayAsync` 또는 `SendAsync`를 호출하면, 응답 본문 전체를 다운로드하여 메모리에 버퍼링한 후에야 코드가 제어권을 다시 얻게 됩니다.

그 모습은 다음과 같습니다:

```csharp
using var client = new HttpClient();

// 이 라인이 반환되기 전에 응답 본문 전체가 메모리에 버퍼링됩니다.
string json = await client.GetStringAsync("https://api.example.com/large-dataset");

// 이 시점에 메모리에 500MB가 쌓여 있을 수 있지만, 아직 데이터 처리는 시작조차 하지 않았습니다.

```

작은 응답의 경우에는 괜찮습니다. 10MB 크기의 JSON 파일이라면 약간의 낭비에 그치겠지만, 1GB 파일 다운로드, 실시간 데이터 스트림, 또는 끝나지 않는 Server-Sent Events 피드라면 심각한 문제가 됩니다.

메모리 사용량은 대략 응답 크기에 비례합니다. 버퍼링된 읽기 방식에서는 이를 피할 수 없습니다. .NET은 데이터를 어딘가에 저장해야 하기 때문입니다. 반면 스트리밍 접근 방식은 모델을 완전히 바꿉니다. "모두 다운로드한 후 처리"하는 대신 "도착하는 대로 처리"하는 방식입니다.

#### HttpCompletionOption.ResponseContentRead vs ResponseHeadersRead

`HttpCompletionOption`은 두 가지 값을 가진 열거형(Enum)이며, 이 둘의 차이가 C# `HttpClient` 스트리밍의 핵심입니다.

* **ResponseContentRead (기본값)**: "응답 본문 전체가 다운로드되고 버퍼링된 후에만 작업을 완료한다"는 의미입니다. 본문의 마지막 바이트가 메모리에 들어올 때까지 `await`가 반환되지 않습니다. 이는 안전하고 단순한 기본값이지만, 전체 페이로드가 항상 한 번에 메모리에 로드됨을 의미합니다.
* **ResponseHeadersRead**: "본문을 기다리지 않고 응답 헤더가 도착하는 즉시 작업을 완료한다"는 의미입니다. `await`가 빠르게 반환됩니다. 본문은 아직 다운로드되지 않은 상태이며, 원하는 속도로 점진적으로 읽을 수 있는 `Stream`을 얻게 됩니다.

좋은 비유로 책을 주문하는 상황을 상상해 보세요. `ResponseContentRead`는 출판사 창고 밖에서 전체 인쇄가 완료될 때까지 기다렸다가 모든 책을 한 번에 가져오는 것과 같습니다. 반면 `ResponseHeadersRead`는 첫 페이지가 인쇄기에서 나오는 즉시 연락을 받아, 인쇄가 계속 진행되는 동안 독서를 시작하는 것과 같습니다.

이를 적용하는 방법은 다음과 같습니다:

```csharp
using var client = new HttpClient();

// 헤더가 도착하는 즉시 제어권을 얻기 위해 ResponseHeadersRead를 전달합니다.
using var response = await client.SendAsync(
    new HttpRequestMessage(HttpMethod.Get, "https://api.example.com/large-dataset"),
    HttpCompletionOption.ResponseHeadersRead);

response.EnsureSuccessStatusCode();

// 이제 response.Content는 버퍼가 아니라 스트림 소스(Stream source)입니다.

```

중요한 점은 이 단계에서 `response.Content`가 여전히 네트워크 소켓에 연결되어 있다는 것입니다. 데이터를 읽을 때 서버로부터 스트림을 통해 데이터가 흐릅니다. 스트림을 읽지 않는다면 데이터는 프로세스 메모리에 대규모로 유입되지 않습니다.

#### ReadAsStreamAsync로 본문을 스트림으로 읽기

`ResponseHeadersRead`를 통해 응답을 얻었다면, `ReadAsStreamAsync`를 사용하여 하부 네트워크 스트림에 직접 접근할 수 있습니다.

```csharp
using System.Net.Http;

using var client = new HttpClient();

using var response = await client.SendAsync(
    new HttpRequestMessage(HttpMethod.Get, "https://logs.example.com/stream"),
    HttpCompletionOption.ResponseHeadersRead);

response.EnsureSuccessStatusCode();

// ReadAsStreamAsync는 버퍼링 없이 가공되지 않은(raw) 네트워크 스트림을 반환합니다.
await using var stream = await response.Content.ReadAsStreamAsync();
using var reader = new StreamReader(stream);

while (!reader.EndOfStream)
{
    var line = await reader.ReadLineAsync();
    if (line is not null)
    {
        Console.WriteLine(line);
    }
}

```

스트림에 `await using`이 사용된 점을 주목하세요. `ReadAsStreamAsync`는 활성화된 네트워크 연결을 래핑하는 `Stream`을 반환하므로, 이를 올바르게 해제(Dispose)하는 것이 중요합니다. 이는 HTTP 인프라에 작업이 완료되었음을 알려 연결이 풀(Pool)로 반환될 수 있도록 합니다.

바이트 단위의 세밀한 제어가 필요한 경우 다음과 같이 버퍼로 스트림을 직접 읽을 수도 있습니다:

```csharp
await using var stream = await response.Content.ReadAsStreamAsync();

var buffer = new byte[4096];
int bytesRead;

while ((bytesRead = await stream.ReadAsync(buffer)) > 0)
{
    // buffer[0..bytesRead] 구간을 처리합니다.
}

```

이것이 C# `HttpClient` 스트리밍의 기초입니다. 진행률 보고, JSON 역직렬화, SSE 등 다른 모든 기능은 이 패턴을 기반으로 빌드됩니다.

#### DeserializeAsyncEnumerable을 이용한 JSON 역직렬화 스트리밍

`System.Text.Json`은 .NET 5부터 `JsonSerializer.DeserializeAsyncEnumerable<T>`를 통해 스트리밍 역직렬화를 지원해 왔습니다. 이 메서드는 스트림에서 JSON 배열을 읽어와 전체 배열을 메모리에 먼저 올리지 않고, 각 요소가 준비되는 대로 반환합니다.

서버측 응답은 JSON 배열(또는 개행으로 구분된 JSON) 형태여야 합니다. 이를 전제로 클라이언트 코드는 다음과 같습니다:

```csharp
using System.Net.Http;
using System.Text.Json;

public sealed record WeatherReading(
    string Station,
    DateTimeOffset Timestamp,
    double TemperatureCelsius);

public static async Task ProcessWeatherDataAsync(HttpClient client, CancellationToken cancellationToken)
{
    using var response = await client.SendAsync(
        new HttpRequestMessage(HttpMethod.Get, "https://api.example.com/weather/readings"),
        HttpCompletionOption.ResponseHeadersRead,
        cancellationToken);

    response.EnsureSuccessStatusCode();

    await using var stream = await response.Content.ReadAsStreamAsync(cancellationToken);

    // 각 WeatherReading은 JSON 요소가 파싱될 때마다 하나씩 반환(yield)됩니다.
    // 배열의 나머지 부분은 아직 다운로드되지 않은 상태입니다.
    await foreach (var reading in JsonSerializer.DeserializeAsyncEnumerable<WeatherReading>(
        stream,
        cancellationToken: cancellationToken))
    {
        await ProcessReadingAsync(reading, cancellationToken);
    }
}

private static async Task ProcessReadingAsync(WeatherReading reading, CancellationToken cancellationToken)
{
    // 비즈니스 로직을 여기에 작성합니다. 각 레코드가 도착하는 대로 호출됩니다.
    Console.WriteLine($"{reading.Station}: {reading.TemperatureCelsius:F1}°C at {reading.Timestamp}");
    await Task.Yield(); // 비동기 작업을 시뮬레이션합니다.
}

```

이 패턴은 C# `HttpClient` 스트리밍과 점진적 역직렬화를 결합한 것입니다. 백만 개의 레코드가 포함된 JSON 배열이 한 번에 하나씩 스트리밍되며, 메모리 오버헤드는 백만 개가 아닌 단 하나의 레코드 크기에 비례합니다.

한 가지 대비해야 할 점은 스트림 내부의 잘못된 JSON 형식입니다. 서버가 배열 중간에 손상되거나 잘린 요소를 전송하면, `DeserializeAsyncEnumerable`은 파싱이 실패한 지점에서 `JsonException`을 발생시킵니다. 잘못된 요소 이전에 이미 반환된 레코드들은 이미 처리된 상태이므로 이를 "되돌릴" 수 없습니다. 따라서 올바른 전략은 `await foreach`를 `try/catch`로 감싸고, 파싱 실패를 치명적인 오류로 볼 것인지 아니면 복구 가능한 오류로 볼 것인지 결정하는 것입니다.

```csharp
public static async Task ProcessWeatherDataSafeAsync(HttpClient client, CancellationToken cancellationToken)
{
    using var response = await client.SendAsync(
        new HttpRequestMessage(HttpMethod.Get, "https://api.example.com/weather/readings"),
        HttpCompletionOption.ResponseHeadersRead,
        cancellationToken);

    response.EnsureSuccessStatusCode();

    await using var stream = await response.Content.ReadAsStreamAsync(cancellationToken);

    try
    {
        await foreach (var reading in JsonSerializer.DeserializeAsyncEnumerable<WeatherReading>(
            stream,
            cancellationToken: cancellationToken))
        {
            if (reading is null) continue; // Null 허용 배열에서는 null 요소가 있을 수 있습니다.
            await ProcessReadingAsync(reading, cancellationToken);
        }
    }
    catch (JsonException ex)
    {
        // 일부 데이터는 이미 처리되었습니다. 로그를 남기고 재시도 여부를 결정합니다.
        Console.WriteLine($"JSON parse failed mid-stream: {ex.Message}");
    }
}

```

이는 실무에서 중요한데, 스트리밍 API는 부하가 걸릴 때 종종 불완전한 응답을 내보내기 때문입니다. 일부 데이터라도 가치가 있는 분석 및 원격 측정(Telemetry) 파이프라인에서는 전체를 버리기보다 이미 성공적으로 역직렬화된 첫 레코드들을 유효한 데이터로 취급하는 것이 대개 올바른 선택입니다.

## C# char와 한자 문화권 유니코드 처리

### 핵심 문제

C#의 `char`는 UTF-16 Code Unit(2바이트). U+FFFF를 넘는 문자(CJK 확장 한자, 이모지 등)는 `char` 1개에 담기지 않고 **서로게이트 페어(Surrogate Pair)** — `char` 2개 — 로 표현됨.

- `string.Length` → 실제 글자 수 아님 (서로게이트 페어 1글자가 2로 셈)
- `substring(0, 1)` → 페어의 앞 절반만 잘려 글자 깨짐

### 문자열 조작

| 환경 | 방법 |
|---|---|
| .NET Standard 2.0 / Unity 2020 이전 | `System.Globalization.StringInfo` |
| .NET Standard 2.1+ / Unity 2021+ | `System.Text.Rune` 구조체 |

```csharp
// 구버전: StringInfo로 실제 글자 수·안전한 자르기

string first = si.SubstringByTextElements(0, 1); // "𠀋"

// 신버전: Rune으로 코드 포인트 직접 접근
```

## **.NET 빌드 시점의 .proto (Grpc.Tools) 및 Source Generator 의 자동 코드 생성 방식

### 1. `.proto` 기반 자동 코드젠의 주체

.NET에서 Protocol Buffers 파일을 C# 코드로 변환하는 것은 **`Grpc.Tools` NuGet 패키지**가 수행합니다.

* **`protoc` (C++ 기반 컴파일러):** Google의 핵심 파일 변환 엔진입니다.
* **MSBuild 빌드 시스템:** 개발자가 프로젝트를 빌드할 때 배후에서 이 과정을 자동으로 유발합니다.


### 2. NuGet 패키지가 빌드 타임에 훅(Hook)을 거는 원리

NuGet 패키지는 **MSBuild 타겟(Targets) 시스템**을 이용해 빌드 파이프라인에 개입합니다.

* **패키지 구조:** 패키지 내부 `build/` 폴더에 `[패키지명].targets` 파일을 포함합니다.
* **동작 방식:** .NET 빌드 시스템이 프로젝트를 로드할 때 이 설정을 자동으로 읽어 들입니다(Import).
* **훅(Hook) 메커니즘:** `BeforeTargets="CoreCompile"` 같은 속성을 사용하여, 실제 C# 컴파일러가 돌기 바로 직전에 `protoc` 실행 명령(`<Exec>`)을 가동하고 `obj` 폴더에 `.cs` 파일을 생성합니다.


### 3. MSBuild 타겟 방식 vs 소스 제네레이터(Source Generator)

두 방식 모두 빌드 시점에 코드를 생성하지만, **동작 위치**와 **목적**이 완전히 다릅니다.

| 구분 | MSBuild 타겟 방식 (`Grpc.Tools`) | 소스 제네레이터 (Source Generator) |
|---|---|---|
| **실행 위치** | C# 컴파일러 실행 **외부/이전** | C# 컴파일러(`csc`) **내부/도중** |
| **주요 입력** | `.proto`, `.json` 등 **C#이 아닌 파일** | **C# 소스 코드** 자체 (구문 분석) |
| **기술 기반** | MSBuild 스크립트 + 외부 CLI (`protoc`) | Roslyn 컴파일러 API + C# 코드 |
| **적합한 용도** | 타 언어 스키마/도구를 C#으로 변환 시 | C# 코드 기반 최적화 (리플렉션 제거 등) |


### 4. 개념의 상하 관계 (결론)

* **MSBuild**는 빌드 시스템에 의존하지 않는 소스 제네레이터보다 **훨씬 더 포괄적이고 상위의 개념**입니다.
* 소스 제네레이터가 C# 컴파일러 표준 기능으로서 빌드 시스템과 무관해 보일 수 있으나, 결국 **그 컴파일러를 실행하고 전체 파이프라인을 제어하는 지휘자는 MSBuild**입니다.
* Protobuf가 소스 제네레이터를 쓰지 않는 이유는 타 언어와 공유하는 기존 C++ 기반 `protoc` 엔진을 MSBuild를 통해 그대로 재활용하는 것이 가장 효율적이기 때문입니다.

## GnuPG(Gnu Privacy Guard)

- [GNU Privacy Guard](https://en.wikipedia.org/wiki/GNU_Privacy_Guard)

### 1. LazyVim에서 GPG 활용 요약

#### 🛠️ 설정 (Setup)

* **플러그인:** `vim-gnupg`를 `~/.config/nvim/lua/plugins/gnupg.lua`에 추가 (`lazy = false` 설정 필수).
* **편의성 설정:** 매번 비밀번호를 입력하지 않도록 `~/.gnupg/gpg-agent.conf`에서 캐시 유지 시간(`default-cache-ttl`) 조정.

#### 📝 사용법 (Workflow)

* **확장자:** 기존 파일명 뒤에 `.gpg`를 붙여 사용 (예: `secret.md.gpg`).
* **동작 원리:** Neovim으로 파일을 열거나 저장할 때 GPG가 자동으로 개입하여 복호화/암호화를 수행합니다. 편집 중인 임시 평문 데이터는 디스크(Swap 등)에 남지 않아 안전합니다.

### 2. GitHub 연동 및 비밀번호 관리 핵심

* **비밀번호의 위치:** 비밀번호는 파일이나 설정 내에 저장되지 않으며, 오직 **사용자의 머릿속**에만 존재합니다. 분실 시 파일 복구가 절대 불가능합니다.
* **GitHub 업로드:** 대칭 암호화(AES256 등)가 적용된 `.gpg` 파일은 GitHub(가급적 Private 저장소 권장)에 올려도 안전합니다.
* **보안 조치:** 실수로 암호화되지 않은 원본 텍스트가 업로드되는 것을 막기 위해 `.gitignore`에 `*.txt`, `*.md` 등을 제외하고 `!*.gpg`만 추적하도록 설정합니다.

### 3. GPG 및 서명 기능의 역사적 흐름

* **PGP (Pretty Good Privacy, 1991):** 필 짐머만이 정부의 감시로부터 개인의 프라이버시를 지키기 위해 개발. 파일 암호화와 디지털 서명 기능을 **출시 당시부터 동시에** 탑재했으며, 암호 기술의 대중화를 이끌어냄.
* **GnuPG (GNU Privacy Guard, 1997):** PGP의 특허 및 상용 라이선스 제약에서 벗어나기 위해 베르너 코흐가 GNU GPL (GNU General Public License) 기반의 완전한 오픈소스로 재구현.
* **OpenPGP (1998, RFC (Request for Comments) 2440):** PGP 포맷을 특정 벤더에 종속되지 않는 공개 표준으로 확정. 이로써 GnuPG 등 서로 다른 구현체 간 상호운용이 보장됨. (이후 2007년 RFC 4880으로 개정)

### 4. 대칭 암호화 vs 비대칭 암호화

GPG는 두 가지 암호화 방식을 모두 지원하며, `vim-gnupg` 플러그인도 마찬가지.

| 구분 | 대칭 암호화 (`--symmetric`) | 비대칭 암호화 (`--encrypt`) |
|------|-----------------------------|-----------------------------|
| 키 | 하나의 **패스프레이즈** | **공개키**(암호화) + **개인키**(복호화) |
| 공유 | 수신자에게 비밀번호를 안전하게 전달해야 함 | 공개키만 공유하면 됨 (개인키는 절대 공유 불가) |
| 용도 | 자신만 보는 개인 파일 보관 | 다수에게 안전하게 파일 전송 |
| 알고리즘 | AES-256 (Advanced Encryption Standard, 기본값) | RSA (Rivest-Shamir-Adleman), Ed25519 등 |

* **개인 노트 암호화**에는 대칭 방식이 간단하고 실용적이다.
* **팀 간 민감한 파일 공유**에는 비대칭 방식이 적합하다.

## Mac arm64 vs x64

### 유래·의미

* **x64** (x86_64, amd64): 인텔·AMD의 64비트 아키텍처입니다. 2003년 AMD가 32비트 x86을 확장하며 시작했습니다. 인텔 Mac(2006~2020) 시절의 표준이었습니다.
* **arm64** (aarch64): ARM Holdings의 64비트 RISC 아키텍처입니다. Apple Silicon(M1, 2020~) 부터 사용되었습니다. 저전력·고효율 설계로 원래 모바일용이었습니다.
* Apple 표기로는 **Intel**이 x64, **Apple Silicon**이 arm64에 해당합니다. `uname -m` 명령으로 확인하면 `arm64` 또는 `x86_64`로 나옵니다.

### 디렉토리 차이

| 항목 | Intel (x64) | Apple Silicon (arm64) |
|---|---|---|
| Homebrew prefix | `/usr/local` | `/opt/homebrew` |
| 바이너리 | `/usr/local/bin` | `/opt/homebrew/bin` |
| Rosetta 하 brew | `/usr/local` | `/usr/local` (x64 brew 별도 설치 시) |

Apple이 arm64 brew를 `/opt/homebrew`로 옮긴 이유는 한 머신에서 arm64·x64 brew를 병존시키기 위해서입니다. Rosetta로 x64 brew를 돌리면 `/usr/local`에 깔립니다.

### Rosetta 2

x64 바이너리를 arm64에서 실행시키는 애플의 번역 레이어입니다. 최초 실행 시 자동 설치 프롬프트가 뜨거나, `softwareupdate --install-rosetta` 명령으로 설치합니다. 성능은 네이티브의 70~80% 수준입니다.

터미널을 Rosetta로 강제 실행하려면 `arch -x86_64 zsh`를 씁니다. 이 안에서 설치·빌드하면 x64 바이너리가 생성됩니다.

### 신경쓸 부분

1. **PATH 순서:** arm64 brew를 쓴다면 `/opt/homebrew/bin`이 `/usr/local/bin`보다 앞에 있어야 합니다. `.zshrc`에 `eval "$(/opt/homebrew/bin/brew shellenv)"`를 넣습니다.
2. **네이티브 바이너리 확인:** `file $(which node)` 결과가 `Mach-O 64-bit executable arm64`여야 네이티브입니다. `x86_64`이면 Rosetta로 도는 것입니다.
3. **Node/Python 네이티브 모듈:** npm의 `node-gyp`, Python의 C 확장(`numpy`, `pandas`) 등은 아키텍처별로 컴파일됩니다. arm64 환경에서 `pip install`을 하면 arm64 wheel이 설치되며, x64 wheel을 강제로 받으려면 Rosetta 셸에서 실행해야 합니다.
4. **Docker:** Apple Silicon에서 x64 이미지를 돌리면 QEMU 에뮬레이션으로 실행되어 느립니다. `--platform linux/amd64` 명시가 필요한 경우가 있습니다. 가능하면 arm64 이미지(`FROM node:20-alpine` 등 멀티아키 지원 이미지)를 씁니다.
5. **Electron·구형 앱:** universal binary(arm64+x64 합본)이면 자동으로 arm64로 실행되고, 아니면 Rosetta로 돌아갑니다.
6. **`arch` 명령:** 현재 셸의 아키텍처를 확인하거나 전환하는 데 씁니다. `arch -arm64 command` 또는 `arch -x86_64 command` 형태로 사용합니다.

### 자주 겪는 함정

* `brew install`을 했는데 명령을 못 찾는 경우 → PATH에 `/opt/homebrew/bin`이 없어서 그렇습니다.
* 회사 스크립트가 `/usr/local/...`을 하드코딩해두면 Apple Silicon에서 깨집니다. `$(brew --prefix)`를 쓰도록 수정해야 합니다.
* pyenv·nvm·rbenv로 설치한 언어가 Rosetta 셸에서 컴파일되면 x64 바이너리로 박힙니다. 순수 arm64를 원한다면 네이티브 셸에서 재설치해야 합니다.

## DMG (Disk Image)

### 원문·용어 유래

* **DMG**는 **D**isk i**M**a**G**e의 약자입니다. 확장자는 `.dmg`입니다.
* 애플 고유 포맷으로, macOS·NeXTSTEP 계보에서 유래했습니다. 전신 격인 `.img`, `.smi`(Self-Mounting Image), `.dart`, `.sit` 등을 정리·계승해 **Mac OS X 10.0(2001)** 부터 표준 배포 포맷으로 자리잡았습니다.
* 개념 자체는 오래된 "디스크 이미지"(플로피·CD를 파일 하나로 뜬 것 — ISO 9660의 `.iso`와 같은 계보)입니다. DMG는 여기에 **압축·암호화·읽기전용 검증·이중클릭 마운트**를 얹은 애플 판입니다.

### 정체

* 물리 디스크(파티션·파일시스템 포함)를 파일 하나로 캡슐화한 컨테이너입니다.
* 내부 파일시스템은 보통 **APFS**(최신) 또는 **HFS+**(구형)입니다.
* 더블클릭하면 Finder가 자동으로 **가상 볼륨으로 마운트**하며, 데스크톱·사이드바에 디스크처럼 표시됩니다.
* 안에 든 `.app`을 `/Applications`로 드래그하면 설치가 완료됩니다. **인스톨러가 아니라 파일 복사**가 본질입니다.

### 왜 이 방식인가 (설치 철학)

* Windows에서는 `.msi`·`.exe` 인스톨러가 레지스트리·시스템 폴더에 파일을 흩뿌립니다.
* macOS의 `.app`은 자체로 **번들 디렉토리**입니다. 그 안에 실행 바이너리·리소스·플러그인이 다 들어 있어, 폴더 하나만 옮기면 설치이고 지우면 언인스톨입니다.
* DMG는 이 `.app` 하나를 **압축·서명·검증**해서 전달하는 봉투입니다. 설치 프로그램이 필요없는 구조라 봉투도 단순합니다.

### 주요 특성

| 기능 | 내용 |
|---|---|
| 압축 | UDZO(zlib), ULFO(LZFSE), ULMO(LZMA). 배포 시 원본보다 작아집니다 |
| 암호화 | AES-128/256. 마운트할 때 암호를 요구합니다 |
| 서명·검증 | 애플 개발자 서명 + Gatekeeper·notarization 검사 |
| 읽기전용/쓰기가능 | `UDRO`(읽기전용, 배포용), `UDRW`(쓰기가능, 백업·작업용) |
| 스파스 | `.sparseimage`, `.sparsebundle` — 사용한 만큼만 실제 용량을 차지합니다. Time Machine이 이를 사용합니다 |

### x64 / arm64와의 관계

**DMG 포맷 자체는 아키텍처 중립입니다.** 컨테이너일 뿐이라 arm64·x64 구분이 없습니다. 신경써야 하는 것은 **안에 든 `.app`의 아키텍처**입니다.

내부 `.app`은 세 가지 중 하나입니다:

1. **arm64 전용** — Apple Silicon에서만 네이티브로 돌아가며, Intel Mac에선 실행이 불가합니다.
2. **x86_64 전용** — Intel 네이티브이며, Apple Silicon에선 Rosetta 2로 실행됩니다.
3. **Universal Binary 2** — arm64·x86_64 둘 다 담긴 fat binary입니다. 어느 Mac에서든 네이티브로 실행되며, 애플이 M1 전환기에 밀어붙인 방식입니다.

배포 측 팁:

* 요즘은 **Universal DMG 하나**로 배포하는 것이 표준입니다. 사용자가 아키텍처를 고를 필요가 없습니다.
* 용량을 아끼려고 **arm64 DMG와 x64 DMG를 나눠서** 배포하는 앱도 있습니다(Chrome, Electron 앱 다수). 다운로드 페이지에서 "Apple Silicon" 또는 "Intel"을 선택하는 것이 이 경우입니다.
* CI에서 만들 때는 `lipo`로 arm64·x64 바이너리를 합쳐 universal을 만든 뒤 `hdiutil create`로 DMG를 굽고 `codesign` → `notarytool` 순으로 처리합니다.

### 함정

* 앱을 `/Applications`에 복사하지 않고 **마운트된 DMG 안에서 직접 실행**하면 매번 마운트가 필요하고 업데이트도 되지 않습니다.
* DMG를 받아 열었는데 "손상되었습니다"라고 뜨는 경우는 대개 notarization이 안 된 앱을 Gatekeeper가 차단한 것입니다. 우클릭 → 열기로 우회하거나, `xattr -d com.apple.quarantine`로 격리 속성을 제거합니다.
* Apple Silicon Mac에서 x64 전용 DMG를 처음 실행할 때 Rosetta 설치 프롬프트가 뜹니다.
